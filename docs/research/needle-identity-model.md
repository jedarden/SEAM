# NEEDLE's Current Identity/Session Model

Investigated 2026-07-16 directly from the `~/NEEDLE` source, to ground the SEAM↔NEEDLE handshake design in what NEEDLE actually does today rather than an assumed mechanism.

## Finding: NEEDLE has no credential/token concept for spawned agents today

**1. Identity/session concept assigned at dispatch**

NEEDLE's dispatch entrypoint is `needle run` (`src/cli/mod.rs:420` `cmd_run`, calling into `src/worker/mod.rs`). Identifiers that exist, all purely internal/local:

- `worker_name` — a human-readable NATO-alphabet label (`alpha`, `bravo`, …) or user-supplied `--identifier` string (`src/cli/mod.rs:75-124`). Used for tmux session naming, not a secret.
- `qualified_id` = `"{adapter}-{worker_name}"` (e.g. `claude-code-glm-4_7-bravo`), built at `src/worker/mod.rs:386,403,464,817,2765`. Used for telemetry/log-file naming and CLI stop/attach matching — not passed to the spawned agent process.
- `session_id` — an 8-hex-char random token generated by `generate_session_id()` (`src/telemetry/mod.rs:2451-2465`, via `/dev/urandom` or PID⊕timestamp fallback). Used **only** as a telemetry/log filename suffix (`{worker_id}-{session_id}.jsonl`, `src/telemetry/mod.rs:2044`) — never passed into the child process's environment or command line.
- A bead-claim `actor` string (`src/claim/mod.rs:81-84`) attributes a bead lock in the `br` bead store — again just the worker_name, used for the store, not for authenticating the agent process to anything external.

**None of these are exposed to the spawned agent process as a credential.** The agent's process spawn (`src/dispatch/mod.rs:703-738`, `run_process`) constructs a `bash -c <rendered invoke_template>` command; the only template variables ever substituted are `{workspace}`, `{prompt_file}`, `{bead_id}`, `{model}` (`src/dispatch/mod.rs:188, 410-421`) — no session/worker ID or token variable exists today.

**2. Environment variables set for the spawned agent**

The only env vars injected into the agent child process come from `adapter.environment: HashMap<String,String>` (`src/dispatch/mod.rs:190-192, 731` — `.envs(&adapter.environment)`), a field defined per-adapter in YAML. In this repo's built-in adapters and `.needle.yaml`, that map is empty today — no adapter sets `NEEDLE_WORKER_ID`, `NEEDLE_SESSION_TOKEN`, or similar. It's a generic passthrough mechanism an adapter author could use, but nothing NEEDLE-identity-related is wired into it. `NEEDLE_INNER=1` is set only as a self-relaunch guard (`src/cli/mod.rs:3886`); `NEEDLE_SUPERVISOR_SOCKET` (`src/health/mod.rs:676`) is read by NEEDLE itself, not set for children.

**3. HTTP server / IPC / auth mechanism**

None found. `src/health/mod.rs:664-746` only *checks for the existence* of a Unix socket file (`/tmp/needle-supervisor.sock` or `$NEEDLE_SUPERVISOR_SOCKET`) as a liveness signal for an external supervisor — it doesn't bind, listen, or implement any protocol/auth itself (the only `UnixListener::bind` calls are in test fixtures). No `axum`/`warp`/`hyper::Server`/`actix` server code exists anywhere in `src/`; `tonic` is present only as an OTLP telemetry *client*. **NEEDLE has no existing authenticated IPC/HTTP surface at all.**

**4. Agent type/role concept**

Yes — `AgentAdapter` (`src/dispatch/mod.rs:172-220`), keyed by `name` (e.g. `claude-sonnet`, `claude-opus`, `claude-code-glm-4.7`, `codex`, `aider`), selected via `needle run --agent NAME`. Adapters carry `provider`, `model`, `invoke_template`, `environment`, `token_extraction`. This adapter name is a natural "agent class" that could map to a route-scoping policy (e.g. `claude-code-glm-4.7` → restricted route set) — but currently it only drives invocation mechanics, not any authorization.

**5. Dispatch code locations**

- `~/NEEDLE/src/dispatch/mod.rs` — adapter config, template rendering, process spawn (`run_process` line 703, `execute_agent` line 674).
- `~/NEEDLE/src/worker/mod.rs` — worker lifecycle, `qualified_id`, claim/dispatch/outcome loop.
- `~/NEEDLE/src/cli/mod.rs` — `cmd_run` (line 420), the `needle run` entrypoint.
- `~/NEEDLE/src/claim/mod.rs` — bead claiming (`claim_next`, line 81).

## Bottom line

NEEDLE currently has **no credential/token concept for spawned agent sessions at all** — identifiers that exist (worker_name, qualified_id, session_id) are local labels for logging/tmux/locking, never injected into the agent's process as an auth token or passed to any external service. A SEAM integration requires new NEEDLE-side work: minting an identity in `run_process` and injecting it via the existing `adapter.environment` mechanism (or a new template variable) — most naturally, a `tsnet` ephemeral auth key per worker (see `docs/research/tailscale-identity-mechanisms.md`), set as an env var the spawned agent process (or a thin wrapper around it) uses to bring up its own tailnet identity before calling SEAM.
