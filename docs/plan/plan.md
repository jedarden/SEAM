# SEAM Plan

## Overview

SEAM (Self-documenting Endpoint Access Mediator) is a single unified endpoint that acts as a proxy to multiple authenticated services. It injects the required secret into a request received from an agent, sends the request on to the destination service, and passes the response back — the agent never sees the secret. It also self-documents: a merged OpenAPI spec drives `/docs` and `/openapi.json`, and malformed requests get a structured error naming the bad field and pointing at the right shape, instead of a bare 400.

## Motivation

Today, each backend service (ArgoCD read-only proxy, per-cluster kubectl-proxy, the z.ai/GLM proxy, the twitterapi.io proxy) is documented by hand in CLAUDE.md with its own discovery and auth pattern. This doesn't scale: every new service needs new hand-written prose, agents have no standard way to discover what's available, and there's no mechanism to guide an agent that sends a malformed request back toward the correct one.

## Architecture

- Single gateway process (reverse proxy) in front of N upstream services.
- Route table = one OpenAPI spec, assembled at startup (and reloaded on change) by merging multiple fragment files. Each fragment is an OpenAPI paths/components fragment plus custom extension fields — `x-upstream` (the base URL to forward to), `x-vault-path` (which OpenBao secret to fetch) and `x-inject-as` (header / bearer / query param — how to inject it) — before proxying upstream.
- Route fragments are delivered as Kubernetes ConfigMaps, one per upstream service, mounted into the gateway pod via a `projected` volume at `/etc/gateway/routes.d/*.yaml`. Each fragment is owned and synced independently from its own service's path in declarative-config — no single monolithic file to edit for every new route.
- Route reloads are **in-process and sub-second** (decided 2026-07-16, superseding the earlier restart-via-`reloader` idea): the gateway file-watches `/etc/gateway/routes.d/`, re-merges and re-validates on change, and atomically swaps the route table — in-flight requests finish against the old table, new requests see the new one, zero dropped connections. With dozens of concurrent workers hitting this authentication layer, restart-based reload would be the disruptive option; no blue/green machinery or Traefik-side intelligent routing is needed for config changes, because nothing about the pod changes. Kubelet propagates projected-ConfigMap updates on its sync period (git→pod latency up to ~1 min after ArgoCD sync); if that latency ever matters, the upgrade path is watching ConfigMaps via the Kubernetes API (informer) instead of file mounts. Gateway **binary** deploys remain ordinary rolling updates with connection draining (see Version Migration Strategy §1) — for a stateless proxy that already provides blue/green's guarantee without running a second fleet.
- `/docs` serves a human-readable rendering of the merged spec (Redoc/Swagger UI); `/openapi.json` serves the raw merged spec for agents to fetch programmatically.
- Incoming requests are validated against the merged spec. A malformed request gets a structured 400 naming the bad field, what was expected, and a link to `/docs/{route}`.
- **`/docs/{route}` is a real server-side endpoint, not a `/docs` anchor** (decided 2026-07-20; Phase 1 deliverable). The pointer carried by every structured 400 is the affordance the Overview names as SEAM's differentiator, and its consumer is an agent, not a browser — so it must resolve to something fetchable in one request. A Redoc/Swagger UI section anchor (`/docs#/paths/...`) cannot serve that role: URL fragments are client-side only, never sent to the server, and no server-side path template can produce one. SEAM therefore serves `GET /docs/{route}`, where `{route}` is the URL-escaped OpenAPI path template (e.g. `/docs/%2Fk8s%2F%7Bcluster%7D%2Fapi%2Fv1%2Fpods`), with optional `?method=` and `?version=` selectors — defaulting to every method declared on that path and to the version resolution of Version Migration Strategy §2. The response is that route's slice of the merged spec and nothing else: parameters, request and response schemas, `x-required-scope`, any guard annotations in force, and one concrete worked example request. It is `application/json` by default (the agent path) and `text/html` under content negotiation, where it redirects to the corresponding `/docs` anchor for a human. It is scope-filtered exactly like `/openapi.json` (Per-Agent Tool Scoping point 5) — 404 for an unknown route, 403 naming the missing scope for a route the caller may not invoke. Because the structured 400 (Architecture), the loop-breaker 429 (Per-Route Guards), and the level-2 drill-down all point at it, it ships in **Phase 1** alongside `/docs` and `/openapi.json`, not in a later phase.
- Secret injection reads from OpenBao (the central secrets store — the 4th instance runs on rs-manager) at request time, and never surfaces secret values to the calling agent.
- **OpenBao authentication and secret-path allowlist** (decided 2026-07-20; Phase 2 deliverable). SEAM authenticates to OpenBao via **Kubernetes auth** — the pod's projected ServiceAccount token is exchanged for a short-lived OpenBao token, renewed in-process; no static token, AppRole secret-id, or credential file on disk (AppRole is the fallback only for a future non-Kubernetes deployment). Critically, because fragments are authored largely by agents, **the fragment must not be able to name an arbitrary secret**: SEAM's OpenBao policy grants `read` on a single bounded prefix, `seam/routes/*`, and nothing else — so even a hostile fragment cannot reach `armor/`, `kalshi/`, cluster kubeconfigs, or any other tenant's material, because SEAM's own token has no capability there. The allowlist is enforced twice: `seam lint` rejects any `x-vault-path` outside the prefix at CI time, and the gateway re-checks it at merge time and quarantines the fragment (never trusting lint to have run). Additionally, a fragment's `x-vault-path` must be **co-owned with its upstream target**: a fragment shipped from declarative-config service directory `<svc>/` may only reference `seam/routes/<svc>/*`, so the exfiltration shape "point service A's secret at service B's upstream" fails on ownership even inside the allowlist. Path traversal, globs, and templated segments in `x-vault-path` are rejected outright.
- **Inbound header stripping is a requirement, not an implementation detail** (decided 2026-07-20; Phase 2 deliverable). Before injection, SEAM **strips** every header it may itself set, rather than overlaying values onto whatever the caller sent: the header named by each matched route's `x-inject-as`, `Authorization`, and all `X-SEAM-*` request headers except the documented caller-supplied ones — `X-SEAM-Dry-Run` (Per-Route Guards) and `X-SEAM-API-Version` (Version Migration Strategy §2). Comparison is on the canonicalized header name (case-insensitive, per-key delete of *all* values, not a single-value overwrite), so `authorization`, `AUTHORIZATION`, and duplicate keys are all removed. This is strip-then-inject, never trust-then-overlay — an attacker-supplied `Authorization` or a custom header matching some other route's `x-inject-as` must never survive into an upstream request. Note that Go's `httputil.ReverseProxy` `Rewrite` hook removes only hop-by-hop and `X-Forwarded-*` headers (ADR-001 §3); everything in this bullet is SEAM's own responsibility on top of that.
- **Secret caching policy** (decided 2026-07-20; Phase 2 deliverable): secrets are held in an in-memory cache with a **short 30-second TTL**, never rendered to a file or held for the process lifetime. The cache is invalidated eagerly on an upstream `401` (which triggers refetch-and-retry-once, see Credential Health Sentinel), and TTL expiry forces a refetch otherwise. Rotation latency is therefore bounded at 30s worst case for a silent rotation, and effectively zero for a rotation the upstream signals by rejecting the stale credential — the point being that an OpenBao rotation takes effect without redeploying or restarting SEAM. The cache is keyed by resolved `x-vault-path`, so instances sharing a path share one entry; values are never logged, never written to disk, and never included in `/config/status` or any health surface.

## Components

- **Gateway service** — HTTP server, written in **Go** (decided 2026-07-20, ADR-001 below): OpenAPI merge/validation (pb33f `libopenapi`/`libopenapi-validator`), secret injection, `/docs` + `/openapi.json`.
- **Route fragments** — per-service ConfigMaps, each an OpenAPI fragment + injection metadata, authored alongside the service they describe.
- **Tailscale Connectors** — per-cluster egress, added incrementally as new upstreams are onboarded.
- **`seam` CLI** — the gateway's merge/validation engine packaged as a command-line tool for CI and fragment authoring (`lint`, `diff`, `import`); see Fragment Toolchain below.

## Data Models

- Route fragment schema (OpenAPI 3.1 fragment + extension fields) — not yet designed; needs a concrete JSON Schema before the first fragment is written. Extension fields accepted so far:
  - `x-upstream` — the single upstream base URL this fragment's routes forward to (e.g. `http://argocd-ro-ardenone-manager-ts.ardenone.com:8444`). Required for an ordinary single-instance route; the request path is appended to it. **Mutually exclusive with `x-upstream-map`** — a fragment declares exactly one of the two. `x-upstream-map` supersedes `x-upstream` when (and only when) the path declares an `{instance}` parameter; a fragment carrying both, or carrying `x-upstream-map` with no `{instance}` parameter, is a lint error and is quarantined at merge time. OpenAPI's native `servers:` is **not** used to express the upstream target (decided 2026-07-20): `servers:` in the merged spec is reserved for SEAM's own base URL — that is the address agents must actually call — and its root/path/operation inheritance rules make a merged multi-fragment document ambiguous about which server a given operation resolves against. Keeping upstream targets in a SEAM-owned extension also keeps them out of the served spec: `/openapi.json` never discloses upstream hostnames, only SEAM's front door.
  - `x-vault-path` — which OpenBao secret to fetch; must resolve inside SEAM's OpenBao allowlist prefix and be co-owned with the upstream (see Architecture)
  - `x-inject-as` — how to inject it (header / bearer / query param)
  - `x-required-scope` — scope(s) a caller must hold (per the per-agent scoping design below)
  - `x-credential-probe` — a designated safe request (path + method + **per-fragment probe interval**; decided 2026-07-16 — some credentials rotate every ~3 days, others essentially never, so cadence is per-fragment, not a global setting) the credential sentinel uses to validate this fragment's secret against its upstream — on a multi-instance fragment the interval is a default that each `x-upstream-map` entry may override per instance, since instances of one API can hold credentials with very different rotation cadences
  - `x-seam-schema` — an explicit fragment-schema version marker (e.g. `v1`) carried by every fragment from day one, so the fragment schema itself can evolve detectably — `seam lint` and the gateway can distinguish a v1 fragment from a v2 instead of guessing from shape
  - `x-loop-guard` — per-route repeated-identical-request thresholds (max repeats, window)
  - `x-cost-per-call` / `x-quota` — per-route cost annotation and per-caller budget; guards are configured individually per endpoint, never SEAM-wide
  - `x-upstream-map` — for multi-instance routes: maps an `{instance}` path parameter to a **per-instance object**, not a bare base URL (decided 2026-07-20). Each entry is `{url, vaultPath, injectAs, probeInterval}`; only `url` is required, the fragment-level `x-vault-path`, `x-inject-as`, and `x-credential-probe` interval act as defaults for any key an entry omits. This exists because the motivating case demands it: the per-cluster kubectl-proxies expose a byte-identical API but each needs a *different* credential, differing in kind — long-lived ServiceAccount tokens for the read-only observers, ~3-day OIDC `cloudspace-admin` tokens for the read/write ones — so a single scalar `x-vault-path` cannot serve the collapsed fragment, and a 3-day token needs a far tighter probe cadence than one that never expires. The reserved instance `_all` fans out. Consequently every per-secret surface keys on **(fragment, instance)** rather than fragment alone — see `/health/credentials` below; for a single-instance fragment the instance key is the reserved value `_default`.
  - `x-api-version` — the route contract's version marker (e.g. `v1`), distinct from `x-seam-schema`: `x-seam-schema` versions the *fragment format SEAM parses*, `x-api-version` versions the *contract SEAM serves to callers*. A breaking route change ships as a new fragment bearing a new `x-api-version` (or a new path segment), never as an in-place edit — see Version Migration Strategy and `docs/notes/versioning.md`.
  - `x-seam-deprecated` — a **root-level boolean on the fragment**, marking every route in that fragment deprecated; when true, responses served from it carry `Deprecation` (RFC 9745) and `Sunset` (RFC 8594) headers, and it is counted by the per-route-version request metric that gates retirement. This is the authoritative representation (decided 2026-07-20): OpenAPI's standard `deprecated` is an **operation-level** boolean with no document-level equivalent, so "a fragment marked `deprecated: true`" has no meaning in the spec — since SEAM's unit of versioning is the whole fragment (a breaking change ships as a new fragment), the marker must be fragment-scoped, hence a SEAM extension. Standard per-operation `deprecated: true` remains valid and is honored for deprecating an individual operation within an otherwise-live fragment; `x-seam-deprecated: true` is equivalent to setting it on every operation in the fragment.
  - `x-adapter` — declarative request/response transforms carried by an **older** coexisting fragment, converting its contract into that of the newest version so SEAM does not need N parallel implementations (decided 2026-07-20; Phase 8 deliverable). Shape: `{targetVersion, request: [...], response: [...]}`, where `targetVersion` names the live `x-api-version` this fragment adapts into and each list is an ordered sequence of transforms drawn from a **closed** vocabulary — rename/move a field by JSON Pointer, default a missing field, drop a field, wrap/unwrap an envelope, rename a query or header parameter. No expressions, no scripting, no code hooks: adapters are data, so a contract migration is a ConfigMap edit and never a gateway binary deploy (see Version Migration Strategy §2). A breaking change that cannot be expressed in this vocabulary gets no adapter — the old fragment simply stays standalone until usage-gated retirement drains it.
  - `x-unscrubbable: acknowledged` — a fragment- or route-level opt-in, and the **only** accepted value (decided 2026-07-20). Secret-echo scrubbing (Credential Health Sentinel) cannot scan a protocol-upgraded, opaque-binary, or undecodable-encoding response, so on such a route SEAM refuses injection by default; this marker is the explicit, auditable acknowledgement that a secret is being injected into a route whose responses cannot be scrubbed. `seam lint` flags every occurrence for human review rather than passing it silently, and `/config/status` enumerates every route currently running unscrubbed. Absence of the field is refusal, not permission — there is no `false` form, because the field exists to make the exposure enumerable, and an enumerable exposure cannot be expressed by a field nobody has to write.

## Future: Per-Agent Tool Scoping

Near-future requirement (see `docs/notes/objective.md`): configure specifically which routes a given agent/NEEDLE-worker may call, rather than every caller getting every route. Researched in detail in `docs/research/` (four threads: secret-injection prior art, agent tool-access-control state of the art, Tailscale-native identity mechanisms, NEEDLE's current identity model). Combined recommendation, from `docs/research/README.md`:

1. **Identity:** each NEEDLE worker/agent gets its own `tsnet`-embedded ephemeral Tailscale node, tagged (e.g. `tag:needle-worker`), so it's a first-class tailnet peer calling SEAM directly. Required, not optional — today's shared Tailscale Connector/egress proxy pods collapse every worker in a cluster into one indistinguishable caller from SEAM's `WhoIs` point of view. No SPIFFE/SPIRE or custom PKI — Tailscale already provides workload identity for this topology (precedent: Tailscale's own TailSQL/Setec/Golink).
2. **Authorization data:** scope claims (`service:action`, e.g. `k8s-ro:get`, `argocd:sync`) travel in the Tailscale Grant's opaque `app` capability field, read via `WhoIs` on the inbound connection. No separate JWT-issuing auth server needed for tailnet-native callers; fall back to a bearer-claim-in-header model only for callers that can't be first-class tailnet peers. The fallback bucket is real, not hypothetical (decided 2026-07-16): **"foreign" NEEDLE workers running outside the tailnet** will still need these tools, likely ingressing via something like a Cloudflare Tunnel — so the bearer-claim path is a first-class (if secondary) auth mode, and the design must cover how that non-tailnet ingress itself authenticates (e.g. Cloudflare Access service tokens) before any scope claim is trusted. That design is **point 7 below** (written 2026-07-20); until it ships, non-tailnet callers are default-denied, not provisionally trusted.
3. **Enforcement:** every SEAM route is tagged with its required scope(s) (`x-required-scope` in the route fragment); the gateway checks the caller's Grant-supplied scopes against them, default-deny. Never rely on the calling agent's own tool list as the boundary — client-side restriction is UX, not security (OWASP Agentic Top 10 consensus, and a live AutoGen cross-agent-tool-leak issue found in research).
4. **Gap to close first:** NEEDLE currently mints no identity/credential for spawned agents at all (`docs/research/needle-identity-model.md`). This needs new NEEDLE-side work in `run_process` (`~/NEEDLE/src/dispatch/mod.rs:703`) — provision a per-worker `tsnet` identity, inject it via the existing `adapter.environment` passthrough (or a new template variable) — before SEAM-side scoping can be enforced against anything real.
5. **Self-service surface:** authorization must be discoverable, not opaque. `/whoami` returns the caller's resolved tailnet identity, tags, and effective scopes; `/openapi.json` and `/docs` filter to the routes the caller may actually invoke, so agents never spend context on unreachable routes; and a 403 names the exact missing scope plus the Grant snippet that would fix it — authorization debugging becomes "ask the gateway," not "read the tailnet policy file."
6. **Scope taxonomy:** grows route-by-route as fragments declare `x-required-scope` — no up-front `service:action` catalog (decided 2026-07-16). The live taxonomy must stay dynamic and discoverable: enumerable from the merged spec (e.g. a `/scopes` view) rather than maintained as a separate document. A future SEAM shows different taxonomy slices to different agents, RBAC-style — an agent sees only the portion of the taxonomy its grants intersect, consistent with the scope-filtered spec in point 5. Since discoverability *replaces* a maintained catalog here, `/scopes` is a concrete Phase 7 endpoint, not a figure of speech (decided 2026-07-20): it is **scope-filtered by default, exactly like `/openapi.json`** — returning the scopes the caller's grants actually intersect, each with a one-line meaning and the (likewise filtered) routes that require it, so "what may I do here" is one request that costs no context on scopes the caller can never hold. The **unfiltered** taxonomy is available at `/scopes?all=1`, but is itself gated on a scope (`seam:scopes:read-all`): a complete inventory of every capability across every service SEAM fronts is precisely the reconnaissance that filtering `/openapi.json` exists to withhold, so it is an operator view, never an ambient one. Both forms are derived from the merged spec's `x-required-scope` declarations at request time — there is no stored catalog, and therefore nothing that can drift.
7. **Non-tailnet ingress authentication** (decided 2026-07-20) — the design point 2 defers to, written down here because a declared first-class auth mode with no design gets built ad hoc as "trust the `X-Scopes` header," which on a secret-injecting proxy is a trivially forgeable authorization bypass that hands an attacker every credential SEAM holds. Four rules, none of them per-deployment negotiable:
   - **The ingress identity is verified at the gateway, not at the edge.** Foreign workers reach SEAM through a Cloudflare Tunnel fronted by Cloudflare Access; Access issues a service-token JWT, and **SEAM itself validates that JWT on every request** — signature against Cloudflare's published JWKS for the team domain, `aud` matching SEAM's own Access application ID, plus `iss`/`exp`/`nbf`. SEAM never treats "this request arrived on the tunnel listener" as proof of anything: a tunnel is transport, and edge-only verification means anyone who reaches the origin by any other path is authenticated by default.
   - **Scopes are bound to the verified ingress identity and never read from a client header.** The caller's scope set is looked up from the verified token subject (`sub` / `common_name`) against a SEAM-side service-token→scopes mapping — structurally the same move as looking up a tailnet caller's scopes from the `WhoIs` Grant. There is no request header, query parameter, or body field through which any caller, in this mode or any other, can assert its own scopes.
   - **Client-supplied scope headers are stripped before evaluation, not merely ignored.** `X-SEAM-Scopes` and every equivalent are deleted on ingress by the same strip-then-inject rule that removes `Authorization` and `x-inject-as` headers (Architecture), so a forged header cannot be read even by a future code path that mistakenly looks for one, and cannot survive into an upstream request.
   - **Default-deny on the mode itself.** The foreign-worker path is off unless explicitly enabled, and a request presenting no valid Access JWT is rejected *before* route matching, secret lookup, or any upstream contact. The failure mode is a 403 — never "fall through to the tailnet path with no identity resolved."
   - **Phasing: explicitly out of Phase 7 scope.** Phase 7 ships the tailnet-native Grant path only and default-denies every non-tailnet caller. The foreign-worker mode lands as a distinct later phase of its own, gated on Phase 7's enforcement being real and exercised, because a second identity source is only safe once the enforcement point it feeds already works.

## Version Migration Strategy

NEEDLE workers run perpetually-live sessions — a session may learn the API contract once (fetch `/openapi.json` early) and keep calling routes on that understanding for hours, with no guarantee it ever restarts or refetches the spec. Full design in `docs/notes/versioning.md`; precedent research in `docs/research/version-migration-precedent.md`. **Three** distinct problems (corrected 2026-07-20 — this read "two" from before passive drift signaling was adopted as item 3). The numbering here maps one-to-one onto the section labels in `docs/notes/versioning.md`, so a `§`-reference means the same thing in either document: **§1 = §A** (gateway process deploys), **§2 = §B** (route contract versioning), **§3 = §C** (passive drift signaling). Cite them as "Version Migration Strategy §2" / "versioning.md §B" interchangeably.

1. **Gateway process deploys** (SEAM's own code changes, contract unchanged) — standard Kubernetes rolling-update mechanics: readiness-gated rollout and graceful `SIGTERM` draining. **SEAM runs as a single replica** (amended 2026-07-20 — this bullet previously mandated ≥2 replicas and `PodDisruptionBudget minAvailable: 1`). Every stateful feature in this plan is per-process and in-memory by design — loop-breaker counters and cost-governor budgets (Per-Route Guards), last-2xx tracking and circuit-breaker state (Passive Route Health), the credential sentinel's probe loop — and two replicas behind one Service silently double all of it: a loop-guard threshold of 10 permits 20 identical requests, a `$50` `x-quota` permits `$100` against a metered upstream, `X-SEAM-Budget-Remaining` depends on which pod answered, `/docs` shows a different last-success on each refresh, and the breaker opens on one replica while the other keeps hammering a dead upstream. On the twitterapi.io credit and z.ai quota routes that is real money, so correctness of the guards outranks a second replica of a stateless proxy. The Deployment is therefore `replicas: 1` with `strategy.rollingUpdate: {maxSurge: 0, maxUnavailable: 1}` — a rollout never runs two SEAM pods at once — at the cost of a sub-second gap during a binary deploy, which a retrying agent absorbs and which binary deploys are rare enough to make cheap (route and contract changes are ConfigMap edits that never restart the pod at all). No `PodDisruptionBudget` demanding an available replica is set, since with one replica it would only deadlock node drains. The single-replica invariant is load-bearing: **no feature may be built assuming it can be replicated**, and each stateful section below states its own multi-replica semantics explicitly. If SEAM ever genuinely needs HA, the documented upgrade path is a shared state store for guard counters, quota ledgers, and breaker state (Redis or equivalent) plus leader election for the sentinel — a scoped, later piece of work, not something to be back-doored by raising `replicas`. The credential sentinel's probe loop is leader-elected regardless (see Credential Health Sentinel), so the invariant survives both that future scale-out and any accidental transient overlap.
2. **Route contract versioning** (a route's params/response/injection shape changes in a breaking way) — borrows Kubernetes' API deprecation policy (old and new versions coexist as long as anything calls the old one — this falls out of the existing ConfigMap-per-fragment model for free, a breaking change just ships as a new fragment rather than an in-place edit) and Stripe's adapter model (implement the route once against the newest contract, add a thin request/response adapter per older version still being served, rather than N parallel full implementations). Deprecated fragments emit `Deprecation`/`Sunset` headers (advisory only) and are retired when a per-route-version request-count metric has read zero for a quiet window. **Zero observed traffic is a necessary condition for retirement; the length of the quiet window is traffic-adaptive with a calendar floor** (clarified 2026-07-20 — this read "usage-gated, never calendar-gated", which was heard as forbidding the calendar floor the recommended composition in `docs/notes/quiet-window-options.md` depends on). The distinction that matters is the one originally intended: no route is ever retired because a date arrived, only because it stopped being called — a calendar term may only ever *lengthen* the wait, never trigger a removal on its own. See Phase 8 for the adopted composition.

   **Adapters are declarative transforms in the old fragment's own YAML, never Go code in the gateway binary** (decided 2026-07-20). Stripe's model assumes a route implementation to adapt *toward*; SEAM is a proxy and has no implementation of its own — the implementation is upstream. So the model translates as: the **newest** fragment is the one whose contract matches what the upstream actually speaks today, and every older coexisting fragment carries an `x-adapter` block (see Data Models) declaring the request-in / response-out transforms that convert its older contract into that newest one. Keeping adapters in the fragment is what preserves SEAM's config-only property: a route-contract change stays a ConfigMap edit synced by ArgoCD, with **no gateway binary deploy** — "nothing about the pod changes" (Architecture) holds for contract migrations too. Adapters as gateway Go code would invert that, making every route-contract change a fleet redeploy and dragging per-service transform logic back into exactly the monolithic file the ConfigMap-per-fragment model exists to abolish. The transform vocabulary is deliberately **small and closed** — rename/move a field by JSON Pointer, supply a default for a field the old contract omits, drop a field, wrap/unwrap an envelope, rename a query or header parameter — and there is **no escape hatch into arbitrary code or expressions**. A change that cannot be expressed in that vocabulary is simply not adapted: the old fragment stays mounted as a standalone fragment against its own upstream target and lives out its usage-gated retirement unmodified. Accordingly, **Phase 8 must deliver** the `x-adapter` schema and its transform vocabulary, the request/response transform executor in the proxy path, `seam lint` validation of adapter blocks (including that `x-adapter.targetVersion` names a version that is actually live and coexisting), and the version-selection mechanism below.

   **Selecting among coexisting versions** (decided 2026-07-20). When a breaking change ships as a new **path segment**, selection is the URL and needs no mechanism. When it ships as a new **`x-api-version`** under the same (path, method), the caller selects with an explicit `X-SEAM-API-Version: <v>` **request header** — and when that header is absent, the request resolves to **the oldest still-served version, not the newest**. That default is the only one consistent with the rest of this strategy: a perpetually-live worker that learned the contract hours ago sends no version header at all, so defaulting to newest would break precisely the callers coexistence exists to protect, turning every breaking change into a silent fleet-wide outage. Newly-arriving and re-reading callers are steered forward instead of shoved: `/openapi.json` and `/docs` advertise the newest version as canonical and render older coexisting ones as deprecated, `Deprecation`/`Sunset` and `/changes` do the pushing, and the effective default advances on its own the moment usage-gated retirement removes the drained old fragment. Every response echoes the version that actually served it in `X-SEAM-API-Version`, so a caller is never guessing which contract it is talking to. Note that `X-SEAM-API-Version` is an inbound `X-SEAM-*` header and must therefore be added to the documented exception list in the inbound-header-stripping rule (Architecture), alongside `X-SEAM-Dry-Run`.
3. **Passive drift signaling** (helping perpetually-live callers notice contract change mid-session) — every response carries an `X-SEAM-Spec-Version` header (a hash of the currently-served merged spec), so a worker that cached the contract hours ago sees the version change on a call it was already making, at no extra request cost. The catch-up diff is **progressively discoverable** (decided 2026-07-16), giving schema-diff fidelity without the token cost of dumping full schemas: level 1, `/changes?since=<version>`, returns a token-cheap route-level list (path, verb, change kind, one-line summary, drill-down link); level 2, `/changes?since=<version>&route=<path>`, returns the full field-level schema diff for that one route; prior full specs stay fetchable at `/openapi.json?version=<v>`. An agent notices the header change, pulls level 1, and drills into only the routes it actually uses. Backed by a ring buffer of the last N merged specs; this gives the advisory `Deprecation`/`Sunset` headers the actionable catch-up path they lacked, and usage-gated retirement remains the enforcement backstop. Full design in `docs/notes/versioning.md`.

   **Archived specs and diffs are stored unfiltered and filtered at serve time** (decided 2026-07-20) — the reconciliation between this section and the scope-filtered `/openapi.json` of Phase 7 (Per-Agent Tool Scoping point 5). The ring buffer holds the **unfiltered global** merged spec, one entry per spec version, and explicitly *not* one entry per caller-grant combination: that would multiply the buffer by the number of distinct grant sets in the fleet and is not a cost this design will carry. Instead, **every read path applies the caller's *current* scope filter at serve time.** `/openapi.json?version=<v>` returns the archived spec reduced to the routes that caller may invoke today; `/changes?since=<version>` (level 1) omits routes the caller cannot see; `/changes?since=<version>&route=<path>` (level 2) returns a 403 naming the missing scope (per point 5) for a route outside the caller's grants. An archived spec is **never** served verbatim — serving it raw would make the archive a one-request bypass of the entire filtering guarantee, disclosing the full route inventory of every service SEAM fronts to anyone who asks for yesterday's version.

   `X-SEAM-Spec-Version` hashes the **global** merged spec, not the caller's filtered view. It is a change *signal*, not an authorization surface, and a per-caller hash would fragment the version namespace so thoroughly that a `since=` value could not be reasoned about across two callers, across logs, or across the ring buffer at all. The deliberate consequence is that a caller whose *grants* changed can see routes appear or disappear in a diff for reasons having nothing to do with the spec changing — and that is signaled explicitly rather than left to look like contract drift. Level-1 change kinds therefore separate **contract** changes (`added` / `removed` / `params-changed` / `response-changed` / `deprecated`) from **visibility** changes (`granted` / `revoked`), and responses carry `X-SEAM-Scope-Version` — a hash of the caller's effective scope set — alongside `X-SEAM-Spec-Version`, with `/whoami` reporting the same value. A caller may pass `&scope-since=<scope-version>` to `/changes`; SEAM retains the scope set behind each `X-SEAM-Scope-Version` it has issued (a small hash→scope-set map, which is SEAM's own derived data and cheap, unlike a per-caller spec archive), so it can diff the two filtered views and label each route's delta as contract or visibility. Omitting `scope-since` yields a diff flagged `scopeChanged: unknown`, telling the agent it cannot distinguish the two causes for itself. Net effect: a worker can always tell "the contract moved" from "my access moved" without guessing.

## Credential Health Sentinel

SEAM holds both the secret (`x-vault-path`) and a way to test it (the upstream), so it can detect dead credentials before an agent does. Recurring pain this targets: cloudspace-admin OIDC tokens expiring every ~3 days, and a dead ESO static token going unnoticed for 16 days.

- A background loop validates each fragment's credential against its upstream via the fragment's `x-credential-probe` (a designated safe authenticated request), on a per-fragment cadence.
- `/health/credentials` reports per **(fragment, instance)** — the reporting key, since one multi-instance fragment can hold N distinct credentials (see Data Models); single-instance fragments report under the reserved instance `_default` — giving last verified, current status, known expiry — never secret values.
- Self-healing on rotation: an upstream `401` invalidates the cached secret, refetches from OpenBao, and retries once (safe for any method — a 401 means the upstream rejected the request before processing it).
- **The probe loop is leader-elected, always** (decided 2026-07-20). SEAM runs single-replica (Version Migration Strategy §1), so today exactly one process holds the loop — but the loop is the one stateful feature whose duplication is visible *outside* SEAM: two copies double authenticated probe traffic against credentials that several upstreams rate-limit, which turns a health check into the thing that breaks the credential. So the loop acquires a Kubernetes `Lease` before probing and drops it on `SIGTERM`, and a process that does not hold the Lease serves traffic normally but probes nothing. This costs almost nothing at one replica and makes both the future shared-state HA path and any accidental transient two-pod overlap safe by construction. `/health/credentials` is served by every replica from its own last-known results, which is why the endpoint reports `lastVerified` timestamps rather than implying live truth.
- **Secret-echo scrubbing belongs to the injection guarantee, not to this sentinel — it is a Phase 2 deliverable, not Phase 12** (decided 2026-07-20). Some upstreams echo the token back in an error body; if SEAM proxies that through, the agent sees the secret and the plan's single core guarantee ("the agent never sees the secret", Overview) is void. Injection ships in Phase 2 and agents are cut over in Phase 6, so scheduling the scrub with the sentinel in Phase 12 would leave a multi-phase window in which the guarantee is simply false for any echoing upstream. Scrubbing therefore ships **with injection, in the same phase**, and applies to every injected route whether or not that route declares an `x-credential-probe`. Mechanics, stated because "scan the response" is not implementable as written against the streaming and WebSocket passthrough ADR-001 treats as a hard requirement:
  - **Buffered responses** (known `Content-Length` under a configured cap, default 1 MiB): decompress if `Content-Encoding` is one SEAM can decode, scan for the exact injected value(s) for that request, redact to `[REDACTED-BY-SEAM]`, re-encode. Response headers and trailers are scanned on the same terms — an echoed token in a `WWW-Authenticate` or a vendor debug header leaks exactly as well as one in a body.
  - **Chunked and event-stream responses:** scanned incrementally with a carry-over overlap — SEAM holds back the trailing `len(secret) - 1` bytes of each chunk and prepends them to the next, so a secret straddling a chunk boundary is still matched. This preserves streaming (no full buffering) and bounds added latency to one chunk's worth of tail. Where multiple secrets could be present, the overlap is the longest of them.
  - **Unscannable responses** — protocol upgrades (WebSocket and anything else that leaves HTTP framing), opaque binary bodies, and content encodings SEAM cannot decode — **cannot be scrubbed, and SEAM does not pretend otherwise.** On such a route, injection is refused by default: the fragment must carry `x-unscrubbable: acknowledged` to opt in, `seam lint` flags it for human review, and `/config/status` lists every route currently running unscrubbed so the exposure is enumerable rather than discovered. A route that both injects a secret and upgrades the connection is a deliberate, visible exception — never a silent one.
  - Scrubbing is **defense in depth, not the primary control.** It cannot catch a secret the upstream returns transformed (truncated, re-encoded, hashed), so it never licenses treating an echoing upstream as safe; such an upstream is a bug to report upstream and a candidate for a narrower credential.

## Per-Route Guards: Loop Breaker and Cost Governor

SEAM is the sole chokepoint for metered upstreams (twitterapi.io per-call credits; the z.ai/GLM quota that has no queryable API) and for runaway agent retry loops (precedent: NEEDLE re-dispatching one failing bead 310×/$500). Guards are **configured individually per endpoint** in the route fragment — there are no SEAM-wide defaults, and a route without guard fields gets no guard.

**Guard state is per-process, in-memory, and single-replica by construction** (decided 2026-07-20). Counters, budget ledgers, and their windows live in the gateway process and are lost on restart — which is safe only because SEAM runs exactly one replica (Version Migration Strategy §1, amended for this reason). Two replicas would double every threshold here: a max-repeats of 10 would permit 20, a `$50` `x-quota` would permit `$100`, and `X-SEAM-Budget-Remaining` would report whichever pod happened to answer. Guards are therefore never to be built assuming replication, and raising `replicas` is not a valid operational response to load; if HA is ever required, the shared quota/counter store named in §1 lands first. Restart-scoping is deliberate and stated so callers can rely on it: a SEAM restart resets budget windows and loop counters, so budgets are a runaway-spend brake, not an accounting system of record.

- **Loop breaker** (`x-loop-guard`): counts repeated identical requests keyed by (caller, route, normalized request hash); exceeding the route's threshold returns a structured 429 telling the agent, in plain language, that it has sent the same failing request N times and should stop and re-read `/docs/{route}`. Before Phase 7 identity exists, callers behind shared egress are indistinguishable, so the key degrades to (route, hash) — thresholds must be set with that in mind. **This ships in Phase 13 without waiting for identity** (decided 2026-07-20, resolving the corresponding Open Question): the pathology being caught is one agent replaying a byte-identical failing request, and the degraded (route, hash) key *aggregates* callers rather than splitting them, so it is strictly more conservative — it trips earlier than the identity-aware version, never later, and can never let a loop through that per-caller keying would have caught. The failure mode it does introduce is collateral: a second, innocent caller sending the same request can be 429'd by the first caller's loop. That is bounded and acceptable because the guard is opt-in per route (no `x-loop-guard`, no guard), thresholds on shared-key routes are set with headroom for legitimate concurrent repeats, and the 429 body states plainly that the guard is currently route-wide rather than per-caller so an agent can tell "I am looping" from "someone else is." Phase 7 narrows the key to (caller, route, hash) with no change to fragment config or thresholds' meaning — it only removes the collateral.
- **Cost governor** (`x-cost-per-call`, `x-quota`): per-caller budgets on routes that cost money; responses carry `X-SEAM-Budget-Remaining` so agents can self-throttle. Doubles as the usage meter z.ai never shipped. Per-caller precision also sharpens with Phase 7. **Before Phase 7, `x-quota` degrades symmetrically with the loop breaker: it is enforced as a single route-wide budget** (decided 2026-07-20) — the key is (route) alone, every caller draws from one shared pool, and the pool is what `x-quota` names. Stating this is not pedantry: "per-caller budget" with no callers is unimplementable, and an implementer left to invent the degradation could plausibly choose per-source-IP (meaningless behind shared egress) or, worse, no enforcement at all — on the twitterapi.io credit and z.ai quota routes that is the difference between a capped spend and an uncapped one. Consequently `X-SEAM-Budget-Remaining` reports the **shared remainder**, and callers must not treat it as their own allowance: another agent can consume it between one response and the next request, so it is a fleet-wide fuel gauge, not a reservation. The guard is deliberately fail-closed — an exhausted route-wide budget 402/429s every caller, including well-behaved ones — because the alternative on a metered upstream is spending money SEAM was installed to stop spending. Phase 7 splits the pool per caller with no fragment change; `x-quota` is then read as the per-caller figure it always claimed to be. Note that an `_all` fan-out (Multi-Instance Routes) is billed as one call per instance actually dispatched, not one call, so a fan-out route's `x-quota` must be sized against the map's width.
- **Dry-run** (`X-SEAM-Dry-Run: 1` request header): validate a request against the spec and return the verdict without injecting a secret, proxying, or spending quota — agents can test their contract understanding against expensive routes for free.

## Multi-Instance Routes and Fan-Out

Many upstreams are N near-identical instances of the same resource — the per-cluster kubectl-proxies all expose a byte-identical API. Instead of N copy-paste fragments, one fragment declares an `{instance}` path parameter (e.g. `/k8s/{cluster}/...`) resolved against an `x-upstream-map` lookup table, collapsing the config N:1. Each map entry is an object (`url`, plus optional per-instance `vaultPath` / `injectAs` / `probeInterval` overriding the fragment-level defaults — see Data Models), because near-identical APIs routinely need per-instance credentials of differing kind and rotation cadence.

- The reserved instance `_all` fans the request out to every mapped upstream concurrently and returns one merged result labeled by instance — e.g. `/k8s/_all/api/v1/pods?fieldSelector=status.phase!=Running` answers "what's broken across the entire fleet" in one call.
- Partial-failure semantics for fan-out (some instances error or time out) are an open question below.

## Passive Route Health — Last-Success Tracking

`/docs` shouldn't just describe routes; it should demonstrate them. The signal is passive, derived from live traffic rather than synthetic probes:

- The gateway records the timestamp of the most recent successful (2xx) response per route (its "last-200"). `/docs` renders it per path — "last succeeded 2m ago" — an honest signal distinguishing verified-working from merely-documented. A route with no successful call renders an **intentionally empty** last-success value, presented as "no attempt since last restart" (decided 2026-07-16) — never "never succeeded," which reads as broken for a freshly added route. Tracking is in-memory and restart-scoped, deliberately not persisted.
- `/health/upstreams` aggregates the same data per upstream into a single status page for the whole mesh: most recent success across the upstream's routes plus current breaker state.
- Per-upstream circuit breaker: consecutive failures open the breaker, and callers get a structured 503 naming the upstream, when it started failing, the last error seen, and a retry-after — so agents stop burning tokens retrying a dead service.
- **Both signals are per-process and single-replica by construction** (decided 2026-07-20). Last-2xx tracking is already declared in-memory and restart-scoped; breaker state is the same. This is coherent only at one replica (Version Migration Strategy §1, amended for this reason): with two, `/docs` would show a different "last succeeded" on each refresh depending on which pod rendered it, and the breaker would open on one replica while the other kept hammering a dead upstream — the breaker's entire purpose defeated by the deployment topology. Neither signal may be built assuming replication; if HA ever lands, both move into the shared state store named in §1. The restart-scoping remains deliberate, and its honest rendering — "no attempt since last restart" — is what keeps a restarted gateway from reading as a broken one.
- Synergy with the credential sentinel: its probe traffic keeps at least one authenticated route per upstream warm, bounding staleness of the signal for probed upstreams.

## Fragment Toolchain (`seam` CLI)

The config surface is YAML fragments authored largely by agents and merged at runtime — one bad file must never break the gateway, and mistakes should be caught before merge, not at reload.

- **Runtime quarantine:** a fragment that fails schema validation or collides with an already-loaded fragment is skipped, logged, and surfaced at `/config/status` (loaded fragments, versions, quarantined ones with their errors) — never fatal to the remaining routes. **The collision key is the triple (path, method, `x-api-version`)** (decided 2026-07-20), never the path alone: two fragments declaring the same path and method but *different* `x-api-version` values are a legal coexistence pair, not a collision — that is precisely the shape a breaking change ships as (see Version Migration Strategy §2), so keying on path alone would quarantine every versioned migration and silently dismantle the coexistence model the whole version strategy rests on. A fragment carrying no `x-api-version` is keyed under the reserved version `_unversioned`, so two unversioned fragments on the same (path, method) still collide as before. Only an exact triple match is a collision, and the loser is always the later-merged fragment (deterministic filename order), never the incumbent — a bad new fragment cannot evict a working route.
- **Reserved control-plane namespace** (decided 2026-07-20; Phase 1 deliverable). The collision rule above is fragment-vs-fragment; nothing in it protects SEAM's *own* surface, which now spans `/docs`, `/docs/{route}`, `/openapi.json`, `/whoami`, `/scopes`, `/changes`, `/health/credentials`, `/health/upstreams`, and `/config/status` and keeps growing. Since fragments are authored largely by agents, a fragment declaring `/whoami` or `/config/status` is not a hypothetical: depending on a match order this plan never specified, it would either shadow SEAM's endpoint or be silently shadowed by it, and the first case is a spoofing surface the moment agents make authorization decisions from those responses. Three rules close it. **(1) The reserved set is explicit and closed:** the exact paths listed above, plus everything under the prefixes `/docs/`, `/health/`, `/config/`, and `/_seam/`. `/_seam/` is reserved *forward* — every control-plane endpoint added after this decision lives under it, so the reserved set stops accreting top-level names into the space fragments legitimately use, while the names already published to agents and CLAUDE.md (`/docs`, `/openapi.json`, `/whoami`, `/scopes`, `/changes`) are grandfathered and keep working unchanged. **(2) Control-plane routes always win the match**, unconditionally: SEAM resolves a request against its own surface before consulting the merged route table, so no fragment can shadow a control-plane endpoint even if one is somehow loaded. **(3) A fragment declaring any reserved path is rejected twice** — `seam lint` fails the CI gate in declarative-config, and the gateway quarantines the whole fragment at merge time with the offending path named at `/config/status`, never trusting lint to have run (the same belt-and-braces posture as the OpenBao allowlist in Architecture). Quarantine is fragment-wide rather than per-path so a fragment cannot smuggle a reserved declaration through alongside valid routes.
- **`seam lint`:** the gateway's own merge/validation engine packaged as a CLI, run as a CI gate in declarative-config — validates fragment schema and detects collisions — on the same (path, method, `x-api-version`) key defined above — before a ConfigMap ever ships.
- **`seam diff`:** renders the effective merged-spec change a PR would cause, for review.
- **`seam import --from-url`:** bootstraps a fragment from an upstream's own published OpenAPI spec (ArgoCD and Kubernetes both publish one), filtered to selected paths/verbs; injection metadata (`x-vault-path` etc.) is then curated by hand. Onboarding a new upstream drops from authoring YAML from scratch to curating a generated file.

## Pre-Phase-7 Trust Boundary

**All inbound caller authentication and authorization lands in Phase 7. Everything before it runs behind a network trust boundary, and that boundary is a hard requirement, not an operational detail** (decided 2026-07-20). Stating it is not belt-and-braces: by the 6b cutover SEAM is a live credential-injecting proxy — it holds the z.ai/GLM and twitterapi.io credentials that cost money per call, and (after Phase 5) reaches cluster API servers — while CLAUDE.md prose is being deleted to push agents onto it. Without this section an implementer could read the phase list and reasonably ship that as an unauthenticated open proxy on the tailnet, which is the worst outcome available: every credential SEAM holds, reachable by anything that can route to it.

- **Tag-restricted tailnet ACL, enforced in the tailnet policy file.** SEAM's Tailscale node is reachable only from an explicit, enumerated source set (e.g. `tag:needle-worker`, `tag:devpod`, the operator workstation) via a grant in the tailnet policy file. Default-deny at the network layer — not "on the tailnet, therefore allowed." Tailnet membership is not the boundary; the ACL is.
- **No Funnel, no public ingress, no Cloudflare Tunnel.** SEAM is not exposed to the public internet in any phase before 7. The foreign-worker/non-tailnet ingress path (Per-Agent Tool Scoping point 7) is explicitly a post-Phase-7 phase of its own and must not be stood up early "just for testing" — it is the one path that would put an unauthenticated credential-injecting proxy on the open internet.
- **The 6b cutover is GATED on that ACL existing.** Deploying SEAM (6a) with a permissive ACL is tolerable while nothing is cut over and no real agent traffic flows; deleting a service's CLAUDE.md prose and pointing agents at SEAM (6b) is not. The ACL grant is a checklist item in the 6b change, verified before the first service is cut over, not after.
- **Residual risk, stated plainly.** This boundary is **coarse and node-level, not per-agent**. Every caller that clears the ACL gets every route SEAM serves; a compromised or merely buggy worker on an allowed tag has the same access as any other. Guards (Phase 13) cap the *spend*, not the *reach*, and before Phase 7 they key on (route, hash) rather than per-caller — so a single misbehaving agent can consume a route-wide budget and 429 well-behaved peers. Accept this only because the alternative — deferring all of Phases 1-6 until identity exists — trades a bounded, known exposure for no delivery at all.
- **Phase 7 replaces this boundary; it does not merely add to it.** When Grant-based scope enforcement lands, per-route authorization becomes the real control and the ACL degrades to defense in depth (a coarse outer fence around a fine inner one). Nothing in Phase 7 is designed on the assumption that the ACL will still be there, and nothing before Phase 7 may be designed on the assumption that Phase 7 will arrive soon.

## Implementation Phases

**Phase numbers are stable identifiers, not execution order** (decided 2026-07-20). They are referenced from a dozen places in this document and from filed beads, so renumbering them to express sequence would break more than it fixes — and the previous convention of implying order by number had already produced a contradiction the plan flagged but never resolved (Phase 10 "should land before (or with) Phase 5"). Order is therefore stated once, explicitly, below; a phase's number says only which phase it is.

**Ship order (topological):**

`1a → 1b → 9a → 2 → 6a → 3 → 4 → 10 → 5 → 13 → 6b → 7 → 11 → 12 → 8 → 9b`

**The dependency edges that force it** — anything not implied by these is scheduling preference, free to reorder:

- `1a → 1b → everything`: nothing merges, validates, or proxies without the core.
- `1b → 9a → 3`: `seam lint` must be a live CI gate **before the first fragment ships**, not after the last. Fragments start shipping in Phase 3 and every phase through 6b adds more; landing lint at the end would mean the entire pilot and cutover ships with runtime quarantine as its only validation — exactly the outcome Fragment Toolchain's design intent ("caught before merge, not at reload") rejects. It also lands the `x-vault-path` allowlist and reserved-namespace checks *before* the surface they protect, rather than after.
- `2 → 6a → 3`: fragments arrive as mounted ConfigMaps, so the gateway must be running in the cluster before the fragment-delivery phase, not two phases later.
- `10 → 5`: multi-instance routes precede kubectl-proxy onboarding. Reversed, all nine clusters get copy-pasted single-instance fragments and Phase 10 becomes a migration of already-shipped ConfigMaps instead of a greenfield build — the N:1 collapse is the entire point of onboarding those upstreams that way.
- `4 + 13 → 6b`: the cost governor is live before agents are cut over onto metered upstreams. Phase 4's z.ai and twitterapi routes spend real money per call; 6a/4 can carry them with no agent traffic on them, 6b cannot.
- `Pre-Phase-7 trust boundary ACL → 6b`: see the section above.
- `7 → 8's per-caller breakdown`, `7 → 13's per-caller keying`, `7 → 8's Option-E drain`: identity-dependent refinements, all designed to land without changing fragment config.

**Where the gateway runs, per phase** (decided 2026-07-20 — the plan previously never said, while Phases 3-5 onboarded upstreams whose reachability is a property of rs-manager): **Phases 1a-1b, 9a and 2 run locally, off-cluster** — a developer machine or devpod, fragments read from a static local directory, OpenBao reached over the tailnet. This is what Phase 1's "provable standalone" means and it is deliberately preserved: the core must be demonstrable without a cluster. **From 6a onward SEAM runs on rs-manager**, and every subsequent phase's upstream reachability (Connectors, `x-upstream` targets) is a property of that host.

- [ ] Phase 1a: Gateway scaffold (Go, ADR-001) — HTTP server, config, container build, plus `/docs`, `/docs/{route}`, and `/openapi.json` served over a **hand-written whole spec** (no fragment merge yet), request validation with structured error responses, and the `X-SEAM-Spec-Version` response header. Genuinely unblocked today: it needs the language decision (ADR-001) and nothing else. Ships the reserved control-plane namespace as a *routing* rule — control-plane paths resolve before any route table (Fragment Toolchain rule 2).
- [ ] Phase 1b: Fragment merge — OpenAPI merge from a static local directory, per-fragment schema validation, collision detection on the (path, method, `x-api-version`) triple, quarantine + `/config/status`, and runtime rejection of fragments declaring a reserved control-plane path (Fragment Toolchain rule 3). **Gated on the route-fragment schema** (Open Question 1, bead `bf-2wt`) — that schema is the critical path, not the language choice; see ADR-001 Consequences. Still no secret injection or k8s — provable standalone.
- [ ] Phase 2: Secret injection — OpenBao client authenticating via **Kubernetes auth** with the `seam/routes/*` path allowlist and per-service co-ownership rule enforced at merge time (Architecture), `x-vault-path`/`x-inject-as` extension handling, strip-then-inject inbound header hardening, the 30-second secret cache TTL with eager 401 invalidation, and **secret-echo scrubbing of responses** — buffered scan under the size cap, incremental scan with `len(secret)-1` carry-over overlap for chunked/event-stream bodies, headers and trailers included, and refusal to inject on unscrubbable or protocol-upgraded routes absent `x-unscrubbable: acknowledged` (Credential Health Sentinel). Scrubbing ships **here, with injection** — not with the sentinel — because otherwise the plan's one core guarantee is false for every echoing upstream from Phase 2 until Phase 12.
- [ ] Phase 3: ConfigMap-mounted route fragments — projected volume, in-process file-watch hot reload with atomic route-table swap (see Architecture), first fragment (**pilot: migrate the existing hand-rolled ArgoCD read-only proxy** — unmetered and read-only, so it needs neither the cost governor nor caller auth to be safe, the lowest blast radius available for a first fragment). **Precondition: the `seam lint` CI gate (9a) is live in declarative-config** — no fragment ships through an ungated path, including this one.
- [ ] Phase 4: Onboard z.ai/GLM proxy and twitterapi.io proxy fragments (both already reachable from rs-manager — no new Tailscale Connector needed). Both are metered, so no agent is cut over onto them until Phase 13's cost governor is live (see 6b).
- [ ] Phase 5: Onboard kubectl-proxy endpoints for additional clusters as **one parametrized multi-instance fragment** (requires Phase 10), adding a Tailscale Connector per cluster on rs-manager as needed.
- [ ] Phase 6a: Deploy SEAM to rs-manager via declarative-config — single replica per Version Migration Strategy §1, OpenBao Kubernetes-auth ServiceAccount and policy, Tailscale node, and the tag-restricted ACL grant in the tailnet policy file (Pre-Phase-7 Trust Boundary). No agent traffic yet: this phase makes the gateway *reachable*, not *adopted*.
- [ ] Phase 6b: Agent cutover — cut agents over **service by service** (decided 2026-07-16); as each service's fragment goes live, its hand-written CLAUDE.md proxy prose is deleted in the same change; never a big-bang cutover. The end state flips the default from "here are your prescribed tools" to "here is your tool interface — list the routes and choose from that list": CLAUDE.md eventually carries only the SEAM endpoint pointer. **Gated on the trust-boundary ACL existing and on Phase 13 for any metered route.**
- [ ] Phase 7: Per-agent tool scoping — NEEDLE-side per-worker `tsnet` identity provisioning, `x-required-scope` route tagging, Grant-based scope enforcement at the gateway, plus the self-service surface (`/whoami`, `/scopes` — scope-filtered by default, full taxonomy at `/scopes?all=1` behind the `seam:scopes:read-all` scope — scope-filtered `/openapi.json` + `/docs` + `/docs/{route}`, and 403s naming the missing scope and the Grant snippet that would fix it). Tailnet-native Grant path only; non-tailnet callers are default-denied (Per-Agent Tool Scoping point 7). Depends on 1b, 2, 3 at minimum.
- [ ] Phase 8: Version migration tooling — `Deprecation`/`Sunset` header emission for fragments marked `x-seam-deprecated` (or per-operation `deprecated: true`), the `x-adapter` schema plus its closed transform vocabulary, the request/response transform executor in the proxy path, `seam lint` validation of adapter blocks, `X-SEAM-API-Version` selection defaulting to the **oldest** still-served version, per-route-version request-count metric, and the `/changes?since=<spec-version>` diff endpoint backed by a ring buffer of prior merged specs (see Version Migration Strategy above). Adapters are declarative fragment YAML — **never gateway Go code**, so a contract migration stays a ConfigMap edit.

  **Retirement workflow — the adopted composition** (decided 2026-07-20, adopting the recommendation in `docs/notes/quiet-window-options.md`, which this phase previously named only as "usage-gated retirement workflow" while leaving the mechanism unselected): **(D) PR-gated human retirement is the always-on gate** — when the quiet condition is met SEAM opens the declarative-config PR with the traffic evidence in the description and a human merges it; nothing is ever removed autonomously, which costs a little latency and fits the GitOps rule that all cluster changes go through declarative-config anyway. **(B over A) the window is traffic-adaptive with a calendar floor** — `max(3 × observed max inter-request gap on that route-version, 7 days)` — since a fixed window must exceed the longest inter-call gap of any live caller, a number nobody knows; the floor covers cold route-versions with too little history to self-size. This does not reopen the usage gate: zero observed traffic remains a necessary condition, and the calendar term can only lengthen the wait (see Version Migration Strategy §2). **(C) brownout 410s in the final week** — the deprecated route-version is served as a structured `410 Gone` for short scheduled windows, body naming the replacement version and pointing at `/changes`, converting "we believe nobody calls this" into a tested claim that an LLM caller can actually read and self-migrate from. This is a bounded, scheduled, pre-announced exception to the coexistence guarantee, not a hole in it: it applies only to a route-version already past its quiet threshold and already carrying `Deprecation`/`Sunset`, and a caller that appears during a brownout **cancels the retirement** rather than being sacrificed to it. **(E) post-Phase-7 upgrade to caller-complete drain** — retire once every identity that ever called v-old has either called v-new or left the tailnet, within a bounded lookback; this replaces the window approximation with the thing it approximates and shrinks retirement latency from weeks to days. Depends on 1b (fragment loading + spec-version header); the per-caller breakdown and (E) depend on Phase 7.
- [ ] Phase 9a: `seam lint` — the gateway's merge/validation engine packaged as a CLI and wired as a declarative-config CI gate. Validates fragment schema, detects collisions on the (path, method, `x-api-version`) triple, rejects any `x-vault-path` outside the `seam/routes/*` allowlist or violating per-service co-ownership, **rejects any fragment declaring a reserved control-plane path** (Fragment Toolchain rule 3 — the runtime quarantine half ships with 1b; lint is the pre-merge half), and flags every `x-unscrubbable: acknowledged` for human review. Marginal cost over 1b is CLI packaging, which is why it lands here rather than at the end.
- [ ] Phase 9b: Fragment authoring convenience — `seam diff` (renders the effective merged-spec change a PR would cause) and `seam import --from-url` (bootstraps a fragment from an upstream's own published OpenAPI spec, filtered to selected paths/verbs). Genuinely deferrable: these make fragment authoring pleasant, whereas 9a makes it safe.
- [ ] Phase 10: Multi-instance routes — `{instance}` path parameter + object-form `x-upstream-map` resolution with per-instance `vaultPath`/`injectAs`/`probeInterval`, then `_all` fan-out with instance-labeled merged results. Depends on 1b; **must precede Phase 5.**
- [ ] Phase 11: Passive route health — per-route last-2xx tracking surfaced in `/docs`, `/health/upstreams` aggregation, per-upstream circuit breaker with structured 503s. Depends on 1b and 2 (needs real proxied traffic).
- [ ] Phase 12: Credential health sentinel — `x-credential-probe` background validation loop with per-(fragment, instance) reporting at `/health/credentials`, leader-elected via a Kubernetes `Lease`, and 401-triggered refetch-and-retry-once. Depends on Phase 2. (Secret-echo scrubbing was moved out of this phase to Phase 2 on 2026-07-20 — see Credential Health Sentinel.)
- [ ] Phase 13: Per-route guards — `x-loop-guard` loop breaker, `x-cost-per-call`/`x-quota` cost governor with `X-SEAM-Budget-Remaining`, `X-SEAM-Dry-Run` validation-only mode. **Ships here without waiting for identity** (decided 2026-07-20): pre-Phase-7 the loop breaker keys on (route, request-hash) and `x-quota` is enforced as a single **route-wide** budget — strictly more conservative than per-caller enforcement, never less (see Per-Route Guards). Nothing in this phase enforces per-caller limits, and `X-SEAM-Budget-Remaining` is a fleet-wide fuel gauge, not a reservation, until Phase 7. Depends on 1b; Phase 7 narrows both keys with no fragment change and no change to what thresholds mean.

- **Route-fragment schema — narrowed, not open** (narrowed 2026-07-20; bead `bf-2wt`). ## ADR-001: 2026-07-20 — Ratify Go as the SEAM Gateway Implementation Language

### Context

Phase 1 (gateway core) has been blocked since the repo was scaffolded on 2026-07-15 — as of this writing (2026-07-20) the repo contains only `docs/`, zero implementation code. `docs/research/language-runtime-choice.md` (2026-07-16) benchmarked Go, Rust, TypeScript/Node, and Python against SEAM's seven hard requirements (3.1 fragment merge, genuine 3.1 request validation with structured field-level errors, reverse-proxy with streaming/WebSocket passthrough, OpenBao client, Tailscale WhoIs/tsnet, file-watch hot reload with atomic swap, small-footprint k8s container), with library-version and 3.1-conformance claims checked against actual registries/repos/changelogs rather than recalled from memory, and the Python findings additionally executed live on this host. The plan's Open Questions section has carried "decision pending review" since that research landed, and every phase from 1 onward depends on it. This ADR closes that gap.

### Decision

Adopt **Go** as the SEAM gateway's implementation language, per the research's primary recommendation.

1. **The validator is the product.** SEAM's differentiating feature is schema-validated, structured, field-level 400 responses. pb33f `libopenapi` + `libopenapi-validator` is the only mature, off-the-shelf OpenAPI 3.1 request validator found across all four languages surveyed, and its error model (JSON pointer + spec line/col + `HowToFix`) maps directly onto SEAM's structured-error feature — it is better than what we would hand-design. Every other language means either building this by hand (Rust) or accepting a wiring/bus-factor risk on SEAM's core value proposition (Python, Node).
2. **Tailscale support is first-party and production-grade in Go alone.** `tsnet` (embedded node) and `client/local` (WhoIs against a sidecar) are maintained by Tailscale itself, directly relevant to the pilot phases and to Phase 7's per-agent Grant-based scoping. The Rust equivalent (`tailscale-rs`) is explicitly experimental with no WhoIs API as of this research.
3. **The proxy layer's security posture comes free.** stdlib `httputil.ReverseProxy`'s `Rewrite` hook strips hop-by-hop and client-spoofable `X-Forwarded-*` headers before user code runs, by design — closing the exact secret/header-injection pitfall flagged in `docs/research/secret-injection-gateways.md`, rather than requiring SEAM to reimplement that hardening itself.
4. **The cost is real but bounded and one-time.** Go is a new toolchain in a shop whose CI is otherwise Rust-centric (`rust-verify` on iad-ci), but that cost is paid once at setup, not per-route the way a hand-rolled validator's maintenance would recur.

### Alternatives Considered

- **Rust** (the shop's primary language; `rust-verify` remote CI already exists) — clears six of seven requirements with best-in-class parts (axum/hyper/tower proxy stack maps cleanly onto validate→strip→inject→forward; `jsonschema` 0.48 is the most actively maintained 2020-12 engine surveyed; best footprint at ~10-25MB). The one gap is the product's core feature: no mature off-the-shelf 3.1 request validator exists (the sole candidate is a 45-star crate with no crates.io release). Building one is a scoped, bounded effort (~1-2k LOC, 1-2 weeks, per the research) — a defensible runner-up if "no new language in this shop" is weighted above "don't own the validator," but it puts SEAM's differentiator on our own maintenance budget instead of an actively-maintained upstream's. Kept as the fallback (see Consequences).
- **Python** — every requirement clears, and uniquely among the four candidates was verified by live execution on this host (openapi-core 0.23.1: 0.59ms/request validation, 16ms full spec rebuild; a real WhoIs call over `tailscaled`'s Unix socket). Ruled out on footprint (170-200MB image vs. Go's 15-30MB) and on openapi-core being a single-maintainer dependency with a documented history of breaking 0.x API churn — a bus-factor risk on the validator, the same category of risk this decision is trying to avoid.
- **TypeScript/Node** — Ajv2020 is the most battle-tested 2020-12 schema engine surveyed, but every library wrapping it for OpenAPI made an overstated 3.1 claim under inspection (openapi-backend silently runs draft-07-mode Ajv despite its README's 3.1 checkbox; express-openapi-validator is Express-bound and drags in `multer`). WebSocket proxy support across the viable options is self-described "partial" (first subprotocol only). No footprint advantage over Go to compensate.

### Consequences

- **The LANGUAGE blocker is closed; Phase 1 is not wholly unblocked** (amended 2026-07-20 — this bullet previously read "Phase 1 (gateway core) is unblocked and can start as a Go scaffold", which overclaimed: this ADR resolved the language question and was then read as having resolved Phase 1's blocker generally). Phase 1's original deliverable included fragment merge, validation, and quarantine — none of which can be built against a fragment schema that does not exist, which is why bead `bf-4ks` ("Scaffold Phase 1 gateway core in Go") remained BLOCKED while this ADR asserted the opposite. Phase 1 is therefore split (see Implementation Phases): **Phase 1a — Go scaffold, HTTP server, `/docs` + `/docs/{route}` + `/openapi.json` over a hand-written whole spec, structured validation errors — is genuinely unblocked by this ADR and can start now.** **Phase 1b — fragment merge, collision detection, quarantine — remains gated on the route-fragment schema (Open Question 1, bead `bf-2wt`), which is now the critical path for the project.** `seam lint`/`seam diff`/`seam import` (Phases 9a/9b, Fragment Toolchain) are Go CLI tools by the same extension, and 9a inherits 1b's gate for the same reason.
- SEAM becomes the second language in this shop's container fleet alongside Rust. The existing `cargo`/`cargo-remote` CI-offload path (`~/.local/bin/cargo`, iad-ci `rust-verify`) does not apply; a Go build path is needed once code exists — tracked as a follow-on bead (see beads filed alongside this ADR), not a blocker to starting Phase 1, since SEAM ships as a plain container per the Hosting section regardless of build tooling.
- The validator boundary is narrow (parse + merge + validate, per the comparison matrix in `docs/research/language-runtime-choice.md`) — if Go proves to be the wrong call in practice, replacing that one component (or, worst case, the whole gateway in Rust per the runner-up analysis) is a bounded rewrite, not a redesign of SEAM's architecture.
