# SEAM Plan

## Overview

SEAM (Self-documenting Endpoint Access Mediator) is a single unified endpoint that acts as a proxy to multiple authenticated services. It injects the required secret into a request received from an agent, sends the request on to the destination service, and passes the response back — the agent never sees the secret. It also self-documents: a merged OpenAPI spec drives `/docs` and `/openapi.json`, and malformed requests get a structured error naming the bad field and pointing at the right shape, instead of a bare 400.

## Motivation

Today, each backend service (ArgoCD read-only proxy, per-cluster kubectl-proxy, the z.ai/GLM proxy, the twitterapi.io proxy) is documented by hand in CLAUDE.md with its own discovery and auth pattern. This doesn't scale: every new service needs new hand-written prose, agents have no standard way to discover what's available, and there's no mechanism to guide an agent that sends a malformed request back toward the correct one.

## Architecture

- Single gateway process (reverse proxy) in front of N upstream services.
- Route table = one OpenAPI spec, assembled at startup (and reloaded on change) by merging multiple fragment files. Each fragment is an OpenAPI paths/components fragment plus custom extension fields — `x-upstream` (the base URL to forward to), `x-vault-path` (which OpenBao secret to fetch) and `x-inject-as` (how to inject it — an object naming both the kind, header / bearer / query, and the header or parameter *name*; see Data Models) — before proxying upstream.
- Route fragments are delivered as Kubernetes ConfigMaps, one per upstream service, mounted into the gateway pod via a `projected` volume at `/etc/gateway/routes.d/*.yaml`. Each fragment is owned and synced independently from its own service's path in declarative-config — no single monolithic file to edit for every new route.

  **Where fragments live in declarative-config, exactly** (decided 2026-07-20 — the owner chain that carries the whole anti-exfiltration control starts at "the declarative-config directory name", and this plan named that directory only as "its own service's path" and "`<svc>/`" — neither of which is a location. The repo's layout is `k8s/<cluster>/...`, so `<svc>/` was ambiguous between a per-cluster path, a top-level `seam/` path, and the service's own ArgoCD application directory — and `seam lint`, whose distinguishing value is that it is the only place the whole chain is visible at once, cannot walk a chain whose first link has no address). The location is fixed: **`k8s/rs-manager/seam/routes/<svc>/fragment.yaml`** — one directory per service, exactly one `fragment.yaml` in it, generating ConfigMap `seam-routes-<svc>`; and **SEAM's own Deployment, Service, `seam-upstream-ca` ConfigMap and upstream-host allowlist live one level up at `k8s/rs-manager/seam/`**, outside every `routes/<svc>/` directory. That separation is the point rather than a filing convenience: the projected `path: <svc>.yaml` rename and the upstream-host allowlist are the two operator-owned links a fragment author must not be able to edit, so they sit in a directory no fragment author has business touching, while `routes/<svc>/` is the only place that author writes. `k8s/rs-manager/` because SEAM runs on rs-manager from Phase 6a (Hosting); a fragment describing an upstream *in* another cluster still files here, since the directory names the pod that mounts it, never the cluster it talks to.

  **`<svc>` is SEAM's own owner token and need not equal the service's ArgoCD app directory.** The upstream service typically has its own application directory elsewhere in declarative-config (`k8s/<cluster>/<app>/`), which is where that service is *deployed*; `k8s/rs-manager/seam/routes/<svc>/` is where its SEAM route contract is *published*, and the two are separate PRs against separate paths with separate reviewers. Where the names can match they should, purely for legibility, but nothing depends on it: `<svc>` is bound by the projected filename and `x-seam-owner` and constrains only the `seam/routes/<svc>/*` OpenBao prefix, so co-ownership is well-defined even for an upstream that has no declarative-config presence at all (twitterapi.io, z.ai). `<svc>` matches `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$` — one flat segment, no slashes, no dots — so it is simultaneously a legal directory name, a legal ConfigMap name suffix, a legal filename stem, and a legal OpenBao path segment, and the chain cannot be broken by a name that is valid at one link and not the next.
- Route reloads are **in-process and sub-second** (decided 2026-07-16, superseding the earlier restart-via-`reloader` idea): the gateway file-watches `/etc/gateway/routes.d/`, re-merges and re-validates on change, and atomically swaps the route table — in-flight requests finish against the old table, new requests see the new one, zero dropped connections. With dozens of concurrent workers hitting this authentication layer, restart-based reload would be the disruptive option; no blue/green machinery or Traefik-side intelligent routing is needed for config changes, because nothing about the pod changes. Kubelet propagates projected-ConfigMap updates on its sync period (git→pod latency up to ~1 min after ArgoCD sync); if that latency ever matters, the upgrade path is watching ConfigMaps via the Kubernetes API (informer) instead of file mounts. Gateway **binary** deploys are the narrow exception, and they are *not* zero-downtime (corrected 2026-07-20 — this clause previously called SEAM "a stateless proxy that already provides blue/green's guarantee without running a second fleet", pre-amendment text the single-replica decision invalidated). SEAM is emphatically **not** stateless: guard counters, quota ledgers, last-2xx, and breaker state are per-process and load-bearing enough to forbid a second replica, so Version Migration Strategy §1 mandates `replicas: 1` with `maxSurge: 0` — a rollout never runs two SEAM pods at once and therefore has no overlap at all. A binary deploy therefore opens a **real gap, on the order of seconds to tens of seconds** — the sum of `terminationGracePeriodSeconds`, in-flight drain, pod scheduling, image pull, process start, initial merge and OpenBao login, and the first passing readiness probe, **typically 5-30 seconds** (corrected 2026-07-20; this clause previously said "sub-second", which was wrong by one to two orders of magnitude). **Version Migration Strategy §1 owns that sizing and the caller requirement it implies** — any agent calling SEAM must retry with backoff over at least 60 seconds. The gap is nonetheless cheap precisely because it is rare: **route and contract changes are ConfigMap edits that never restart the pod**, so the hot-reload path above — not a deploy strategy — is what carries the traffic that actually changes.
- `/docs` serves a human-readable rendering of the merged spec (Redoc/Swagger UI); `/openapi.json` serves the raw merged spec for agents to fetch programmatically.
- Incoming requests are validated against the merged spec. A malformed request gets a structured 400 naming the bad field, what was expected, and a link to `/docs/{route}`.
- **`/docs/{route}` is a real server-side endpoint, not a `/docs` anchor** (decided 2026-07-20; Phase 1a deliverable). The pointer carried by every structured 400 is the affordance the Overview names as SEAM's differentiator, and its consumer is an agent, not a browser — so it must resolve to something fetchable in one request. A Redoc/Swagger UI section anchor (`/docs#/paths/...`) cannot serve that role: URL fragments are client-side only, never sent to the server, and no server-side path template can produce one. SEAM therefore serves **`GET /docs/route?path=<openapi-path-template>`**, with optional `&method=` and `&version=` selectors — e.g. `GET /docs/route?path=/k8s/{cluster}/api/v1/pods&method=GET&version=v2` — defaulting to every method declared on that path. **The route selector is a query parameter, not a path segment** (corrected 2026-07-20). It was previously specified as `GET /docs/{route}` with `{route}` percent-encoded into a single segment (`/docs/%2Fk8s%2F%7Bcluster%7D%2Fapi%2Fv1%2Fpods`), which cannot survive the stack this plan actually chose: Go's `net/http` mux — and most routers, and Traefik, which fronts several fleet services and is named below — normalize and clean request paths before matching, decoding `%2F` to `/`, so that URL arrives as the different, multi-segment path `/docs/k8s/{cluster}/api/v1/pods` and never matches. Encoded slashes in a path segment are a well-known failure mode there, and this endpoint is a Phase 1a deliverable and the target of every structured 400, every loop-breaker 429, and every `docsUrl` in `/changes` — so a normalization failure would break the plan's stated differentiator on day one. A query parameter is normalization-safe: nothing in the chain rewrites a query value, `/` and `{}` need no special treatment there, and the endpoint's own path is a fixed two-segment literal every router matches trivially. Elsewhere in this plan and in the sibling notes this endpoint is still referred to by the shorthand **`/docs/{route}`**; that name denotes *this* endpoint, and this bullet is the authority on its wire shape. The response is that route's slice of the merged spec and nothing else: parameters, request and response schemas, `x-required-scope`, any guard annotations in force, and one concrete worked example request. It is `application/json` by default (the agent path) and `text/html` under content negotiation, where it redirects to the corresponding `/docs` anchor for a human. 
  **Which version it shows** (decided 2026-07-20; resolves the disagreement with Version Migration Strategy §2, where `/docs` advertises the *newest* version as canonical while unversioned requests are *served* the oldest). Every SEAM-generated pointer at this endpoint carries an explicit `&version=` equal to the version that actually handled the request — the same value echoed in that response's `X-SEAM-API-Version`. So a structured 400's pointer, and the loop-breaker 429's, always render the contract in force **for that caller**, which is the whole reason the pointer exists: a pointer that showed a different contract from the one rejecting the request would manufacture the confusion it was added to prevent. A **bare** `/docs/route?path=<p>` with no `&version=` resolves by §2 — the oldest still-served version, i.e. what this caller would get if it sent no `X-SEAM-API-Version` — and says so prominently in the response rather than leaving it inferred: the payload carries `version`, `isDefaultForUnversionedCallers: true`, `canonicalVersion` (the newest), `deprecated`, and a `versions[]` list of every coexisting version with its own `docsUrl`. `/docs` and `/openapi.json` keep advertising newest-as-canonical unchanged; the two surfaces no longer disagree because this one now states which of the two it is showing.

  **When each half of that contract lands** (decided 2026-07-20; the split is built in Phase 1a and completed in Phase 8). Before Phase 8 there is exactly one served contract, keyed `_unversioned`: `/docs/route` takes `?path=`/`?method=`, emits `version: _unversioned` and `isDefaultForUnversionedCallers: true`, and omits `canonicalVersion`, `deprecated` and `versions[]`, which are Phase 8 additions. Pointers carry `&version=_unversioned` from Phase 1a — there is no bare-pointer form in any phase.

  **Scope filtering discloses nothing a caller cannot already see** (decided 2026-07-20). It is scope-filtered exactly like `/openapi.json` (Per-Agent Tool Scoping point 5), and the filter is applied *before* the existence check: a route outside the caller's filtered spec returns **404, identically to a route that does not exist**. The earlier "404 for unknown, 403 naming the missing scope for a route you may not invoke" was an enumeration oracle — a caller could probe path templates one request at a time and reconstruct the withheld route inventory of every service SEAM fronts, which is exactly the reconnaissance that filtering `/openapi.json` and gating `/scopes?all=1` exist to withhold. The same rule binds level-2 `/changes?since=&route=` (see Version Migration Strategy §3, whose 403-for-out-of-grant-route text this supersedes). The scope-naming 403 is **not** abolished, only narrowed to where it leaks nothing: it applies when the caller can already see the route in its own filtered spec but lacks the scope for the specific method or guard it is attempting — the visible-but-not-invocable case, which is real (a caller holding `k8s-ro:get` sees the path and is refused `DELETE`). That preserves the plan's "authorization debugging becomes ask the gateway" value for every route an agent legitimately knows about. For a route it does *not* yet know about, the sanctioned path is a scope, not an oracle: `/scopes?all=1` behind `seam:scopes:read-all`, plus `/whoami`. The 404 body therefore names `/whoami` and the caller's `X-SEAM-Scope-Version` and states that an unreachable-but-believed-real route is an operator request; SEAM logs the resolved route server-side on every such 404 so the question is answerable off the request path.

  Because the structured 400 (Architecture), the loop-breaker 429 (Per-Route Guards), and level 1 of `/changes` all point at it, it ships in **Phase 1a** alongside `/docs` and `/openapi.json`, not in a later phase. Note that a level-1 `/changes` entry carries **two distinct links, not one** (decided 2026-07-20 — Version Migration Strategy §3 and `docs/notes/versioning.md` §C call the level-1 field a "drill-down link" and define level 2 as the field-level diff, while this bullet previously claimed "the level-2 drill-down" pointed here; they answer different questions and both are wanted): **`diffUrl`** → `/changes?since=<version>&route=<path>`, the level-2 field-level schema diff — *what changed*; and **`docsUrl`** → `/docs/route?path=<path>&version=<v>`, this endpoint — *what the contract now is*. It is `docsUrl` that points here. That is the canonical `docsUrl` shape everywhere it appears in this plan and in `docs/notes/versioning.md` §C; the older `/docs/{route}?version=<v>` spelling is superseded by the query-parameter correction above.
- Secret injection reads from OpenBao (the central secrets store — the 4th instance runs on rs-manager) at request time, and never surfaces secret values to the calling agent.
- **OpenBao authentication and secret-path allowlist** (decided 2026-07-20; Phase 2 deliverable). SEAM authenticates to OpenBao via **Kubernetes auth** — the pod's projected ServiceAccount token is exchanged for a short-lived OpenBao token, renewed in-process; no static token, AppRole secret-id, or credential file on disk (AppRole is the fallback only for a future non-Kubernetes deployment). Critically, because fragments are authored largely by agents, **the fragment must not be able to name an arbitrary secret**: SEAM's OpenBao policy grants `read` on a single bounded prefix, `seam/routes/*`, and nothing else — so even a hostile fragment cannot reach `armor/`, `kalshi/`, cluster kubeconfigs, or any other tenant's material, because SEAM's own token has no capability there. The allowlist is enforced twice: `seam lint` rejects any `x-vault-path` outside the prefix at CI time, and the gateway re-checks it at merge time and quarantines the fragment (never trusting lint to have run). Additionally, a fragment's `x-vault-path` must be **co-owned with its upstream target**: a fragment shipped from declarative-config service directory `<svc>/` may only reference `seam/routes/<svc>/*`, so the exfiltration shape "point service A's secret at service B's upstream" fails on ownership even inside the allowlist. Path traversal, globs, and templated segments in `x-vault-path` are rejected outright.

  **How the gateway knows `<svc>` at runtime** (decided 2026-07-20). The co-ownership rule was previously declared enforceable twice — at lint time and again by the gateway "never trusting lint to have run" — but the gateway half was unimplementable as written: at runtime SEAM sees only files at `/etc/gateway/routes.d/*.yaml`, and nothing in that projection carried the originating declarative-config directory, leaving lint (which this design explicitly refuses to trust) as the sole enforcement of the anti-exfiltration control. The owner is therefore made a **first-class, projection-preserved property** in two mutually-checking forms:
  - **The projected filename is authoritative.** Each service's fragment ships as ConfigMap `seam-routes-<svc>` from declarative-config directory `<svc>/`, and SEAM's `projected` volume mounts each source with an explicit `items: [{key: fragment.yaml, path: <svc>.yaml}]`. The gateway derives `<svc>` from the basename of each file it loads, and enforces `x-vault-path` ⊂ `seam/routes/<svc>/*` against *that*. This is the forgery-resistant half: the `path:` rename lives in **SEAM's own Deployment manifest**, not in the service's directory, so a fragment author editing only `<svc>/` cannot change the name their content is filed under, cannot smuggle a second key into another service's slot, and cannot make their file land as `otherservice.yaml`. Changing the binding requires a PR against SEAM's own manifest, which is exactly where that decision belongs.
  - **`x-seam-owner: <svc>` is required in every fragment and must match.** A required in-file owner field on its own would be worthless — a hostile fragment author simply lies in it — so it is never the authority, only a redundant declaration checked *against* the filename. A fragment whose `x-seam-owner` disagrees with its projected basename, or that omits the field, is quarantined whole (never partially loaded) with the mismatch named at `/config/status`. The value is that it makes misfiling loud rather than silent, gives `seam lint` something to check in CI before any volume exists, and turns a mounting mistake in SEAM's own manifest into a quarantine instead of a privilege grant.

  `seam lint` checks the full chain — directory name → ConfigMap name → projected `path:` → `x-seam-owner` → `x-vault-path` prefix — and the gateway independently re-checks the last three at merge time from what it can actually observe. Belt and braces are now both real.
- **Upstream-host allowlist — the peer control to the vault-path allowlist** (decided 2026-07-20; Phase 2 deliverable). Everything above constrains **which secret** a fragment may name and nothing at all constrains **where that secret is sent**: `x-upstream` and every `x-upstream-map` entry `url` were unbounded. That leaves the whole anti-exfiltration design guarding the wrong half. Co-ownership defeats "point service A's secret at service B's upstream", but the exfiltration that actually matters is simpler and passes every check written so far: a fragment filed correctly as `<svc>.yaml`, declaring `x-seam-owner: <svc>` and `x-vault-path: seam/routes/<svc>/token` — its own secret, its own slot, no rule broken — and `x-upstream: https://attacker.example`. SEAM fetches the live credential and injects it into a request to that host, exactly as designed. **None** of the plan's other controls touch this: response scrubbing is response-side and the attacker needs no response; TLS verification passes, since an attacker controls a host with a perfectly valid certificate; `x-upstream-plaintext` is irrelevant because `https://` is the *preferred* form here. Since fragments are authored largely by agents, this is the shortest path from "hostile fragment merged" to "credential in an attacker's log", and it must be closed by the same posture as the vault prefix:
  - **The allowlist is an operator-owned set of permitted upstream hosts and suffixes, and it lives in SEAM's own Deployment/manifest — never in a fragment.** This is the load-bearing half. A fragment-supplied allowlist is self-certifying and therefore worthless: the author who names the malicious host also names the rule permitting it. Like the projected `path:` binding and the `seam-upstream-ca` ConfigMap above, changing it requires a PR against SEAM's own manifest, which is exactly where that decision belongs.
  - **What it contains**: the tailnet suffixes `*.ts.net` and `*.ardenone.com`, the in-cluster suffix `*.svc.cluster.local` (plus bare in-cluster Service DNS), and **explicitly enumerated public hosts, one line each** — `api.twitterapi.io` and the z.ai/GLM proxy host being the Phase 4 entries. Suffix matching only, on the resolved host of the URL, never on a substring or a regex; a port may be constrained alongside a host. IP-literal upstreams are rejected outright, since they defeat the enumeration the allowlist exists to provide. **Onboarding a new public upstream is therefore a two-PR operation** — one against SEAM's manifest to permit the host, one against the service's directory to ship the fragment — and that friction is the control, not a cost of it.
  - **Enforced twice, exactly as the vault prefix is.** `seam lint` rejects any `x-upstream` or `x-upstream-map` entry `url` whose host is outside the allowlist at the CI gate, and **the gateway re-checks every resolved upstream host at merge time and quarantines the whole fragment on violation** — never trusting lint to have run, and never partially loading a fragment whose other routes happen to be fine. `/config/status` names the offending host. There is no per-fragment opt-out and no `acknowledged` form: unlike `insecureSkipVerify` or `x-unscrubbable`, which make a *known* exposure enumerable, an unlisted upstream host has no legitimate case that a manifest edit cannot express, and a per-fragment override would hand the exfiltration path straight back to the fragment author.
- **Inbound header stripping is a requirement, not an implementation detail** (decided 2026-07-20; Phase 2 deliverable). Before injection, SEAM **strips** every header it may itself set, rather than overlaying values onto whatever the caller sent: the header named by each matched route's `x-inject-as.name` (see Data Models), `Authorization`, and all `X-SEAM-*` request headers except the documented caller-supplied ones — `X-SEAM-Dry-Run` (Per-Route Guards) and `X-SEAM-API-Version` (Version Migration Strategy §2). Comparison is on the canonicalized header name (case-insensitive, per-key delete of *all* values, not a single-value overwrite), so `authorization`, `AUTHORIZATION`, and duplicate keys are all removed. This is strip-then-inject, never trust-then-overlay — an attacker-supplied `Authorization` or a custom header matching some other route's `x-inject-as` must never survive into an upstream request.

  **The rule covers the query string too, not headers only** (extended 2026-07-20 — it previously spoke of headers exclusively, which silently exempted the one injection kind where the caller controls the same namespace SEAM writes into). For a route whose `x-inject-as.kind` is `query`, SEAM **deletes every occurrence of the injected parameter name from the request query string before injecting its own** — all repetitions, not the first, since duplicate keys are legal in a query string and upstreams disagree wildly on which one wins (first, last, or an array). Without this, a caller could send `?api_key=<theirs>` and, depending on that upstream's merge order, either override SEAM's injected credential or have theirs forwarded alongside it — precisely the trust-then-overlay failure this rule exists to forbid, just relocated from the header namespace to the query namespace. Deletion is by exact parameter name (query keys are case-sensitive, unlike headers) and applies to the names declared by **any** matched route's `x-inject-as`, not just the credential kind actually in play on this request. Same treatment for the fan-out and adapter paths: the strip happens on the inbound request, before the adapter's request transform can rename a caller-supplied parameter into the injected one. Note that Go's `httputil.ReverseProxy` `Rewrite` hook removes only hop-by-hop and `X-Forwarded-*` headers (ADR-001 §3); everything in this bullet is SEAM's own responsibility on top of that.
- **Secret caching policy** (decided 2026-07-20; Phase 2 deliverable): secrets are held in an in-memory cache with a **short 30-second TTL**, never rendered to a file or held for the process lifetime. The cache is invalidated eagerly on an upstream `401` (which triggers refetch-and-retry-once, see Credential Health Sentinel), and TTL expiry forces a refetch otherwise. Rotation latency is therefore bounded at 30s worst case for a silent rotation, and effectively zero for a rotation the upstream signals by rejecting the stale credential — the point being that an OpenBao rotation takes effect without redeploying or restarting SEAM. The cache is keyed by resolved `x-vault-path`, so instances sharing a path share one entry; values are never logged, never written to disk, and never included in `/config/status` or any health surface.
- **The request/response pipeline, in order** (decided 2026-07-20; stage numbering revised 2026-07-20). Operations this plan specifies independently — header and query stripping, identity resolution, scope authorization, spec validation, dry-run short-circuit, adapter transforms, guard evaluation, secret injection, the upstream call, secret-echo scrubbing — compose into **eleven inbound stages and four outbound**, and **order determines behavior**, so it is fixed here once rather than left to the implementer. **The two numbered lists below are the authority for stage numbering**: every "stage N" reference elsewhere in this plan means a position in these lists, and no other count of these operations is normative. An implementation may not reorder these stages.

  **Request (inbound):**
  1. **Control-plane resolution** — SEAM's own surface resolves before the merged route table, unconditionally (Fragment Toolchain rule 2).
  2. **Inbound sanitation** — strip-then-inject hardening: delete every injectable header and every injectable query parameter, plus `Authorization` and all `X-SEAM-*` except the documented caller-supplied `X-SEAM-Dry-Run` and `X-SEAM-API-Version`. **First, always** — nothing downstream may ever read a caller-supplied value in a namespace SEAM writes into.
  3. **Caller identity — resolution only** — resolve *who* is calling (tailnet Grant via `WhoIs`, or verified Cloudflare Access JWT), and default-deny a request whose identity cannot be resolved at all. **Authorization is deliberately not done here** (split out of this stage 2026-07-20): `x-required-scope` is a *per-route* field, and on an adapting fragment it is the **union of the matched fragment's and its `targetVersion`'s** requirements (Version Migration Strategy §2) — so it is not computable until both the fragment and its adapter target are known, which is stage 4's job. Splitting identity out keeps the property the merged stage was reaching for: identity is established before any secret lookup or upstream contact, and nothing below runs for an unidentified caller.
  4. **Route and version match** — (path, method), then version selection per §2 (`X-SEAM-API-Version`, else oldest still-served). This picks the *fragment*, and therefore the contract everything below is relative to. For a fragment carrying `x-adapter`, it also resolves the `targetVersion` fragment the adapter delegates to, since stages 5, 8 and 9 all need it.
  5. **Authorization** — evaluate `x-required-scope` against the identity resolved at stage 3, default-deny. On an adapting fragment this is the **union** of the matched fragment's and the target version's requirements, which is why it cannot precede stage 4 — union is the only direction that cannot be gamed, since otherwise pinning an old `X-SEAM-API-Version` would be a privilege-de-escalation path (§2). Still before any secret lookup, adapter transform, guard draw, or upstream contact. Scope-filtered visibility applies here too: a route outside the caller's filtered spec is a **404 identical to a route that does not exist**, and only the visible-but-not-invocable case gets the scope-naming 403 (see the `/docs/route` rule above).
  6. **Spec validation — against the matched version's own declared schema**, not the newest one. This is why validation precedes the adapter and is nonetheless correct: an old-contract request is validated against the old fragment's schema, which is exactly the contract that caller was served and the only one it can be held to. Failure is the structured 400, whose `/docs/route` pointer carries this same version.
  7. **Dry-run short-circuit** — `X-SEAM-Dry-Run: 1` returns the validation verdict here and stops. No guard counter is incremented, no quota drawn, no secret fetched, no upstream contacted.
  8. **Adapter request transform, and re-validation against the target's schema** — the matched older fragment's `x-adapter.request` converts the validated old-contract request into the newest contract, the one the upstream actually speaks, and the transform's output is then re-validated against the *target* version's request schema. A failure is a **500 attributed to the adapter**, never a 400 (§2). A request on a fragment carrying no `x-adapter` passes through this stage unchanged. **This precedes guards** (corrected 2026-07-20 — it was specified after them, which was unbuildable): both guard rules take this stage's output as input. The loop breaker's normalized request hash is computed **after** the request transform, so two callers on different `x-api-version`s sending semantically identical requests count as one loop; and guards are charged to the **target** version's route, which only adapter resolution makes known (both from §2). A guard stage placed before the adapter could satisfy neither.
  9. **Guards** — loop breaker, then cost governor. After validation so a malformed request is a 400 rather than a budget draw, after the adapter so the hash and the charged route are the target's, and before injection so an over-budget request never causes a secret to be read out of OpenBao. Moving the adapter above guards costs nothing that argument depended on: **re-validation still concludes before secret resolution**, so a broken transform still can never cause a credential to be injected into a malformed upstream request — injection simply now follows two stages later instead of one.
  10. **Secret resolution and injection** — cache or OpenBao fetch, then inject per `x-inject-as` into the now-sanitized request.
  11. **Upstream call.**

  **Response (outbound):**
  1. **Upstream response received.**
  2. **Adapter response transform** — `x-adapter.response` converts the newest-contract response back into the contract this caller is on.
  3. **Bookkeeping and SEAM headers** — last-2xx and breaker state, budget decrement, `X-SEAM-Spec-Version` / `X-SEAM-Scope-Version` / `X-SEAM-API-Version`, `Deprecation`/`Sunset`.
  4. **Secret-echo scrubbing — last.**

  **The invariant: scrubbing is the final operation applied to a response body, headers, and trailers before they leave SEAM.** Nothing may run after it. Placing the adapter's response transform *after* scrubbing would let an envelope unwrap or a re-encode relocate or resurface a value the scrubber had already passed over, converting the plan's one core guarantee ("the agent never sees the secret") into a function of transform authorship. Two consequences follow and are accepted deliberately:
  - The chunked scrubber's `len(secret) - 1` carry-over overlap (Credential Health Sentinel) operates on the **transform's output stream**, not the raw upstream bytes. That is fine for a pass-through stream, which is byte-identical.
  - But a body-restructuring adapter transform (envelope wrap/unwrap, field move) **cannot** operate on a stream, so any route whose `x-adapter.response` alters body structure **forces the buffered response path** and is subject to its size cap. A route that needs a body transform *and* cannot be buffered — a protocol upgrade, an oversized or opaque body — is unscrubbable by construction: injection is refused there unless the fragment carries `x-unscrubbable: acknowledged`, exactly as for any other unscannable response. `seam lint` flags this combination specifically, since it is easy to author unknowingly.

  **Where an `_all` fan-out sits in these stages** (decided 2026-07-20 — the lists above are written in the singular throughout, and a fan-out resolves N instances, fetches up to N distinct per-instance secrets, injects under up to N distinct `injectAs` shapes, issues N concurrent upstream calls and consults N breakers; leaving the split point unstated leaves an implementer to invent whether guards run once or N times and whether scrubbing sees one body or N). Stages 1-9 run **exactly once, on the single inbound request**, against the aggregate — identity, authorization, validation, dry-run, adapter transform and both guards are properties of the request the caller sent, not of the instances it happens to reach. The fan-out is the boundary between stage 9 and stage 10:

  - **Guards are evaluated once, against the aggregate**, with the cost governor drawing **per instance actually dispatched** rather than per request — the accounting rule Per-Route Guards already states, restated here as a stage placement: quota is checked once at stage 9 against the *projected* draw (the number of instances that will survive breaker filtering), and instances the breaker refuses are charged nothing. The loop breaker likewise counts the inbound request once; `_all` is one request in its normalized hash, not N. A fan-out route's `x-quota` must therefore be sized against the map's width.
  - **Stages 10 and 11 execute per surviving instance**, concurrently: each instance resolves its own `vaultPath` (so N distinct secrets may be fetched, from N distinct cache entries) and injects under its own `injectAs` kind and name, into its own copy of the sanitized request, over its own `x-upstream-map` entry `url` and per-instance `tls` policy. Instances whose breaker is open are skipped before stage 10 and never reach a secret lookup at all.
  - **Outbound stages 1 and 2 run per instance** — each instance's response is received and adapter-transformed on its own — and the **merge into the instance-labeled result is outbound stage 3's bookkeeping**, alongside the per-instance breaker and last-2xx updates.
  - **Outbound stage 4 — scrubbing — runs exactly once, over the merged body, headers and trailers, against the UNION of every value injected on this request.** Not per instance: a merged body is a single artifact that may relocate or concatenate any instance's echoed credential into a region the per-instance scrubber never saw, and the invariant above is that *nothing runs after scrubbing*. Scanning the merge against the union is the only ordering that keeps that true. This is the same rule the chunked scrubber already states — "where multiple secrets could be present, the overlap is the longest of them" — with the union now explicitly spanning instances rather than just the multiple secrets of one route. A fan-out in which any single instance is unscrubbable makes the whole fan-out unscrubbable, and is refused absent `x-unscrubbable: acknowledged` on the fragment.
- **Upstream TLS: verified by default, and plaintext is an acknowledged exception** (decided 2026-07-20; Phase 2 deliverable). SEAM injects a live credential into every forwarded request, so the transport carrying that request is part of the injection guarantee — yet nothing here previously said a word about certificate verification, and the fleet's own proxies are routinely reached with self-signed certs (`curl -sk https://argocd-ro-...:8444`). An implementer left to infer the rule copies the worked example, ships credentials over plaintext, meets a self-signed cert, and reaches for a blanket skip-verify. Three rules instead:
  - **Verify by default.** Every `https://` upstream is verified against the system trust store with hostname checking on. There is no global skip-verify switch and no environment variable that disables verification fleet-wide — a per-fragment, per-route, enumerable opt-out is the only form this takes.
  - **Self-signed upstreams pin a CA, they do not skip verification.** `x-upstream-tls.caBundle` (see Data Models) names a PEM bundle key in SEAM's own `seam-upstream-ca` ConfigMap — SEAM-manifest-owned, like the fragment `path:` binding above, so a fragment author cannot introduce trust material by editing only their own directory. `x-upstream-tls.serverName` overrides SNI/hostname verification where the cert's name differs from the dialed host. This is the intended answer for every `-k` in today's CLAUDE.md prose, and it is strictly better than what it replaces.
  - **`x-upstream-tls.insecureSkipVerify: acknowledged`** is the last resort, and — in the style of `x-unscrubbable` — `acknowledged` is the **only** accepted value, with no `false` form, because the field exists to make the exposure enumerable and an enumerable exposure cannot be expressed by a field nobody has to write. `seam lint` flags every occurrence for human review; `/config/status` enumerates every route currently forwarding a credential over an unverified connection.

  **Plaintext `http://` carve-out: it exists, narrowly, and it is not free** (decided 2026-07-20 — stated because the plan's other examples use plaintext tailnet and in-cluster URLs, so silently forbidding it would be ignored, and silently permitting it would ship credentials in the clear). Plaintext is acceptable **only** for a hop that never leaves an already-authenticated network path — a tailnet-internal address (WireGuard-encrypted end to end) or an in-cluster Service DNS name — and never for anything reachable off it. It is not inferred from the URL's shape: a plaintext `x-upstream` must carry `x-upstream-plaintext: acknowledged`, on the same only-accepted-value terms, is lint-flagged, and appears in the same `/config/status` enumeration. The point is that "it's just the tailnet" remains a defensible engineering call, but it becomes a decision someone wrote down and can be asked about, rather than a default nobody chose.

## Data Models

- Route fragment schema (OpenAPI 3.1 fragment + extension fields) — not yet designed; needs a concrete JSON Schema before the first fragment is written. Extension fields accepted so far:
  - `x-upstream` — the single upstream base URL this fragment's routes forward to (e.g. `https://argocd-ro-ardenone-manager-ts.ardenone.com:8444` — **`https://`, corrected 2026-07-20**; this example previously read `http://` for an endpoint documented everywhere else as HTTPS, and since SEAM injects a live credential into every forwarded request, an implementer copying it would have shipped that credential in the clear). Required for an ordinary single-instance route; the request path is appended to it. **Its host must resolve inside SEAM's operator-owned upstream-host allowlist** — the peer control to the `x-vault-path` allowlist and this field's single most important constraint, since without it a correctly-filed, correctly-owned fragment can name its own permitted secret and an attacker-controlled `url`. **Architecture owns that rule in full** (decided 2026-07-20): what the allowlist contains, that it lives in SEAM's own Deployment manifest and never in a fragment, how it is enforced twice, and that there is no per-fragment opt-out. The identical rule binds every `x-upstream-map` entry `url`. **Mutually exclusive with `x-upstream-map`** — a fragment declares exactly one of the two. `x-upstream-map` supersedes `x-upstream` when (and only when) the path declares an `{instance}` parameter; a fragment carrying both, or carrying `x-upstream-map` with no `{instance}` parameter, is a lint error and is quarantined at merge time. OpenAPI's native `servers:` is **not** used to express the upstream target (decided 2026-07-20): `servers:` in the merged spec is reserved for SEAM's own base URL — that is the address agents must actually call — and its root/path/operation inheritance rules make a merged multi-fragment document ambiguous about which server a given operation resolves against. Keeping upstream targets in a SEAM-owned extension also keeps them out of the served spec: `/openapi.json` never discloses upstream hostnames, only SEAM's front door.
  - `x-upstream-tls` — per-fragment upstream transport policy (decided 2026-07-20; see the upstream-TLS rule in Architecture). `{caBundle, serverName, insecureSkipVerify}`, all optional; absent means **verify against the system trust store with hostname checking**, which is the default and needs no declaration. `caBundle` names a PEM key in SEAM's manifest-owned `seam-upstream-ca` ConfigMap — the correct answer for a self-signed fleet proxy, and the thing that replaces every `curl -k` in today's CLAUDE.md prose. `serverName` overrides SNI/hostname verification. `insecureSkipVerify` accepts the single value `acknowledged` and nothing else (no `false` form), is lint-flagged, and is enumerated at `/config/status`. A per-instance `tls` key inside `x-upstream-map` overrides it, since instances of one API can present certificates from different issuers.
  - `x-upstream-plaintext: acknowledged` — required on any fragment whose `x-upstream` is `http://`. Only accepted value, lint-flagged, enumerated at `/config/status`. Justified only for a hop that never leaves an authenticated network path — tailnet-internal or in-cluster Service DNS (see Architecture). **On a fragment using `x-upstream-map` this field does not apply**: acknowledgement there is the per-entry `plaintext: acknowledged` key, because the exposure is per-instance and a fragment-wide marker would misstate which instances are actually in the clear (decided 2026-07-20; see `x-upstream-map` below). Declaring this field alongside `x-upstream-map` is a lint error.
  - `x-seam-owner` — the owning service `<svc>`, required in every fragment from day one (decided 2026-07-20). It is what makes the `x-vault-path` co-ownership rule enforceable *by the gateway at runtime* and not by lint alone: see Architecture, where the authoritative binding is the projected filename `<svc>.yaml` and this field is the redundant in-file declaration checked against it. Deliberately **not** trusted on its own — a hostile author would simply lie in it — so a mismatch against the projected basename quarantines the whole fragment rather than being reconciled in either direction.
  - `x-vault-path` — which OpenBao secret to fetch; must resolve inside SEAM's OpenBao allowlist prefix and be co-owned with the upstream — i.e. inside `seam/routes/<x-seam-owner>/*`, verified against the projected filename (see Architecture)
  - `x-inject-as` — how to inject it. **An object, not a bare enum** (decided 2026-07-20 — this previously read only "header / bearer / query param", naming the *kind* and never the **name** of the header or query parameter, which left both Phase 2's injection and the strip rule that refers to "the header named by each matched route's `x-inject-as`" unimplementable against a field that carried no name). Shape: `{kind, name}`.
    - `kind: header` — `name` is required and is the header to set (e.g. `X-API-Key`). Comparison and deletion are on the canonicalized, case-insensitive name.
    - `kind: bearer` — `name` is **omitted and rejected if present**; it implies `Authorization: Bearer <secret>` exactly. It exists as its own kind rather than as `{kind: header, name: Authorization}` because the `Bearer ` prefix is part of the injection, not part of the secret, and making it a spelled-out convention is how it stays out of the stored value.
    - `kind: query` — `name` is required and is the query parameter to set (e.g. `api_key`). Query keys are case-**sensitive**, unlike headers. Per the strip rule (Architecture), every occurrence of `name` is deleted from the inbound query string before injection.

    The same shape is used by a per-instance `injectAs` override inside `x-upstream-map`, so a multi-instance fragment can hold instances that inject in different kinds *and* under different names.
  - `x-required-scope` — scope(s) a caller must hold (per the per-agent scoping design below)
  - `x-credential-probe` — a designated safe request (path + method + **per-fragment probe interval**; decided 2026-07-16 — some credentials rotate every ~3 days, others essentially never, so cadence is per-fragment, not a global setting) the credential sentinel uses to validate this fragment's secret against its upstream — on a multi-instance fragment the interval is a default that each `x-upstream-map` entry may override per instance, since instances of one API can hold credentials with very different rotation cadences
  - `x-seam-schema` — an explicit fragment-schema version marker (e.g. `v1`) carried by every fragment from day one, so the fragment schema itself can evolve detectably — `seam lint` and the gateway can distinguish a v1 fragment from a v2 instead of guessing from shape
  - `x-breaker` — optional fragment-root tuning of the per-upstream circuit breaker (decided 2026-07-20; full policy and defaults in Passive Route Health). `{threshold, openSeconds, maxOpenSeconds, enabled}`, all optional; absent means the SEAM default policy — **the failure definition, thresholds, open durations, backoff and half-open rule are specified once in Passive Route Health, which owns them, and are deliberately not restated here.** It sits at fragment root because a fragment names exactly one upstream target by construction, and a per-instance `breaker` key inside `x-upstream-map` overrides it per instance. Unlike `x-loop-guard` and `x-cost-per-call`/`x-quota`, absence is **not** "no guard" — the breaker is infrastructure-level and on by default; `enabled: false` is the lint-flagged, `/config/status`-enumerated opt-out.
  - `x-loop-guard` — per-route repeated-identical-request thresholds (max repeats, window)
  - `x-cost-per-call` / `x-quota` — per-route cost annotation and per-caller budget; guards are configured individually per endpoint, never SEAM-wide. **Both are objects carrying an explicit unit, and `x-quota` additionally carries an explicit window** (decided 2026-07-20 — these two fields cap real money on the twitterapi.io credit balance and the z.ai quota, and neither a unit nor a reset period had ever been defined. `$50` was used illustratively throughout while twitterapi.io meters in credits and z.ai in quota units, with no conversion and no per-fragment unit declaration; and "budget window" appeared with no statement of whether it was hourly, daily, calendar-monthly, or rolling. An implementer had to invent both, and either invention silently moves the spend cap by orders of magnitude — which is the one thing this guard exists to prevent).
    - **`x-cost-per-call: {amount, unit}`** — `amount` a positive number, `unit` a short lowercase string naming what is being spent: `usd`, `credits`, `tokens`, `requests`. **SEAM never interprets a unit and never converts between units** — it holds no price table and no exchange rate, and inventing one would be a silent mispricing on exactly the routes that cost money. The unit is a label that makes the number unambiguous to the human sizing the budget and to the agent reading `X-SEAM-Budget-Remaining`; the governor's entire arithmetic is repeated subtraction within one unit.
    - **`x-quota: {amount, unit, window}`** — the budget the governor enforces. `unit` **must be byte-identical to the same route's `x-cost-per-call.unit`**; a mismatch is a lint error and a merge-time quarantine, since a mixed pair would draw a credit balance down in dollars. `x-quota` without `x-cost-per-call` on the same route is likewise a lint error — a budget with no per-call draw can never be decremented, so it caps nothing while reading as though it does.
    - **`window` is a plain duration matching `^[0-9]+(s|m|h|d)$`, and it is a tumbling window anchored at process start — not a calendar period.** A window opens when SEAM starts, closes `window` later, and the next opens immediately with `amount` restored. **A SEAM restart resets it unconditionally**, discarding whatever was drawn, which follows inescapably from guard state being per-process and in-memory (Per-Route Guards; Version Migration Strategy §1). That is also why **calendar-aligned windows are deliberately not expressible**: a `1mo` form would promise a monthly cap that any Tuesday-afternoon binary deploy quietly converts into two monthly caps inside one month — the gap between "per-month" and "per-restart" being precisely the ambiguity this decision closes. Since the honest ceiling on a window is SEAM's own uptime, `seam lint` **warns** on any `window` exceeding `168h` (7 days) and `/config/status` enumerates those routes, so a budget nominally longer than the deploy cadence is visible rather than merely believed. Sizing `window` at or below that cadence is what makes the cap mean what it says.
    - **What `X-SEAM-Budget-Remaining` reports** — on every response from a route carrying `x-quota`: the remainder, its unit, the window length, and the absolute instant the current window closes. `X-SEAM-Budget-Remaining: amount=37.5; unit=credits; window=24h; resets=2026-07-21T04:12:09Z`. All four are load-bearing — a bare number is unreadable without the unit, and without `resets` an agent cannot tell "wait eleven minutes" from "wait eleven hours", which is the difference between self-throttling and hammering. `resets` is the projected close of the current window and moves forward on restart like everything else here. Before Phase 7 the reported remainder is the **shared route-wide** remainder, not the caller's own (Per-Route Guards).
  - `x-upstream-map` — for multi-instance routes: maps an `{instance}` path parameter to a **per-instance object**, not a bare base URL (decided 2026-07-20). **This entry object is specified here and nowhere else** (consolidated 2026-07-20 — it had been restated in four places with four different key sets, and the phase that *builds* map resolution carried one of the short ones; every other mention is now a reference to this one). Each entry is **`{url, vaultPath, injectAs, tls, plaintext, probeInterval, breaker}`**; only `url` is required — and each entry's `url` host is bound by the same operator-owned upstream-host allowlist as `x-upstream` (see Architecture), so a wide map cannot smuggle one attacker-controlled instance in among nine legitimate clusters. The fragment-level `x-vault-path`, `x-inject-as`, `x-upstream-tls`, `x-credential-probe` interval and `x-breaker` act as defaults for any key an entry omits — `tls` overriding `x-upstream-tls` per instance, since instances of one API can present certificates from different issuers.

    **`plaintext` is the one key that is per-entry and never inherited** (decided 2026-07-20). Plaintext exposure is triggered by a *per-entry* `url` being `http://`, so a fragment-wide acknowledgement misrepresents it in both directions: a map with one plaintext instance among eight HTTPS ones would take a fragment-wide marker that reads at `/config/status` as though all nine ship credentials in the clear, and a reader auditing the enumeration could not tell which instance to fix. So on a map-bearing fragment the acknowledgement is `plaintext: acknowledged` **on each plaintext entry**, on the same only-accepted-value terms as everywhere else; a fragment-level `x-upstream-plaintext` alongside `x-upstream-map` is a lint error rather than a blanket cover, and an `http://` entry without its own `plaintext` is a hard error. `/config/status` therefore enumerates **(fragment, instance)** pairs here, matching the reporting key every other per-secret surface already uses. This exists because the motivating case demands it: the per-cluster kubectl-proxies expose a byte-identical API but each needs a *different* credential, differing in kind — long-lived ServiceAccount tokens for the read-only observers, ~3-day OIDC `cloudspace-admin` tokens for the read/write ones — so a single scalar `x-vault-path` cannot serve the collapsed fragment, and a 3-day token needs a far tighter probe cadence than one that never expires. The reserved instance `_all` fans out. Consequently every per-secret surface keys on **(fragment, instance)** rather than fragment alone — see `/health/credentials` below; for a single-instance fragment the instance key is the reserved value `_default`.
  - `x-api-version` — the route contract's version marker, **`v<positive integer>` matching `^v[1-9][0-9]*$` and ordered numerically** (decided 2026-07-20; full grammar, rationale and enforcement in Version Migration Strategy §2). No dotted, dated, or `v1beta1` forms; `seam lint` rejects a non-conforming value and the gateway quarantines the fragment at merge time. A fragment carrying no `x-api-version` is keyed under the reserved `_unversioned`, which **sorts strictly oldest** — so it remains the default for callers sending no `X-SEAM-API-Version` until it drains and is retired; `_unversioned` is assigned by SEAM and is a lint error if written literally.

    **The prohibition is on *authoring* `_unversioned`, not on *addressing* it** (decided 2026-07-20 — the two were conflated, and the conflation was not academic: `_unversioned` is the key under which every fragment shipped in Phases 3-5 lands and the only version that exists at all from Phase 3 through Phase 8, while Architecture requires every SEAM-generated pointer to carry a `?version=` equal to the version that actually served the request. Read as a blanket ban, the reserved key would make that pointer requirement unmeetable for the overwhelming majority of routes — and for *every* route during the phases when the pointer is SEAM's stated differentiator). `_unversioned` names a real served contract, so it is addressable exactly like a `v<n>`:
    - **Rejected** as a value an author writes into a fragment's `x-api-version` — `seam lint` fails it and the gateway quarantines at merge, so the reserved slot cannot be occupied deliberately.
    - **Accepted** as a caller-supplied `X-SEAM-API-Version: _unversioned`, selecting the unversioned contract explicitly rather than by omission. The two are equivalent only while `_unversioned` is still the oldest served version; sending it explicitly is how a caller pins to that contract and keeps landing on it, and it is the honest thing for a worker that fetched the spec before any `v1` existed.
    - **Accepted** as `?version=_unversioned` on `/docs/route` and `/openapi.json?version=`, and **emitted** by SEAM in both — every pointer SEAM generates for an unversioned route (the structured 400's, the loop-breaker 429's, the `docsUrl` in `/changes`, the `Link rel="deprecation"`) carries `&version=_unversioned`, and every response served by such a route echoes `X-SEAM-API-Version: _unversioned`. There is no bare-pointer exception for unversioned routes.
    - **Listed** in the `versions[]` array of a `/docs/route` payload like any other coexisting version — its own entry, its own `docsUrl` carrying `&version=_unversioned`, sorted first because it sorts oldest, and carrying `isDefaultForUnversionedCallers: true` for as long as it remains the oldest served version. A `versions[]` that omitted it would misreport the contract most callers are actually on. Distinct from `x-seam-schema`: `x-seam-schema` versions the *fragment format SEAM parses*, `x-api-version` versions the *contract SEAM serves to callers*. A breaking route change ships as a new fragment bearing a new `x-api-version` (or a new path segment), never as an in-place edit — see Version Migration Strategy and `docs/notes/versioning.md`.
  - `x-seam-deprecated` — a **root-level object on the fragment**, `{since, sunset?}`, marking every route in that fragment deprecated and feeding it into the per-route-version request metric that gates retirement. **It is an object rather than a bare boolean because both headers it drives carry dates** (decided 2026-07-20 — it was previously specified as a boolean, which made the deliverable unimplementable: `Deprecation` (RFC 9745) and `Sunset` (RFC 8594) are both date-valued, and a boolean supplies neither, leaving an implementer to invent a value or silently omit the header).
    - **`since` is required** — the date the deprecation was declared, an ISO date the author writes in the same PR that adds the marker. It populates `Deprecation` directly. This is a fact the author knows, so it needs no computation and can never be wrong.
    - **`sunset` is optional and advisory.** When present it is emitted as `Sunset` verbatim — the author's honest best guess at a removal date, useful for signaling intent on a route somebody has actually planned around. When absent, **`Sunset` is emitted only once the quiet window has actually started and its projected end is computable** — that is, once the route-version's request counter has read zero long enough for the adopted window to resolve to a date (the formula is fixed in Phase 8, which owns it) — and is omitted entirely before then. `Deprecation` alone is emitted in the interval between marking and the quiet window opening, which is correct and RFC-conformant: the route *is* deprecated, and no sunset is yet foreseeable.
    - **Either way `Sunset` is advisory and may move later, never earlier.** Retirement stays usage-gated: a caller appearing during the window resets it, a brownout that finds a caller **cancels** retirement outright, and final removal waits on a human merging the declarative-config PR. So an emitted `Sunset` is a projection, and SEAM will happily blow through it rather than retire a route somebody is calling. A bare `x-seam-deprecated: true` is a lint error with a fix hint naming the object form, rather than being silently coerced to a dateless deprecation.
    - Responses from a deprecated fragment additionally carry `Link` with `rel="deprecation"` pointing at `/docs/route?path=<path>&version=<v>` (the query-parameter shape corrected 2026-07-20 in Architecture) and at `/changes`, so an agent that reads the header lands on the migration path in one hop instead of being told only that something is ending.

    Fragment-scoping is the authoritative representation (decided 2026-07-20): OpenAPI's standard `deprecated` is an **operation-level** boolean with no document-level equivalent, so "a fragment marked `deprecated: true`" has no meaning in the spec — since SEAM's unit of versioning is the whole fragment (a breaking change ships as a new fragment), the marker must be fragment-scoped, hence a SEAM extension. Standard per-operation `deprecated: true` remains valid and is honored for deprecating an individual operation within an otherwise-live fragment; a present `x-seam-deprecated` is equivalent to setting it on every operation in the fragment, and supplies the dates that the bare OpenAPI boolean cannot.
  - `x-adapter` — declarative request/response transforms carried by an **older** coexisting fragment, converting its contract into that of the newest version so SEAM does not need N parallel implementations (decided 2026-07-20; Phase 8 deliverable). Shape: `{targetVersion, request: [...], response: [...]}`, where `targetVersion` names the live `x-api-version` this fragment adapts into and each list is an ordered sequence of transforms drawn from a **closed** vocabulary — rename/move a field by JSON Pointer, default a missing field, drop a field, wrap/unwrap an envelope, rename a query or header parameter. No expressions, no scripting, no code hooks: adapters are data, so a contract migration is a ConfigMap edit and never a gateway binary deploy (see Version Migration Strategy §2). A breaking change that cannot be expressed in this vocabulary gets no adapter — the old fragment simply stays standalone until usage-gated retirement drains it. **An adapting fragment delegates to `targetVersion`'s route rather than forwarding to an upstream of its own** (decided 2026-07-20; full rule in Version Migration Strategy §2): `x-adapter` is mutually exclusive with `x-upstream` / `x-upstream-map`, and `x-vault-path`, `x-inject-as`, `x-upstream-tls`, `x-upstream-plaintext`, `x-credential-probe` and `x-breaker` are likewise taken from the target fragment and are lint errors if declared here — so an adapter can never redirect a credential. Caller-facing fields (schemas, `x-seam-deprecated`, the echoed `X-SEAM-API-Version`) stay this fragment's own; `x-required-scope` is enforced as the **union** of both fragments' requirements; and guards (`x-quota`, `x-cost-per-call`, `x-loop-guard`) are charged to the target's route so coexisting versions share one budget and one loop counter. The transform's output is re-validated against the target's request schema before injection, and a failure is a 500 attributed to the adapter.
  - **Fan-out response envelope** — not a fragment field but the response body an `_all` request returns, specified here because it is a data model two other sections now depend on (decided 2026-07-20; see Multi-Instance Routes, which fixes the status code, and Per-Route Guards, which fixes the billing). The body is a single JSON object with two members and nothing else at the top level:
    - **`instances`** — an object keyed by instance name, one entry per mapped instance including the ones nothing was sent to. Each entry carries `status` ∈ `ok` | `error` | `timeout` | `breaker-refused`, and: for `ok`, `httpStatus` and `body` (that instance's own response body, adapter-transformed and merged in as-is); for `error`, `httpStatus` and the upstream's scrubbed body or SEAM's structured transport error; for `timeout`, the elapsed budget; for `breaker-refused`, the breaker's `openedAt`, `lastError` and `retryAfter` — the same fields the structured 503 carries, so a refused instance is diagnosable without a second call. `breaker-refused` is deliberately its own status rather than an error with a synthetic 503: it means SEAM declined to dispatch, no credential was read, and nothing was charged, which is a materially different fact from an upstream that answered badly.
    - **`summary`** — `{total, dispatched, ok, error, timeout, breakerRefused}`, integers, always present even on a clean fan-out. `total` is the map's width and `dispatched` is what survived breaker filtering, so `total - dispatched` is exactly what the cost governor did not charge for.

    Responses additionally carry **`X-SEAM-Fanout-Partial: 1`** whenever the status is 207, so a client streaming or proxying the body can branch before parsing it. Scrubbing runs once over this whole merged artifact against the union of every value injected on the request (Architecture, pipeline) — which is why the envelope is a single buffered object rather than a stream of per-instance bodies.
  - `x-unscrubbable: acknowledged` — a fragment- or route-level opt-in, and the **only** accepted value (decided 2026-07-20). Secret-echo scrubbing (Credential Health Sentinel) cannot scan a protocol-upgraded, opaque-binary, or undecodable-encoding response, so on such a route SEAM refuses injection by default; this marker is the explicit, auditable acknowledgement that a secret is being injected into a route whose responses cannot be scrubbed. `seam lint` flags every occurrence for human review rather than passing it silently, and `/config/status` enumerates every route currently running unscrubbed. Absence of the field is refusal, not permission — there is no `false` form, because the field exists to make the exposure enumerable, and an enumerable exposure cannot be expressed by a field nobody has to write.

## Future: Per-Agent Tool Scoping

Near-future requirement (see `docs/notes/objective.md`): configure specifically which routes a given agent/NEEDLE-worker may call, rather than every caller getting every route. Researched in detail in `docs/research/` (the **four scoping-related threads** of the six indexed there — secret-injection prior art, agent tool-access-control state of the art, Tailscale-native identity mechanisms, NEEDLE's current identity model; the other two, version-migration precedent and language/runtime choice, inform Version Migration Strategy and ADR-001 respectively). Combined recommendation, from `docs/research/README.md`:

1. **Identity:** each NEEDLE worker/agent gets its own `tsnet`-embedded ephemeral Tailscale node, tagged (e.g. `tag:needle-worker`), so it's a first-class tailnet peer calling SEAM directly. Required, not optional — today's shared Tailscale Connector/egress proxy pods collapse every worker in a cluster into one indistinguishable caller from SEAM's `WhoIs` point of view. No SPIFFE/SPIRE or custom PKI — Tailscale already provides workload identity for this topology (precedent: Tailscale's own TailSQL/Setec/Golink).
2. **Authorization data:** scope claims (`service:action`, e.g. `k8s-ro:get`, `argocd:sync`) travel in the Tailscale Grant's opaque `app` capability field, read via `WhoIs` on the inbound connection. No separate JWT-issuing auth server needed for tailnet-native callers; fall back to a bearer-claim-in-header model only for callers that can't be first-class tailnet peers. The fallback bucket is real, not hypothetical (decided 2026-07-16): **"foreign" NEEDLE workers running outside the tailnet** will still need these tools, likely ingressing via something like a Cloudflare Tunnel — so the bearer-claim path is a first-class (if secondary) auth mode, and the design must cover how that non-tailnet ingress itself authenticates (e.g. Cloudflare Access service tokens) before any scope claim is trusted. That design is **point 7 below** (written 2026-07-20); until it ships, non-tailnet callers are default-denied, not provisionally trusted.
3. **Enforcement:** every SEAM route is tagged with its required scope(s) (`x-required-scope` in the route fragment); the gateway checks the caller's Grant-supplied scopes against them, default-deny. Never rely on the calling agent's own tool list as the boundary — client-side restriction is UX, not security (OWASP Agentic Top 10 consensus, and a live AutoGen cross-agent-tool-leak issue found in research).
4. **Gap to close first:** NEEDLE currently mints no identity/credential for spawned agents at all (`docs/research/needle-identity-model.md`). This needs new NEEDLE-side work in `run_process` (`~/NEEDLE/src/dispatch/mod.rs:703`) — provision a per-worker `tsnet` identity, inject it via the existing `adapter.environment` passthrough (or a new template variable) — before SEAM-side scoping can be enforced against anything real.
5. **Self-service surface:** authorization must be discoverable, not opaque. `/whoami` returns the caller's resolved tailnet identity, tags, and effective scopes; `/openapi.json` and `/docs` filter to the routes the caller may actually invoke, so agents never spend context on unreachable routes; and a 403 names the exact missing scope plus the Grant snippet that would fix it — authorization debugging becomes "ask the gateway," not "read the tailnet policy file." **The scope-naming 403 applies only where the route is already visible in the caller's own filtered spec** (cross-reference, 2026-07-20): a route the filter removed returns **404, identically to a route that does not exist**, on `/docs/{route}` and on level-2 `/changes` alike — see the scope-filtering rule under `/docs/{route}` in Architecture, which owns this decision. So the 403 covers the visible-but-not-invocable case (right path, wrong method or guard), which is where naming the scope helps the caller and leaks nothing; it never covers a route the caller was not meant to know exists, because that would turn this self-service surface into the enumeration oracle filtering exists to prevent.
6. **Scope taxonomy:** grows route-by-route as fragments declare `x-required-scope` — no up-front `service:action` catalog (decided 2026-07-16). The live taxonomy must stay dynamic and discoverable: enumerable from the merged spec (e.g. a `/scopes` view) rather than maintained as a separate document. A future SEAM shows different taxonomy slices to different agents, RBAC-style — an agent sees only the portion of the taxonomy its grants intersect, consistent with the scope-filtered spec in point 5. Since discoverability *replaces* a maintained catalog here, `/scopes` is a concrete Phase 7 endpoint, not a figure of speech (decided 2026-07-20): it is **scope-filtered by default, exactly like `/openapi.json`** — returning the scopes the caller's grants actually intersect, each with a one-line meaning and the (likewise filtered) routes that require it, so "what may I do here" is one request that costs no context on scopes the caller can never hold. The **unfiltered** taxonomy is available at `/scopes?all=1`, but is itself gated on a scope (`seam:scopes:read-all`): a complete inventory of every capability across every service SEAM fronts is precisely the reconnaissance that filtering `/openapi.json` exists to withhold, so it is an operator view, never an ambient one. Both forms are derived at request time from **two sources, merged**: the merged spec's `x-required-scope` declarations, and a small **built-in set of control-plane scope declarations compiled into the SEAM binary** (decided 2026-07-20). The built-in set exists because SEAM's own control plane resolves *before* the merged route table (Fragment Toolchain rule 2) and therefore has no `x-required-scope` in any fragment — so under a fragments-only derivation the two scopes an operator most needs to discover, **`seam:ops:read`** (gating `/config/status`, `/health/credentials`, `/health/upstreams`) and **`seam:scopes:read-all`** (gating `/scopes?all=1`), would be exactly the two `/scopes` could not show, while being hardcoded in the binary regardless. Each built-in declaration carries the same fields a fragment-derived one does — the scope, its one-line meaning, and the control-plane paths requiring it — and is labeled `source: builtin` so an operator can tell a capability SEAM owns from one a service declared. It is scope-filtered on the same terms: a caller holding neither sees neither in `/scopes`, and both appear in `/scopes?all=1`. The claim this replaces — "there is no stored catalog, and therefore nothing that can drift" — is corrected to the accurate one: **there is no *separately maintained* catalog. Both sets are computed at request time**, one from the live merged spec and one from the binary's own routing table, so neither can drift from what SEAM actually enforces; what would drift is a hand-written scope document, and there still is none. Adding a control-plane scope is therefore a binary change that updates enforcement and discovery in the same commit, by construction.
7. **Non-tailnet ingress authentication** (decided 2026-07-20) — the design point 2 defers to, written down here because a declared first-class auth mode with no design gets built ad hoc as "trust the `X-Scopes` header," which on a secret-injecting proxy is a trivially forgeable authorization bypass that hands an attacker every credential SEAM holds. **Four rules, none of them per-deployment negotiable — plus one phasing constraint, which is scheduling rather than a rule and is set apart below for that reason** (clarified 2026-07-20; this read "four rules" over five bullets, and on a list whose whole claim is non-negotiability an ambiguous count invites reading the odd one out as advisory). The four rules:
   - **The ingress identity is verified at the gateway, not at the edge.** Foreign workers reach SEAM through a Cloudflare Tunnel fronted by Cloudflare Access; Access issues a service-token JWT, and **SEAM itself validates that JWT on every request** — signature against Cloudflare's published JWKS for the team domain, `aud` matching SEAM's own Access application ID, plus `iss`/`exp`/`nbf`. SEAM never treats "this request arrived on the tunnel listener" as proof of anything: a tunnel is transport, and edge-only verification means anyone who reaches the origin by any other path is authenticated by default.
   - **Scopes are bound to the verified ingress identity and never read from a client header.** The caller's scope set is looked up from the verified token subject (`sub` / `common_name`) against a SEAM-side service-token→scopes mapping — structurally the same move as looking up a tailnet caller's scopes from the `WhoIs` Grant. There is no request header, query parameter, or body field through which any caller, in this mode or any other, can assert its own scopes.
   - **Client-supplied scope headers are stripped before evaluation, not merely ignored.** `X-SEAM-Scopes` and every equivalent are deleted on ingress by the same strip-then-inject rule that removes `Authorization` and `x-inject-as` headers (Architecture), so a forged header cannot be read even by a future code path that mistakenly looks for one, and cannot survive into an upstream request.
   - **Default-deny on the mode itself.** The foreign-worker path is off unless explicitly enabled, and a request presenting no valid Access JWT is rejected *before* route matching, secret lookup, or any upstream contact. The failure mode is a 403 — never "fall through to the tailnet path with no identity resolved."

   And the phasing constraint, which is not one of the four rules — it says *when* this mode ships, not *how* it must behave, and unlike the rules above it is discharged once, by the schedule, rather than enforced on every request:

   - **Phasing: this is Phase 14** (assigned 2026-07-20; dependency edge `7 → 14`). Phase 7 ships the tailnet-native Grant path only and default-denies every non-tailnet caller. The foreign-worker mode is gated on Phase 7's enforcement being real and exercised, because a second identity source is only safe once the enforcement point it feeds already works.

## Version Migration Strategy

NEEDLE workers run perpetually-live sessions — a session may learn the API contract once (fetch `/openapi.json` early) and keep calling routes on that understanding for hours, with no guarantee it ever restarts or refetches the spec. Full design in `docs/notes/versioning.md`; precedent research in `docs/research/version-migration-precedent.md`. **Three** distinct problems (corrected 2026-07-20 — this read "two" from before passive drift signaling was adopted as item 3). The numbering here maps one-to-one onto the section labels in `docs/notes/versioning.md`, so a `§`-reference means the same thing in either document: **§1 = §A** (gateway process deploys), **§2 = §B** (route contract versioning), **§3 = §C** (passive drift signaling). Cite them as "Version Migration Strategy §2" / "versioning.md §B" interchangeably.

1. **Gateway process deploys** (SEAM's own code changes, contract unchanged) — standard Kubernetes rolling-update mechanics: readiness-gated rollout and graceful `SIGTERM` draining. **SEAM runs as a single replica** (amended 2026-07-20 — this bullet previously mandated ≥2 replicas and `PodDisruptionBudget minAvailable: 1`). Every stateful feature in this plan is per-process and in-memory by design — loop-breaker counters and cost-governor budgets (Per-Route Guards), last-2xx tracking and circuit-breaker state (Passive Route Health), the credential sentinel's probe loop — and two replicas behind one Service silently double all of it: a loop-guard threshold of 10 permits 20 identical requests, a `$50` `x-quota` permits `$100` against a metered upstream, `X-SEAM-Budget-Remaining` depends on which pod answered, `/docs` shows a different last-success on each refresh, and the breaker opens on one replica while the other keeps hammering a dead upstream. On the twitterapi.io credit and z.ai quota routes that is real money, so correctness of the guards outranks a second replica of a stateless proxy. The Deployment is therefore `replicas: 1` with `strategy.rollingUpdate: {maxSurge: 0, maxUnavailable: 1}` — a rollout never runs two SEAM pods at once — at the cost of a **real outage window during a binary deploy, on the order of seconds to tens of seconds** (corrected 2026-07-20; this previously claimed a "sub-second gap", which was wrong by one to two orders of magnitude and was the *sole* justification offered for accepting the single-replica tradeoff, so it would have been used to size agent retry behavior at cutover). With `maxSurge: 0` the old pod is fully terminated **before** the replacement is scheduled, so the gap is the sum of `terminationGracePeriodSeconds` plus in-flight drain, pod scheduling, image pull (usually cached), process start, initial fragment merge and OpenBao login, and the first passing readiness probe — **typically 5-30 seconds**, not a blip. The tradeoff itself is unchanged and still holds, because binary deploys are rare and route and contract changes are ConfigMap edits that never restart the pod at all; what changes is what callers must be built to absorb. **The corresponding requirement on callers: any agent calling SEAM must implement retry-with-backoff spanning at least 60 seconds**, and an agent whose retry budget is one second will fail across every SEAM deploy. That is a checklist item in the 6b cutover, not an aspiration. No `PodDisruptionBudget` demanding an available replica is set, since with one replica it would only deadlock node drains. The single-replica invariant is load-bearing: **no feature may be built assuming it can be replicated**, and each stateful section below states its own multi-replica semantics explicitly. If SEAM ever genuinely needs HA, the documented upgrade path is a shared state store for guard counters, quota ledgers, and breaker state (Redis or equivalent) plus leader election for the sentinel — a scoped, later piece of work, not something to be back-doored by raising `replicas`. The credential sentinel's probe loop is leader-elected regardless (see Credential Health Sentinel), so the invariant survives both that future scale-out and any accidental transient overlap.
2. **Route contract versioning** (a route's params/response/injection shape changes in a breaking way) — borrows Kubernetes' API deprecation policy (old and new versions coexist as long as anything calls the old one — this falls out of the existing ConfigMap-per-fragment model for free, a breaking change just ships as a new fragment rather than an in-place edit) and Stripe's adapter model (implement the route once against the newest contract, add a thin request/response adapter per older version still being served, rather than N parallel full implementations). Deprecated fragments emit `Deprecation`/`Sunset` headers (advisory only) and are retired when a per-route-version request-count metric has read zero for a quiet window. **`Deprecation` is emitted from the moment the marker lands; `Sunset` only once a removal date is actually projectable** (decided 2026-07-20) — retirement here is deliberately dateless, so there is no date to emit until the quiet window has opened and its projected end resolves (the window's formula is fixed in Phase 8, which owns it), unless the fragment's author supplied an advisory one. See the `x-seam-deprecated` entry in Data Models, which carries `{since, sunset?}` for exactly this reason. **Zero observed traffic is a necessary condition for retirement; the length of the quiet window is traffic-adaptive with a calendar floor** (clarified 2026-07-20 — this read "usage-gated, never calendar-gated", which was heard as forbidding the calendar floor the recommended composition in `docs/notes/quiet-window-options.md` depends on). The distinction that matters is the one originally intended: no route is ever retired because a date arrived, only because it stopped being called — a calendar term may only ever *lengthen* the wait, never trigger a removal on its own. See Phase 8 for the adopted composition.

   **Where the gap history lives: outside the gateway process, in the metrics store** (decided 2026-07-20). The adopted window (Phase 8, which owns the formula) is traffic-adaptive with a calendar floor, sized off the **observed max inter-request gap on that route-version** — and `docs/notes/quiet-window-options.md` Option B flags the catch — that history "needs to be persistent to be meaningful across gateway restarts." Every other stateful feature in this plan is deliberately per-process, in-memory and restart-scoped (§1, Per-Route Guards, Passive Route Health), and if the gap history were too, each binary deploy would reset both the observed maximum and the quiet clock: retirement would either never converge, or converge on the 7-day floor forever — i.e. silently degrade Option B back into Option A, the option it was chosen over. The resolution is **not** to weaken the in-memory invariant but to observe that this is the one consumer that does not need in-process state at all. §2 already exports a **per-route-version request-count metric** (mirroring Kubernetes' `apiserver_requested_deprecated_apis`), and that time series is already durable, already retained, and already survives deploys in the fleet's existing metrics store. **The retirement evaluator is therefore a query over that series, not a reader of gateway memory:** the observed max inter-request gap is the longest interval between increments of the counter for that (route, `x-api-version`), and "quiet since" is the timestamp of the last increment. The gateway keeps only a live in-process view of the same counter, which may reset freely because nothing depends on it. Three consequences, stated so nobody re-derives them: the evaluator runs as the small out-of-band job Option D already assumes (it is the thing that opens the declarative-config PR), so this adds **no** request-path state, **no** PVC, and **no** second stateful component to a single-replica deployment; gap resolution is bounded below by the scrape interval, which is irrelevant against a 7-day floor; and the metrics store's retention window is now load-bearing — a route-version older than retention reports "insufficient history" and falls back to the 7-day floor rather than inventing a gap, exactly as a cold route-version does. Retirement remains usage-gated throughout: this decides only *where the evidence is read from*, never that a date may trigger a removal.

   **Adapters are declarative transforms in the old fragment's own YAML, never Go code in the gateway binary** (decided 2026-07-20). Stripe's model assumes a route implementation to adapt *toward*; SEAM is a proxy and has no implementation of its own — the implementation is upstream. So the model translates as: the **newest** fragment is the one whose contract matches what the upstream actually speaks today, and every older coexisting fragment carries an `x-adapter` block (see Data Models) declaring the request-in / response-out transforms that convert its older contract into that newest one. Keeping adapters in the fragment is what preserves SEAM's config-only property: a route-contract change stays a ConfigMap edit synced by ArgoCD, with **no gateway binary deploy** — "nothing about the pod changes" (Architecture) holds for contract migrations too. Adapters as gateway Go code would invert that, making every route-contract change a fleet redeploy and dragging per-service transform logic back into exactly the monolithic file the ConfigMap-per-fragment model exists to abolish. The transform vocabulary is deliberately **small and closed** — rename/move a field by JSON Pointer, supply a default for a field the old contract omits, drop a field, wrap/unwrap an envelope, rename a query or header parameter — and there is **no escape hatch into arbitrary code or expressions**. A change that cannot be expressed in that vocabulary is simply not adapted: the old fragment stays mounted as a standalone fragment against its own upstream target and lives out its usage-gated retirement unmodified. Accordingly, **Phase 8 must deliver** the `x-adapter` schema and its transform vocabulary, the request/response transform executor in the proxy path, `seam lint` validation of adapter blocks (including that `x-adapter.targetVersion` names a version that is actually live and coexisting), and the version-selection mechanism below.

   **Adapter resolution: an adapting fragment delegates to its target version's route and owns none of the upstream-facing fields** (decided 2026-07-20). The paragraph above says the newest fragment "is the one whose contract matches what the upstream actually speaks today" and that an *unadaptable* old fragment "stays mounted as a standalone fragment against its own upstream target" — which implies, without ever stating, that an *adapted* fragment does **not** use its own upstream. Left implied, an implementer must invent the whole resolution, and the invented answers have direct security consequences (which secret gets injected) and money consequences (which budget gets charged). The rule, in full:

   - **A fragment carrying `x-adapter` must not declare `x-upstream` or `x-upstream-map`** — the two are mutually exclusive, a lint error and a merge-time quarantine, exactly like `x-upstream` vs `x-upstream-map`. Declaring an adapter *is* declaring "my upstream is whatever `targetVersion` forwards to." The Data Models rule that `x-upstream` is required for an ordinary single-instance route stands unchanged for every fragment without an adapter, including an unadaptable deprecated one, which is precisely why that fragment keeps its own target.
   - **Every upstream-facing field comes from the target version's fragment, never the adapting one:** `x-upstream` / `x-upstream-map`, `x-vault-path`, `x-inject-as`, `x-upstream-tls`, `x-upstream-plaintext`, `x-credential-probe`, and `x-breaker` (the breaker keys on the resolved upstream origin, which is the target's — an adapter that could declare its own would fragment one upstream's breaker state across coexisting versions and defeat the guard). An adapting fragment declaring any of them is a lint error rather than a silent override. This is the security-critical half: if an old fragment could name its own `x-vault-path` while delegating to a live route, "ship an adapter" would become a way to point any secret in the allowlist at any upstream, defeating the co-ownership rule in Architecture. One route-version-family, one credential.
   - **Every caller-facing field comes from the matched (older) fragment:** its own request/response schemas — validated at pipeline stage 6 against "the matched version's own declared schema", which is exactly the contract that caller was served — plus its own `x-seam-deprecated`, and the `X-SEAM-API-Version` echoed back to the caller, which is the **old** version, not the target's.
   - **`x-required-scope` is enforced as the union of the two, at pipeline stage 5.** This union is why authorization is a stage of its own *after* route-and-version match rather than part of identity resolution at stage 3: the requirement is not computable until both fragments are known (Architecture, pipeline). The caller's own fragment declares what it always did, but a scope requirement added on the newer version applies as well — otherwise pinning an old `X-SEAM-API-Version` would be a privilege-de-escalation path, letting a caller shed a guard by claiming an older contract. Union is the only direction that cannot be gamed.
   - **Guards are charged to the target version's route, not per version.** `x-cost-per-call` / `x-quota` and `x-loop-guard` are keyed on the **target** (path, method, `targetVersion`), so all coexisting versions of one route draw from one budget and one loop counter. Otherwise N coexisting versions would multiply the spend cap N-fold on a metered upstream, and a looping agent could reset its own counter by changing a header — both of which defeat the guards outright. The loop breaker's normalized request hash is taken **after** the request transform, so two callers on different versions sending semantically identical requests are correctly counted as the same loop. Both properties are what force the adapter transform (pipeline stage 8) to precede guards (stage 9) rather than follow them, as the pipeline previously had it: this bullet's charged-to-the-target rule and the hash rule both consume the transform's output (Architecture, pipeline). Where the old and target fragments both declare guard fields, the target's govern.
   - **The transform's output is re-validated against the target version's request schema, and a failure is SEAM's fault, not the caller's.** Re-validation concludes pipeline stage 8, before guards at stage 9 and secret resolution at stage 10 — so a broken transform can never cause a credential to be injected into a malformed upstream request. (That argument is unchanged by the 2026-07-20 reordering that moved the adapter ahead of guards: injection still follows re-validation, one stage further down.) The response is a **500** naming the adapter, the failing JSON pointer, and both versions, never a 400: the caller sent a valid request against the contract it was given, and reporting a 400 would send an agent to re-read a `/docs/{route}` that already agrees with what it sent. `seam lint` checks the same property statically where it can (that the declared transform sequence carries the old schema into the target schema), so most such defects never reach runtime. The **response** transform's output is not re-validated — SEAM does not validate upstream responses on any path — but it remains subject to the buffered-path and scrubbing rules in Architecture.

   **Selecting among coexisting versions** (decided 2026-07-20). When a breaking change ships as a new **path segment**, selection is the URL and needs no mechanism. When it ships as a new **`x-api-version`** under the same (path, method), the caller selects with an explicit `X-SEAM-API-Version: <v>` **request header** — and when that header is absent, the request resolves to **the oldest still-served version, not the newest**. That default is the only one consistent with the rest of this strategy: a perpetually-live worker that learned the contract hours ago sends no version header at all, so defaulting to newest would break precisely the callers coexistence exists to protect, turning every breaking change into a silent fleet-wide outage. Newly-arriving and re-reading callers are steered forward instead of shoved: `/openapi.json` and `/docs` advertise the newest version as canonical and render older coexisting ones as deprecated, `Deprecation`/`Sunset` and `/changes` do the pushing, and the effective default advances on its own the moment usage-gated retirement removes the drained old fragment. Every response echoes the version that actually served it in `X-SEAM-API-Version`, so a caller is never guessing which contract it is talking to. Note that `X-SEAM-API-Version` is an inbound `X-SEAM-*` header and must therefore be added to the documented exception list in the inbound-header-stripping rule (Architecture), alongside `X-SEAM-Dry-Run`.

   **The grammar and total ordering of `x-api-version`** (decided 2026-07-20). "The oldest still-served version" is load-bearing — get it wrong and the result is the silent fleet-wide outage this default exists to prevent — but nothing defined an ordering over a field exemplified only as the free-form string `v1`. Without one, SEAM cannot know that `v2` succeeds `v1`, cannot order `v10` against `v9` (lexically `v10` sorts *first*), and cannot place a date-form value at all. So the value space is closed:

   - **Grammar: `v<positive integer>`, matching `^v[1-9][0-9]*$`.** No leading zeros, no dotted or suffixed forms, no `v1beta1`, no dates. **Ordering is numeric on the integer** — `v9 < v10` — which is total, decidable without a calendar, and stable under any future addition.
   - **Why integers rather than Stripe's dates**, since §2 cites Stripe's model approvingly: SEAM borrows Stripe's *adapter* idea, not its version namespace, and the two design pressures differ. Dates serve a SaaS shipping continuously to callers pinned at first contact — and per-caller version pinning is explicitly **parked** here (`docs/notes/per-caller-version-pinning.md`), so the property dates buy is one SEAM has deliberately not bought. What SEAM has instead is hand-authored fragments in git, where a date is a value an author types: two fragments can collide on one date, a backdated value silently reorders the default, and a reader cannot tell a contract's date from its authoring date. An integer is checkable at a glance, collides visibly, and matches both the `v1` already used throughout this plan and the Kubernetes API-versioning precedent §2 already borrows its coexistence policy from. The narrower alphabet is the point: this field selects which contract a caller is held to, and a field with one legal shape cannot be mis-sorted.
   - **`_unversioned` sorts strictly oldest** — below every `v<n>`, unconditionally. This is the case that matters most and was previously undefined: every fragment shipped in Phases 3-5 predates versioning and therefore carries no `x-api-version`, so it is keyed `_unversioned` (Fragment Toolchain). The **first** breaking change on any such route ships as `v1` while the incumbent keeps its `_unversioned` key — and because `_unversioned` sorts oldest, an unversioned caller sending no header keeps landing on the contract it already knows, and keeps doing so until that fragment drains and is retired. Any other sort position would make the very first use of this mechanism break every existing caller at once. `_unversioned` is a **reserved key SEAM assigns, never a value an author writes**: a fragment declaring `x-api-version: _unversioned` literally is a lint error and is quarantined at merge, so the reserved slot cannot be occupied deliberately. **That prohibition is on authoring it, not on addressing it**: `_unversioned` names a real served contract and is a valid value on `X-SEAM-API-Version`, on `?version=`, in SEAM-emitted pointers and echoes, and in a `versions[]` list — see Data Models, `x-api-version` (decided 2026-07-20).
   - **Enforced twice, as everywhere else in this plan:** `seam lint` rejects any non-conforming `x-api-version` at the CI gate, and the gateway re-checks the pattern at merge time and quarantines the whole fragment, never trusting lint to have run. Ordering is evaluated **per (path, method)** — versions of different routes are never compared, so nothing requires version numbers to be globally synchronized across services.
3. **Passive drift signaling** (helping perpetually-live callers notice contract change mid-session) — every response carries an `X-SEAM-Spec-Version` header (a hash of the currently-served merged spec), so a worker that cached the contract hours ago sees the version change on a call it was already making, at no extra request cost. The catch-up diff is **progressively discoverable** (decided 2026-07-16), giving schema-diff fidelity without the token cost of dumping full schemas: level 1, `/changes?since=<version>`, returns a token-cheap route-level list — path, verb, change kind, one-line summary, and **two drill-down links, not one** (respecified 2026-07-20, matching the `/docs/{route}` bullet in Architecture): **`diffUrl`** → `/changes?since=<version>&route=<path>`, the level-2 field-level schema diff, answering *what changed*; and **`docsUrl`** → `/docs/route?path=<path>&version=<v>`, answering *what the contract now is* (the normalization-safe query form fixed in Architecture 2026-07-20; the older `/docs/{route}?version=<v>` spelling this clause carried is superseded). They are different questions and an agent catching up mid-session wants both — the diff to decide whether it cares, the docs to rewrite its call if it does. Level 2, `/changes?since=<version>&route=<path>`, returns the full field-level schema diff for that one route; prior full specs stay fetchable at `/openapi.json?version=<v>`. An agent notices the header change, pulls level 1, and drills into only the routes it actually uses. Backed by a ring buffer of the last N merged specs; this gives the advisory `Deprecation`/`Sunset` headers the actionable catch-up path they lacked, and usage-gated retirement remains the enforcement backstop. Full design in `docs/notes/versioning.md`.

   **Archived specs and diffs are stored unfiltered and filtered at serve time** (decided 2026-07-20) — the reconciliation between this section and the scope-filtered `/openapi.json` of Phase 7 (Per-Agent Tool Scoping point 5). The ring buffer holds the **unfiltered global** merged spec, one entry per spec version, and explicitly *not* one entry per caller-grant combination: that would multiply the buffer by the number of distinct grant sets in the fleet and is not a cost this design will carry. Instead, **every read path applies the caller's *current* scope filter at serve time.** `/openapi.json?version=<v>` returns the archived spec reduced to the routes that caller may invoke today; `/changes?since=<version>` (level 1) omits routes the caller cannot see; `/changes?since=<version>&route=<path>` (level 2) returns **404 for a route outside the caller's grants — byte-identically to a route that has never existed** (aligned 2026-07-20, superseding this clause's earlier "403 naming the missing scope"; the governing decision is the scope-filtering rule under `/docs/{route}` in Architecture, and level 2 is bound by it for the same reason: a 403/404 split here is an enumeration oracle that lets a caller reconstruct the withheld route inventory of every service SEAM fronts, one archived-diff probe at a time). The scope-naming 403 of point 5 survives on this endpoint only in the visible-but-not-invocable case — a route the caller *can* see in its own filtered spec but lacks the scope for the specific method or guard. As on `/docs/{route}`, the 404 body names `/whoami` and the caller's `X-SEAM-Scope-Version`, and SEAM logs the resolved route server-side so an unreachable-but-believed-real route stays answerable off the request path. An archived spec is **never** served verbatim — serving it raw would make the archive a one-request bypass of the entire filtering guarantee, disclosing the full route inventory of every service SEAM fronts to anyone who asks for yesterday's version.

   `X-SEAM-Spec-Version` hashes the **global** merged spec, not the caller's filtered view. It is a change *signal*, not an authorization surface, and a per-caller hash would fragment the version namespace so thoroughly that a `since=` value could not be reasoned about across two callers, across logs, or across the ring buffer at all. The deliberate consequence is that a caller whose *grants* changed can see routes appear or disappear in a diff for reasons having nothing to do with the spec changing — and that is signaled explicitly rather than left to look like contract drift. Level-1 change kinds therefore separate **contract** changes (`added` / `removed` / `params-changed` / `response-changed` / `deprecated`) from **visibility** changes (`granted` / `revoked`), and responses carry `X-SEAM-Scope-Version` — a hash of the caller's effective scope set — alongside `X-SEAM-Spec-Version`, with `/whoami` reporting the same value. A caller may pass `&scope-since=<scope-version>` to `/changes`; SEAM retains the scope set behind each `X-SEAM-Scope-Version` it has issued (a small hash→scope-set map, which is SEAM's own derived data and cheap, unlike a per-caller spec archive), so it can diff the two filtered views and label each route's delta as contract or visibility. Omitting `scope-since` yields a diff flagged `scopeChanged: unknown`, telling the agent it cannot distinguish the two causes for itself. Net effect: a worker can always tell "the contract moved" from "my access moved" without guessing.

   **Both stores here are in-memory and restart-scoped, and both degrade to an already-defined behavior rather than to an error** (decided 2026-07-20 — stated because §1, Per-Route Guards and Passive Route Health all repeat that state is per-process and restart-scoped, while this section introduced two stores and said nothing, leaving an implementer to guess at persistence for a surface whose entire purpose is serving sessions that outlive a deploy).

   - **The ring buffer of prior merged specs is in-memory, bounded to the last N spec versions, and lost on restart.** It is not persisted, and deliberately so: it is a cache of derived data, and the alternative — a PVC or an external store — buys durability by adding the first piece of request-path state to a deployment whose single-replica invariant exists precisely to avoid that. What makes this safe is the fallback, which `docs/notes/versioning.md` §C already specifies and this plan previously omitted: **a `since=` value that is unknown, evicted, or from a pre-restart buffer is not an error.** `/changes` answers **200** with `sinceKnown: false`, no diff, and a pointer to the current full spec (`/openapi.json`, plus the caller's current `X-SEAM-Spec-Version`), telling the agent plainly that it must re-read rather than catch up incrementally. That is strictly the pre-`/changes` behavior — refetch the spec — so the worst case of a deploy is that live workers lose the *cheap* catch-up path for one cycle, never the *correct* one, and no `since=` value a worker holds can ever produce a wrong answer. Should perpetual history ever be wanted, note the cheap escape hatch: prior merged specs are **deterministically reconstructible from git history in declarative-config**, since the fragments are the inputs and the merge is deterministic — so the archive can be rebuilt offline by an out-of-band job without the gateway ever holding it. That is an available upgrade, not a runtime dependency; nothing in the request path may be built assuming it.
   - **The `X-SEAM-Scope-Version` → scope-set map is in-memory and explicitly bounded** — it had no stated eviction rule, and an unbounded map keyed by a hash of caller-controlled-adjacent input is a slow leak. Retention is: **the most recent 4 distinct scope versions per caller identity, evicted after 24 hours idle for that identity, under a global LRU cap on total entries.** Four covers the realistic case (a caller's grants change rarely, and only the last few are diffable against), 24 hours comfortably exceeds a perpetually-live session's useful catch-up horizon, and the global cap makes the memory ceiling a number rather than a hope. A `&scope-since=` value that is unknown, evicted, or pre-restart yields exactly the behavior already defined for **omitting** it — the diff is flagged `scopeChanged: unknown` — so this failure mode needs no new client handling, and a restart simply means every caller gets `scopeChanged: unknown` until it has been issued a fresh scope version. Current-scope *filtering* is unaffected by any of this: it is computed from the caller's live grants on every request and never read from this map, so a restart can never widen what a caller sees.

   **Everything this section introduces is phased, and the phase entries own the assignment** (decided 2026-07-20 — recorded because §3 grew a whole subsystem, and an unassigned deliverable is an unbuilt one). The split is: **Phase 7** owns the scope-identity half, because it is the phase that first makes a scope set exist; **Phase 8** owns the drift-signaling half, because it is the phase that first makes a spec archive exist; and the **`7 → 8` forcing edge** between them is recorded in the dependency-edge list under Implementation Phases. The per-deliverable enumeration lives in the **Phase 7 and Phase 8 entries** and is deliberately not duplicated here, so a future correction has exactly one place to land.

## Credential Health Sentinel

SEAM holds both the secret (`x-vault-path`) and a way to test it (the upstream), so it can detect dead credentials before an agent does. Recurring pain this targets: cloudspace-admin OIDC tokens expiring every ~3 days, and a dead ESO static token going unnoticed for 16 days.

- A background loop validates each fragment's credential against its upstream via the fragment's `x-credential-probe` (a designated safe authenticated request), on a per-fragment cadence.
- `/health/credentials` reports per **(fragment, instance)** — the reporting key, since one multi-instance fragment can hold N distinct credentials (see Data Models); single-instance fragments report under the reserved instance `_default` — giving last verified, current status, known expiry — never secret values.
- Self-healing on rotation: an upstream `401` invalidates the cached secret, refetches from OpenBao, and retries once (safe for any method — a 401 means the upstream rejected the request before processing it). **The retry needs the request body, so retry eligibility is bounded by an explicit request-body buffer** (decided 2026-07-20 — stated because ADR-001 treats streaming passthrough as a hard requirement and this plan set a buffering cap only for *responses*, leaving an implementer to invent whether a request body can be replayed at all, and to silently not retry when it cannot). SEAM buffers the inbound request body up to **`maxReplayableRequestBytes`, default 1 MiB** — an **independently configurable limit** that merely shares a default with the response-scrubbing cap `maxBufferedResponseBytes`, and does not track it (specified as two settings 2026-07-20; this previously read "the same 1 MiB cap used for response scrubbing" and was cited elsewhere as a fixed constant, leaving it unclear whether one knob moved both. It matters: raising the response cap for a chatty upstream would otherwise silently change 401-retry eligibility *and* loop-hash canonicalization for requests, two behaviors nobody was tuning). It tees the body upstream as it streams, and discards it as soon as the response status is known. Within the cap the retry is a genuine replay: the buffered bytes are re-sent, with headers re-stripped per the inbound-sanitation rule and the freshly-refetched secret re-injected. Above the cap — and for any body SEAM cannot replay at all, such as a request-side protocol upgrade or an unbounded chunked stream with no declared length — the body streams through unbuffered and **the request is not retried**. That case is reported rather than silently degraded: the cache is still invalidated and the secret still refetched, so the caller's *next* request succeeds, and SEAM answers with a structured error envelope naming the condition **`credential-refresh-not-retried`**, carrying the upstream's own status and (scrubbed) body plus the plain statement that resending the request will now use the refreshed credential. The self-healing guarantee is therefore conditional, and — the point of naming it — the condition is one the caller can actually detect.
- **Probe traffic is tagged, and its exclusions are per-consumer rather than blanket** (decided 2026-07-20). The sentinel issues a byte-identical authenticated request to every probed fragment on a fixed cadence, which is indistinguishable — to three other subsystems that consume exactly that signal — from the pathology each of them exists to catch. Every sentinel-originated request therefore carries an **in-process origin tag** set by the probe loop itself, never a wire header: all `X-SEAM-*` request headers are stripped on ingress (Architecture), so the tag cannot be forged and no caller can exempt itself from a guard by claiming to be a probe. Each consumer then decides for itself, and they do not all decide the same way:
  - **Excluded from the per-route-version request-count metric** (Version Migration Strategy §2, Phase 8). A deprecated fragment keeps its `x-credential-probe`, so without this exclusion SEAM's own probes hold the retirement counter above zero forever, the quiet window never opens, and **no probed fragment can ever retire**. For the same reason a probe can never reset a quiet window or cancel a brownout (Phase 8 (C)) — only a real caller does.
  - **Excluded from the loop-breaker counter** (Per-Route Guards). A fixed-interval, byte-identical request is precisely what that guard trips on; unexcluded, the sentinel 429s its own probes, and before Phase 7 — where the key degrades to (route, hash) — it would 429 every real caller of the probed route alongside them.
  - **Excluded from quota accounting and from `X-SEAM-Budget-Remaining`.** A probe on a route carrying `x-cost-per-call` would otherwise draw the route-wide budget down on a timer, on exactly the metered upstreams (twitterapi.io credits, the z.ai quota) SEAM was installed to protect. The exclusion is from SEAM's *governor*, which is a runaway-spend brake and not an accounting ledger (Per-Route Guards) — the upstream may well still meter the call, so the probe's own draw is a fixed, known rate of one call per `probeInterval`, and `/health/credentials` reports that rate per (fragment, instance) so the spend is enumerable rather than invisible. On a metered route, prefer a probe the upstream does not bill at all and size the interval against the meter.
  - **Counted, deliberately, in the last-2xx passive-health signal and in circuit-breaker state** (Passive Route Health). This is the one consumer for which probe traffic is the whole point: that section's claim that probe traffic "keeps at least one authenticated route per upstream warm, bounding staleness" is only true if probes count. `/docs` and `/health/upstreams` label a last-success that came from a probe as `source: probe`, so "verified working" is never silently a claim about synthetic traffic alone.
  - **A deprecated fragment keeps probing** for as long as it is still served: its remaining callers are the ones coexistence exists to protect, and a dead credential there is an outage for them. Because probes are excluded from the retirement metric, continuing to probe cannot delay the retirement it would otherwise block. Probing stops when the fragment is removed. An **adapting** fragment never probes at all — it declares no `x-credential-probe` and holds no credential of its own (Version Migration Strategy §2); its target version's probe covers the single credential the version family shares.
- **The probe loop is leader-elected, always** (decided 2026-07-20). SEAM runs single-replica (Version Migration Strategy §1), so today exactly one process holds the loop — but the loop is the one stateful feature whose duplication is visible *outside* SEAM: two copies double authenticated probe traffic against credentials that several upstreams rate-limit, which turns a health check into the thing that breaks the credential. So the loop acquires a Kubernetes `Lease` before probing and drops it on `SIGTERM`, and a process that does not hold the Lease serves traffic normally but probes nothing. This costs almost nothing at one replica and makes both the future shared-state HA path and any accidental transient two-pod overlap safe by construction. `/health/credentials` is served by every replica from its own last-known results, which is why the endpoint reports `lastVerified` timestamps rather than implying live truth.
- **Secret-echo scrubbing belongs to the injection guarantee, not to this sentinel — it is a Phase 2 deliverable, not Phase 12** (decided 2026-07-20). Some upstreams echo the token back in an error body; if SEAM proxies that through, the agent sees the secret and the plan's single core guarantee ("the agent never sees the secret", Overview) is void. Injection ships in Phase 2 and agents are cut over in Phase 6b, so scheduling the scrub with the sentinel in Phase 12 would leave a multi-phase window in which the guarantee is simply false for any echoing upstream. Scrubbing therefore ships **with injection, in the same phase**, and applies to every injected route whether or not that route declares an `x-credential-probe`. Mechanics, stated because "scan the response" is not implementable as written against the streaming and WebSocket passthrough ADR-001 treats as a hard requirement:
  - **Buffered responses** (known `Content-Length` under **`maxBufferedResponseBytes`, default 1 MiB** — one of the two independent limits named above, the other being `maxReplayableRequestBytes`): decompress if `Content-Encoding` is one SEAM can decode, scan for the exact injected value(s) for that request, redact to `[REDACTED-BY-SEAM]`, re-encode. Response headers and trailers are scanned on the same terms — an echoed token in a `WWW-Authenticate` or a vendor debug header leaks exactly as well as one in a body.
  - **Chunked and event-stream responses:** scanned incrementally with a carry-over overlap — SEAM holds back the trailing `len(secret) - 1` bytes of each chunk and prepends them to the next, so a secret straddling a chunk boundary is still matched. This preserves streaming (no full buffering) and bounds added latency to one chunk's worth of tail. Where multiple secrets could be present, the overlap is the longest of them.
  - **Unscannable responses** — protocol upgrades (WebSocket and anything else that leaves HTTP framing), opaque binary bodies, and content encodings SEAM cannot decode — **cannot be scrubbed, and SEAM does not pretend otherwise.** On such a route, injection is refused by default: the fragment must carry `x-unscrubbable: acknowledged` to opt in, `seam lint` flags it for human review, and `/config/status` lists every route currently running unscrubbed so the exposure is enumerable rather than discovered. A route that both injects a secret and upgrades the connection is a deliberate, visible exception — never a silent one.
  - Scrubbing is **defense in depth, not the primary control.** It cannot catch a secret the upstream returns transformed (truncated, re-encoded, hashed), so it never licenses treating an echoing upstream as safe; such an upstream is a bug to report upstream and a candidate for a narrower credential.

## Per-Route Guards: Loop Breaker and Cost Governor

SEAM is the sole chokepoint for metered upstreams (twitterapi.io per-call credits; the z.ai/GLM quota that has no queryable API) and for runaway agent retry loops (precedent: NEEDLE re-dispatching one failing bead 310×/$500). Guards are **configured individually per endpoint** in the route fragment — there are no SEAM-wide defaults, and a route without guard fields gets no guard.

**Guard state is per-process, in-memory, and single-replica by construction** (decided 2026-07-20). Counters, budget ledgers, and their windows live in the gateway process and are lost on restart — which is safe only because SEAM runs exactly one replica. **A second replica would double every threshold in this section** and make `X-SEAM-Budget-Remaining` depend on which pod answered — the worked numbers and the full rationale live in Version Migration Strategy §1, which owns that decision and was amended for this reason. Guards are therefore never to be built assuming replication, and raising `replicas` is not a valid operational response to load; if HA is ever required, the shared quota/counter store named in §1 lands first. Restart-scoping is deliberate and stated so callers can rely on it: a SEAM restart resets budget windows and loop counters, so budgets are a runaway-spend brake, not an accounting system of record.

- **Loop breaker** (`x-loop-guard`): counts repeated identical requests keyed by (caller, route, normalized request hash); exceeding the route's threshold returns a structured 429 telling the agent, in plain language, that it has sent the same failing request N times and should stop and re-read `/docs/{route}`. Before Phase 7 identity exists, callers behind shared egress are indistinguishable, so the key degrades to (route, hash) — thresholds must be set with that in mind. **This ships in Phase 13 without waiting for identity** (decided 2026-07-20, resolving the corresponding Open Question): the pathology being caught is one agent replaying a byte-identical failing request, and the degraded (route, hash) key *aggregates* callers rather than splitting them, so it is strictly more conservative — it trips earlier than the identity-aware version, never later, and can never let a loop through that per-caller keying would have caught. The failure mode it does introduce is collateral: a second, innocent caller sending the same request can be 429'd by the first caller's loop. That is bounded and acceptable because the guard is opt-in per route (no `x-loop-guard`, no guard), thresholds on shared-key routes are set with headroom for legitimate concurrent repeats, and the 429 body states plainly that the guard is currently route-wide rather than per-caller so an agent can tell "I am looping" from "someone else is." Phase 7 narrows the key to (caller, route, hash) with no change to fragment config or thresholds' meaning — it only removes the collateral.

  **What "normalized request hash" means** (defined 2026-07-20 — the term carried the whole discriminating burden of this guard and was never specified, which decides whether it fires at all: with query strings included verbatim, an agent retrying with reordered parameters evades the guard entirely; with `X-SEAM-Dry-Run` included, a dry-run and its real counterpart hash apart. It matters most before Phase 7, where the key is only (route, hash)). The hash is taken over, in this order: the **uppercased method**; the **matched OpenAPI path template** together with its **resolved path-parameter bindings** sorted by name — the template alone would collapse every cluster of a multi-instance route into one counter, and the raw path would split `%2F`-vs-`/` spellings of the same request; the **query parameters sorted by name then value**, percent-decoded before sorting so encoding variants collapse; and the **canonicalized body** — JSON bodies in a canonical form (keys sorted, insignificant whitespace removed, RFC 8785-style), non-JSON bodies as raw bytes, an absent body as empty. **Everything else is excluded**, explicitly: all `X-SEAM-*` headers (so dry-run, version selection, and their real counterparts hash identically), all other caller-supplied headers, every header or query parameter SEAM injects or strips, and the credential itself. Headers are excluded wholesale rather than filtered, so no future injected header can accidentally become part of the key. Two consequences already stated elsewhere follow from this definition and are repeated here so nobody re-derives them: the hash is computed **after** the adapter request transform, so two callers on different `x-api-version`s sending semantically identical requests count as one loop (Version Migration Strategy §2); and a body larger than the replay cap `maxReplayableRequestBytes` (default 1 MiB — Credential Health Sentinel, where it is specified as a limit independent of the response-scrubbing cap, so tuning one never silently moves this behavior) is hashed over its streamed bytes without canonicalization, which is strictly coarser and can only cause the guard to under-match, never to over-match.
- **Cost governor** (`x-cost-per-call`, `x-quota`): per-caller budgets on routes that cost money; responses carry `X-SEAM-Budget-Remaining` so agents can self-throttle. **Both fields are objects carrying an explicit `unit`, and `x-quota` carries an explicit tumbling `window` that any restart resets** (decided 2026-07-20) — the shape, the rule that the two units must match, the 7-day lint ceiling on `window`, and the four-field `X-SEAM-Budget-Remaining` format (`amount`, `unit`, `window`, `resets`) are specified once in Data Models and deliberately not restated here. The `$50` figures used illustratively in this section and in §1 are shorthand for `{amount: 50, unit: usd, window: <duration>}`; SEAM converts between no two units, so a twitterapi.io budget is denominated in credits and a z.ai one in that upstream's own quota units. Doubles as the usage meter z.ai never shipped. Per-caller precision also sharpens with Phase 7. **Before Phase 7, `x-quota` degrades symmetrically with the loop breaker: it is enforced as a single route-wide budget** (decided 2026-07-20) — the key is (route) alone, every caller draws from one shared pool, and the pool is what `x-quota` names. Stating this is not pedantry: "per-caller budget" with no callers is unimplementable, and an implementer left to invent the degradation could plausibly choose per-source-IP (meaningless behind shared egress) or, worse, no enforcement at all — on the twitterapi.io credit and z.ai quota routes that is the difference between a capped spend and an uncapped one. Consequently `X-SEAM-Budget-Remaining` reports the **shared remainder**, and callers must not treat it as their own allowance: another agent can consume it between one response and the next request, so it is a fleet-wide fuel gauge, not a reservation. The guard is deliberately fail-closed — an exhausted route-wide budget 402/429s every caller, including well-behaved ones — because the alternative on a metered upstream is spending money SEAM was installed to stop spending. Phase 7 splits the pool per caller with no fragment change; `x-quota` is then read as the per-caller figure it always claimed to be. Note that an `_all` fan-out (Multi-Instance Routes) is billed as one call per instance actually dispatched, not one call, so a fan-out route's `x-quota` must be sized against the map's width.

  **The accounting point, stated once and generally** (decided 2026-07-20 — the `_all` clause above was the only case ever specified, and it implies dispatch-time billing without generalizing it; the unstated cases decide real spend). **A call is charged at dispatch, per upstream request actually issued.** Everything else falls out of that one rule:
  - **A request the upstream rejects still charges**, 5xx included. SEAM cannot know whether a metered upstream counted the call it just served, and on twitterapi.io credits or the z.ai quota the conservative direction is to assume it did — a governor that refunds on failure is a governor a failing loop can run for free.
  - **The 401-triggered refetch-and-retry charges twice**, because two upstream requests were issued. It is a real double draw and is not smoothed over.
  - **Anything refused before dispatch charges nothing**, because nothing was issued: a circuit-breaker 503 (Passive Route Health), a validation 400, a loop-breaker 429, a quota refusal itself, and a dry-run. This is the same boundary the pipeline already draws — guards sit at stage 9, dispatch at stage 11.
  - **An `_all` fan-out charges once per instance actually dispatched** — now a consequence of the rule rather than an exception to it. Instances the breaker refuses charge nothing, so a partly-open fan-out costs less than a full one.
  - **Sentinel probes charge nothing against the caller-facing budget** (Credential Health Sentinel), which is the one deliberate departure from "every issued request charges" and is documented there with its own reporting.
- **Dry-run** (`X-SEAM-Dry-Run: 1` request header): validate a request against the spec and return the verdict without injecting a secret, proxying, or spending quota — agents can test their contract understanding against expensive routes for free. **A dry-run also does not increment the loop-breaker counter** (decided 2026-07-20, completing the pipeline's stage-7 short-circuit): dry-run is offered as the *safe* way to iterate on a route, so counting repeats would 429 a caller for doing exactly what this plan recommends — iterating on one request's shape until it validates is the expected use, and it is byte-near-identical by construction. Dry-runs are still rate-limited by the ordinary transport, and they contact no upstream, so nothing about them is worth spending a guard on.

## Multi-Instance Routes and Fan-Out

Many upstreams are N near-identical instances of the same resource — the per-cluster kubectl-proxies all expose a byte-identical API. Instead of N copy-paste fragments, one fragment declares an `{instance}` path parameter (e.g. `/k8s/{cluster}/...`) resolved against an `x-upstream-map` lookup table, collapsing the config N:1. Each map entry is an object, **whose authoritative key set is specified once in Data Models** (`x-upstream-map`) and is not restated here — the short list this sentence used to carry was one of four disagreeing copies. What it exists for is the point: near-identical APIs routinely need per-instance credentials of differing kind and rotation cadence, and instances can differ in transport policy and in whether they are plaintext at all.

- The reserved instance `_all` fans the request out to every mapped upstream concurrently and returns one merged result labeled by instance — e.g. `/k8s/_all/api/v1/pods?fieldSelector=status.phase!=Running` answers "what's broken across the entire fleet" in one call.
- **Breaker state is per instance, not per fragment** (decided 2026-07-20): each `x-upstream-map` entry is its own upstream origin and gets its own circuit breaker under the policy in Passive Route Health, tunable by the entry's own optional `breaker` key (Data Models owns the full entry shape). One dead cluster therefore opens exactly one breaker and the other eight keep serving — without this, a single unreachable cluster would fail every instance of a collapsed fragment, and the N:1 collapse would have traded config duplication for a shared failure domain. An `_all` fan-out skips instances whose breaker is open, reports them as breaker-refused entries distinguishable from live errors, and is charged nothing for them (Per-Route Guards).
- **Partial-failure semantics for fan-out: 207 Multi-Status with per-instance result entries; fail-closed is rejected** (decided 2026-07-20, closing the corresponding Open Question, which this plan had already half-answered twice without noticing). The question listed fail-closed — any instance error fails the whole request — as a live candidate, but two decisions elsewhere had already foreclosed it: the breaker rule immediately above specifies that `_all` "skips instances whose breaker is open, reports them as breaker-refused entries distinguishable from live errors", which is a per-instance error-entry response body and is expressible only under fail-open or 207; and Per-Route Guards' "a partly-open fan-out costs less than a full one" presupposes that a partly-successful fan-out is a valid outcome that gets billed. An implementer choosing fail-closed would have contradicted both. It is also wrong on the merits for this route's whole reason to exist: `/k8s/_all/api/v1/pods?fieldSelector=status.phase!=Running` answers "what is broken across the fleet", and a fleet with one broken cluster is precisely when the answer is wanted — fail-closed makes the query fail exactly when it matters, and makes one dead cluster hide the other eight, restoring the shared failure domain the per-instance breaker was added to abolish.
  - **207 rather than fail-open 200**, because the summary must survive a naive client. An agent that checks `response.ok` and moves on is the expected consumer, and under a 200 a fan-out in which eight of nine instances failed is indistinguishable at the status line from a clean one — hard-to-miss summary counts in the body are only hard to miss for a caller that reads the body. 207 puts the partiality where no consumer can skip it, while remaining a 2xx, so the response body is still delivered and still useful.
  - **The status code is derived, never authored:** **200** when every dispatched instance returned 2xx and none was breaker-refused; **207** when any instance errored, timed out, or was breaker-refused; and — the one case that is not a fan-out result at all — **the structured 503 of Passive Route Health when every instance was breaker-refused and nothing was dispatched**, naming each refused instance and its `Retry-After`, since no upstream was contacted and nothing was charged. SEAM's own refusals (validation 400, authorization 403/404, guard 429, quota 402) keep their own status codes and never appear as 207: those resolve at stages 5-9, before the fan-out boundary, and describe the request rather than the instances.
  - **The response envelope shape is specified in Data Models** ("fan-out response envelope"), so instance results, the summary counts, and the `X-SEAM-Fanout-Partial` header have one authoritative definition.

## Passive Route Health — Last-Success Tracking

`/docs` shouldn't just describe routes; it should demonstrate them. The signal is passive, derived from live traffic rather than synthetic probes:

- The gateway records the timestamp of the most recent successful (2xx) response per route (its "last-200"). `/docs` renders it per path — "last succeeded 2m ago" — an honest signal distinguishing verified-working from merely-documented. **Three states are rendered, not two** (decided 2026-07-20, superseding the two-state rendering of 2026-07-16). The original decision was sound in what it avoided — "never succeeded" reads as broken for a freshly added route nobody has called yet — but it conflated two distinct states under one string, and it did so in the case that matters most: the condition tested was *no successful call* while the text asserted *no attempt*, so a route called fifty times since restart and failing every one of them rendered "no attempt since last restart" and read as untested-and-new when it actually meant consistently-broken. That inverts this section's entire stated purpose on the single worst route in the fleet. So the render is keyed on attempts and successes separately:
  - **No attempt since restart** → "no attempt since last restart". The freshly-added-route case, and the only one the original wording described correctly.
  - **Attempted, none successful** → "no success in N attempts since last restart", carrying the **last error** alongside it — which costs nothing to surface, since the circuit breaker already tracks exactly that value per upstream origin for its own structured 503. This is the state that was previously invisible, and it is the one an operator most needs to see.
  - **At least one success** → "last succeeded 2m ago", unchanged, with `source: probe` labeling where the success came from sentinel traffic rather than a real caller (Credential Health Sentinel).

  The last-success value itself stays **intentionally empty** in the first two states rather than being filled with a sentinel timestamp; it is the surrounding text that now distinguishes them. `/health/upstreams` renders the same three states on the same terms. Tracking is in-memory and restart-scoped, deliberately not persisted — which is why every one of these strings is explicit that it is scoped to the current process lifetime.
- `/health/upstreams` aggregates the same data per upstream into a single status page for the whole mesh: most recent success across the upstream's routes plus current breaker state.
- Per-upstream circuit breaker: consecutive failures open the breaker, and callers get a structured 503 naming the upstream, when it started failing, the last error seen, and a retry-after — so agents stop burning tokens retrying a dead service.

  **The breaker's policy, and why it has a default at all** (decided 2026-07-20 — an open breaker fails *every* route on an upstream, so leaving the threshold, the failure definition, the open duration and the close condition to the implementer makes a wrong guess a self-inflicted fleet outage; and no fragment field existed for any of it):
  - **What counts as a failure: transport and server faults only.** A connection refused or reset, a DNS or TLS handshake failure, and a request timeout count; so does any upstream **5xx**. **No 4xx ever counts** — a 4xx proves the upstream is alive and answering, which is the only question the breaker asks. A **401 never counts either**, even one that triggers refetch-and-retry: a credential rotation is not an upstream outage, and counting it would let a routine rotation open the breaker on a perfectly healthy service. Credential state is the sentinel's job and is reported at `/health/credentials`. Failures SEAM produces itself — a validation 400, a guard 429, an adapter 500 — never count, because they never reached the upstream.
  - **Threshold and open duration: 5 consecutive failures opens the breaker for 30 seconds.** Consecutive, not a rate: any qualifying success resets the count to zero, so an upstream that is merely flaky stays closed and only one that is genuinely down trips.
  - **Half-open closes on evidence, not on a timer.** When the open interval elapses the breaker admits **one** real caller request as a trial — no synthetic probe, so a dead upstream is never kept warm by the breaker itself. Any response that is not a failure by the definition above (including a 4xx) closes it; another failure re-opens it with exponential backoff — 30s, 60s, 120s, capped at 5 minutes — so a long outage decays to roughly one trial request every five minutes rather than one every thirty seconds.
  - **`Retry-After` is derived, never guessed:** it is the seconds remaining in the current open interval, floored at 1, and the same value appears in the structured 503 body. An agent that honors it arrives exactly when the trial slot opens.
  - **Where the config lives, given that fragments are per-route and the breaker is per-upstream:** breaker state keys on the **resolved upstream origin** (scheme + host + port), and for a multi-instance route on the resolved origin of each `x-upstream-map` entry — so one dead cluster opens only its own breaker and the other eight keep serving, which is the behavior the `_all` fan-out needs. Config is declared at fragment root as **`x-breaker`** (see Data Models), which is well-defined because a fragment names exactly one upstream target by construction, with a per-instance `breaker` override inside `x-upstream-map`. Where two fragments resolve to the *same* origin and declare disagreeing `x-breaker` blocks, `seam lint` fails the PR, and at runtime the strictest values govern (lowest threshold, longest open duration) with the disagreement named at `/config/status` — the gateway never trusting lint to have run, as everywhere else here.
  - **This does not violate "no SEAM-wide defaults, guards configured individually per endpoint"** — that principle (Per-Route Guards) was written about the **cost governor and loop breaker**, and its reason does not transfer. Those guards spend money and refuse traffic on the author's behalf, so an implicit default would either cap a route nobody sized or, worse, silently fail to; absence of the field there is correctly read as "no guard." The breaker is **infrastructure-level availability protection**, where absence of a default is itself the failure mode: an upstream with no breaker means every agent hammering a dead service burns tokens indefinitely, which is the exact waste this section exists to stop. So the breaker is **on by default for every upstream** with the policy above, and `x-breaker` only *tunes* it — the two rules are not in tension because they answer different questions, and this is stated rather than left as an apparent contradiction. `x-breaker.enabled: false` exists for the rare upstream where failures are routine and a breaker would do more harm than good; it is lint-flagged and enumerated at `/config/status`, like every other opt-out in this plan.
- **Both signals are per-process and single-replica by construction** (decided 2026-07-20). Last-2xx tracking is already declared in-memory and restart-scoped; breaker state is the same. This is coherent only at one replica (Version Migration Strategy §1, amended for this reason): with two, `/docs` would show a different "last succeeded" on each refresh depending on which pod rendered it, and the breaker would open on one replica while the other kept hammering a dead upstream — the breaker's entire purpose defeated by the deployment topology. Neither signal may be built assuming replication; if HA ever lands, both move into the shared state store named in §1. The restart-scoping remains deliberate, and its honest rendering — "no attempt since last restart" — is what keeps a restarted gateway from reading as a broken one.
- Synergy with the credential sentinel: its probe traffic keeps at least one authenticated route per upstream warm, bounding staleness of the signal for probed upstreams.

## Fragment Toolchain (`seam` CLI)

The config surface is YAML fragments authored largely by agents and merged at runtime — one bad file must never break the gateway, and mistakes should be caught before merge, not at reload.

- **Runtime quarantine:** a fragment that fails schema validation or collides with an already-loaded fragment is skipped, logged, and surfaced at `/config/status` (loaded fragments, versions, quarantined ones with their errors) — never fatal to the remaining routes. **The collision key is the triple (path, method, `x-api-version`)** (decided 2026-07-20), never the path alone: two fragments declaring the same path and method but *different* `x-api-version` values are a legal coexistence pair, not a collision — that is precisely the shape a breaking change ships as (see Version Migration Strategy §2), so keying on path alone would quarantine every versioned migration and silently dismantle the coexistence model the whole version strategy rests on. A fragment carrying no `x-api-version` is keyed under the reserved version `_unversioned`, so two unversioned fragments on the same (path, method) still collide as before. Only an exact triple match is a collision, and the loser is always the later-merged fragment (deterministic filename order), never the incumbent — a bad new fragment cannot evict a working route.
- **Reserved control-plane namespace** (decided 2026-07-20; Phase 1a deliverable). The collision rule above is fragment-vs-fragment; nothing in it protects SEAM's *own* surface, which now spans `/docs`, `/docs/{route}`, `/openapi.json`, `/whoami`, `/scopes`, `/changes`, `/health/credentials`, `/health/upstreams`, and `/config/status` and keeps growing. Since fragments are authored largely by agents, a fragment declaring `/whoami` or `/config/status` is not a hypothetical: depending on a match order this plan never specified, it would either shadow SEAM's endpoint or be silently shadowed by it, and the first case is a spoofing surface the moment agents make authorization decisions from those responses. Three rules close it. **(1) The reserved set is explicit and closed:** the exact paths listed above, plus everything under the prefixes `/docs/`, `/health/`, `/config/`, `/approvals/`, and `/_seam/`. `/_seam/` is reserved *forward* — every control-plane endpoint added after this decision lives under it, so the reserved set stops accreting top-level names into the space fragments legitimately use. **Its first concrete users are `/_seam/healthz`, `/_seam/readyz` and `/_seam/metrics`** (Phase 1a), alongside the `/_seam/approvals/{id}` example below: all four post-date the closed enumeration and therefore take the prefix, and all fall under the already-reserved `/_seam/` so none needs a new reservation. The top-level names that stay put do so on a criterion that is actually true (**corrected 2026-07-20** — this previously grandfathered "the names already published to agents and CLAUDE.md", which was false in both halves: SEAM has zero implementation and no endpoint of its appears anywhere in CLAUDE.md, and `/scopes` was itself invented on 2026-07-20, in the same editing round as the rule grandfathering it as pre-existing. As written the "closed" set was reopened by the decision that closed it, and the precedent invited the next new endpoint to claim grandfathering too). The honest criterion, and the whole of it: **the grandfathered names are exactly those fixed in this plan as of 2026-07-20 — `/docs`, `/docs/{route}`, `/openapi.json`, `/whoami`, `/scopes`, `/changes`, `/health/credentials`, `/health/upstreams`, `/config/status`.** That enumeration is closed and may not be appended to; every endpoint conceived after this date takes `/_seam/`, with no exceptions and no second grandfathering clause. **`/approvals/` is reserved but not served** (decided 2026-07-20): approval-gated routes (`docs/notes/approval-gated-routes.md`, accepted 2026-07-16 and unphased) declare a top-level `/approvals/{id}` that agents poll for an approve/deny verdict — post-dating nothing in the enumeration above and therefore, by the forward rule, shipping as **`/_seam/approvals/{id}`** when it ships. The bare `/approvals/` prefix is reserved anyway, as a squat guard rather than a served surface: as written today a fragment could legally claim it, and a fragment occupying the obvious name for an approval verdict is a forgeable-approval surface, so denying the near-miss name costs a prefix nobody needs and closes the hole in both directions. **(2) Control-plane routes always win the match**, unconditionally: SEAM resolves a request against its own surface before consulting the merged route table, so no fragment can shadow a control-plane endpoint even if one is somehow loaded. **(3) A fragment declaring any reserved path is rejected twice** — `seam lint` fails the CI gate in declarative-config, and the gateway quarantines the whole fragment at merge time with the offending path named at `/config/status`, never trusting lint to have run (the same belt-and-braces posture as the OpenBao allowlist in Architecture). Quarantine is fragment-wide rather than per-path so a fragment cannot smuggle a reserved declaration through alongside valid routes.
- **Reserved is not the same as protected: who may *call* the control plane** (decided 2026-07-20). The rules above stop a fragment from *shadowing* SEAM's own surface and say nothing about who may read it — and three of those endpoints return precisely the reconnaissance this plan works hard to withhold everywhere else. `/config/status` enumerates every loaded fragment, every quarantined fragment with its error text, every route forwarding a credential over an unverified or plaintext connection, and "every route currently running unscrubbed" — a directory of routes where a secret provably echoes into responses. `/health/credentials` enumerates every (fragment, instance) credential SEAM holds, with expiry. `/health/upstreams` enumerates the upstream mesh. `/scopes?all=1` is gated behind `seam:scopes:read-all` on exactly this reasoning; these three bypassed it. The control-plane surface is therefore **two tiers, not one**:
  - **Caller-facing and scope-filtered** — `/openapi.json`, `/docs`, `/docs/{route}`, `/whoami`, `/scopes`, `/changes`. Any authorized caller may call them; what they *contain* is reduced to the caller's own grants (Per-Agent Tool Scoping point 5), so there is nothing to withhold at the endpoint level.
  - **Operator-only and default-denied** — `/config/status`, `/health/credentials`, `/health/upstreams`. These require the scope **`seam:ops:read`** and are refused to every caller without it. They are not scope-filterable in any useful sense: a fleet-wide inventory reduced to one caller's grants stops being the operational view it exists to be, and the quarantine, unscrubbed-route and credential-expiry data it carries has no per-caller slice at all. Refusal is a **403 naming `seam:ops:read`**, not the 404-identical-to-unknown rule that governs fragment routes — these are fixed, documented paths whose existence this plan publishes, so hiding them buys nothing and a scope-naming 403 leaks nothing. `/scopes?all=1` keeps its own narrower `seam:scopes:read-all` gate unchanged.
  - **Before Phase 7 there are no scopes, so the boundary is the network one.** Until identity exists, `seam:ops:read` cannot be evaluated, and leaving these three ambiently readable in the meantime would hand the whole inventory to anything that clears the ACL — during exactly the window (Phases 1b-6b) when the Pre-Phase-7 Trust Boundary is honest that the boundary is "coarse and node-level, not per-agent." So in that window the operator surface is **bound to a separate listener port**, and the tailnet policy file grants that port only to the operator source set while the proxy port is granted to the worker tags. That is enforceable with no identity at all, at the layer the trust boundary already mandates. At Phase 7 the `seam:ops:read` check becomes the real control and the port split degrades to defense in depth — the same relationship Pre-Phase-7 Trust Boundary already describes between the ACL and Grant-based enforcement.
- **`seam lint`:** the gateway's own merge/validation engine packaged as a CLI, run as a CI gate in declarative-config — validates fragment schema and detects collisions — on the same (path, method, `x-api-version`) key defined above — before a ConfigMap ever ships.
- **`seam diff`:** renders the effective merged-spec change a PR would cause, for review.
- **`seam import --from-url`:** bootstraps a fragment from an upstream's own published OpenAPI spec (ArgoCD and Kubernetes both publish one), filtered to selected paths/verbs; injection metadata (`x-vault-path` etc.) is then curated by hand. Onboarding a new upstream drops from authoring YAML from scratch to curating a generated file.

## Pre-Phase-7 Trust Boundary

**All inbound caller authentication and authorization lands in Phase 7. Everything before it runs behind a network trust boundary, and that boundary is a hard requirement, not an operational detail** (decided 2026-07-20). Stating it is not belt-and-braces: by the 6b cutover SEAM is a live credential-injecting proxy — it holds the z.ai/GLM and twitterapi.io credentials that cost money per call, and (after Phase 5) reaches cluster API servers — while CLAUDE.md prose is being deleted to push agents onto it. Without this section an implementer could read the phase list and reasonably ship that as an unauthenticated open proxy on the tailnet, which is the worst outcome available: every credential SEAM holds, reachable by anything that can route to it.

- **Tag-restricted tailnet ACL, enforced in the tailnet policy file.** SEAM's Tailscale node is reachable only from an explicit, enumerated source set (e.g. `tag:needle-worker`, `tag:devpod`, the operator workstation) via a grant in the tailnet policy file. Default-deny at the network layer — not "on the tailnet, therefore allowed." Tailnet membership is not the boundary; the ACL is.
- **The ACL is written per port, because the control plane is two tiers** (decided 2026-07-20). SEAM binds two listeners from Phase 1a — the caller-facing proxy port and an operator-only port carrying `/config/status`, `/health/credentials` and `/health/upstreams` (Fragment Toolchain, "who may *call* the control plane") — and the policy-file grant enumerates them separately: worker tags reach the proxy port only, and the operator source set alone reaches the operator port. This is what makes the two-tier rule enforceable in the window where it must be, since `seam:ops:read` cannot be evaluated before Phase 7 and a node-wide grant would publish the whole fragment/credential/unscrubbed-route inventory to every caller that clears the ACL. At Phase 7 the scope check becomes the real control and the port split degrades to defense in depth, exactly as the ACL itself does below.
- **No Funnel, no public ingress, no Cloudflare Tunnel.** SEAM is not exposed to the public internet in any phase before 7. The foreign-worker/non-tailnet ingress path (Per-Agent Tool Scoping point 7) is explicitly a post-Phase-7 phase of its own — **that phase is now numbered: Phase 14** (assigned 2026-07-20; edge `7 → 14`) — and must not be stood up early "just for testing" — it is the one path that would put an unauthenticated credential-injecting proxy on the open internet. Phase 14 is the *only* phase that lifts this rule, and it lifts it having first satisfied point 7's four rules; no earlier phase may lift it for any reason.
- **The ACL governs tailnet ingress and does not govern the in-cluster pod network** (stated 2026-07-20 — an omission that reads as a contradiction the moment SEAM has probes). From Phase 6a SEAM is a pod on rs-manager, and **kubelet reaches `/_seam/healthz` and `/_seam/readyz` at the pod IP, and the VictoriaMetrics scrape reaches `/_seam/metrics` at the pod IP on the operator port** (Phase 1a) — neither traverses the tailnet, so neither is subject to the policy-file grant, and no ACL construction could make them so. This is not a hole to close but a boundary to state: the pod network is a distinct trust surface governed by the cluster's own posture (namespace isolation, NetworkPolicy if it is ever wanted), and the two-tier port split is what keeps the *tailnet* side honest. It is stated because an operator reading the ACL as *the* boundary would otherwise conclude the operator port is unreachable to the scraper, go looking for a third listener, and find none.
- **The 6b cutover is GATED on that ACL existing.** Deploying SEAM (6a) with a permissive ACL is tolerable while nothing is cut over and no real agent traffic flows; deleting a service's CLAUDE.md prose and pointing agents at SEAM (6b) is not. The ACL grant is a checklist item in the 6b change, verified before the first service is cut over, not after.
- **Residual risk, stated plainly.** This boundary is **coarse and node-level, not per-agent**. Every caller that clears the ACL gets every route SEAM serves; a compromised or merely buggy worker on an allowed tag has the same access as any other. Guards (Phase 13) cap the *spend*, not the *reach*, and before Phase 7 they key on (route, hash) rather than per-caller — so a single misbehaving agent can consume a route-wide budget and 429 well-behaved peers. Accept this only because the alternative — deferring all of Phases 1-6 until identity exists — trades a bounded, known exposure for no delivery at all.
- **Phase 7 replaces this boundary; it does not merely add to it.** When Grant-based scope enforcement lands, per-route authorization becomes the real control and the ACL degrades to defense in depth (a coarse outer fence around a fine inner one). Nothing in Phase 7 is designed on the assumption that the ACL will still be there, and nothing before Phase 7 may be designed on the assumption that Phase 7 will arrive soon.

## Implementation Phases

**Phase numbers are stable identifiers, not execution order** (decided 2026-07-20). They are referenced from a dozen places in this document and from filed beads, so renumbering them to express sequence would break more than it fixes — and the previous convention of implying order by number had already produced a contradiction the plan flagged but never resolved (Phase 10 "should land before (or with) Phase 5"). Order is therefore stated once, explicitly, below; a phase's number says only which phase it is.

**Ship order (topological):**

`1a → 1b → 9a → 2 → 6a → 3 → 4 → 10 → 5 → 11 → 13 → 6b → 7 → 12 → 8 → 9b → 14`

(**Phase 11 moved ahead of 6b and Phase 14 appended, 2026-07-20** — see the `11 → 6b` and `7 → 14` edges below.)

**The dependency edges that force it** — anything not implied by these is scheduling preference, free to reorder:

- `1a → 1b → everything`: nothing merges, validates, or proxies without the core.
- `1b → 9a → 3`: `seam lint` must be a live CI gate **before the first fragment ships**, not after the last. Fragments start shipping in Phase 3 and every phase through 6b adds more; landing lint at the end would mean the entire pilot and cutover ships with runtime quarantine as its only validation — exactly the outcome Fragment Toolchain's design intent ("caught before merge, not at reload") rejects. It also lands the `x-vault-path` allowlist and reserved-namespace checks *before* the surface they protect, rather than after.
- `2 → 6a → 3`: fragments arrive as mounted ConfigMaps, so the gateway must be running in the cluster before the fragment-delivery phase, not two phases later.
- `OpenBao role/policy provisioning → 2`: the `seam` Kubernetes-auth role and the `read`-on-`seam/routes/*` policy must exist in OpenBao **before** Phase 2, because Phase 2 is the phase that authenticates against them. They were previously listed as Phase 6a deliverables, which put the server side of the login one phase *after* the client side that needs it (decided 2026-07-20; see Phase 2). This is a precondition rather than a phase — it is an OpenBao configuration change with no SEAM dependency of its own — so it constrains the order without appearing in the topological line above.
- `10 → 5`: multi-instance routes precede kubectl-proxy onboarding. Reversed, every cluster in the Phase 5 target set named in Hosting gets a copy-pasted single-instance fragment and Phase 10 becomes a migration of already-shipped ConfigMaps instead of a greenfield build — the N:1 collapse is the entire point of onboarding those upstreams that way.
- `4 + 13 → 6b`: the cost governor is live before agents are cut over onto metered upstreams. Phase 4's z.ai and twitterapi routes spend real money per call; 6a/4 can carry them with no agent traffic on them, 6b cannot.
- `11 → 6b`: **the circuit breaker is live before agents are cut over — a forcing edge on exactly the reasoning that forced `4 + 13 → 6b`** (recorded 2026-07-20, and Phase 11 moved ahead of 6b in the topological line above to satisfy it). Passive Route Health argues the breaker is categorically different from the opt-in guards precisely because "absence of a default is itself the failure mode: an upstream with no breaker means every agent hammering a dead service burns tokens indefinitely, which is the exact waste this section exists to stop." The plan then created the forcing edge for the cost governor and none for the breaker, leaving the `6b → 7` cutover window — the first window in which real agent traffic meets live metered upstreams — as the one stretch where that failure mode is live and unmitigated, with nothing bounding token burn against a dead z.ai or twitterapi.io but agent-side retry policy nobody controls. Its own argument therefore contradicted its own sequencing. The move is free: **Phase 11 depends only on 1b and 2**, both of which already precede 6b, so nothing else resequences and no fragment config changes.
- `OpenBao role/policy for seam-retirement-evaluator → 8`: **a precondition mirroring the `OpenBao role/policy provisioning → 2` edge above** (recorded 2026-07-20). Phase 8 ships `seam-retirement-evaluator` as its own deployable with its own identity (Components, Hosting), and it authenticates to OpenBao for a Forgejo write credential stored at its own path **outside** `seam/routes/*`. That path, its `read` policy, and the Kubernetes-auth role bound to the evaluator's own ServiceAccount must exist **before** Phase 8, or Phase 8 has precisely the server-side-missing problem Phase 2 was corrected for — a client authenticating against a role nobody created. Read access to the VictoriaMetrics instance is provisioned in the same precondition, since the evaluator is a query and cannot run without it. Like the Phase 2 precondition this is an OpenBao/metrics configuration change with no SEAM dependency of its own, so it constrains order without appearing in the topological line.

  **Retirement workflow — the adopted composition** (decided 2026-07-20, adopting the recommendation in `docs/notes/quiet-window-options.md`, which this phase previously named only as "usage-gated retirement workflow" while leaving the mechanism unselected): **(D) PR-gated human retirement is the always-on gate** — when the quiet condition is met, the declarative-config PR is opened with the traffic evidence in its description and a human merges it; nothing is ever removed autonomously, which costs a little latency and fits the GitOps rule that all cluster changes go through declarative-config anyway.

  **The evaluator and the PR-opener are an out-of-band job, and emphatically not the gateway** (decided 2026-07-20 — correcting this phase's earlier "**SEAM** opens the declarative-config PR", which was both a security escalation and inconsistent with §2's own decision that the gap history lives in the metrics store rather than in gateway memory). Two independent reasons, either sufficient:
  - **Credential scope.** Opening a PR requires a Forgejo write credential against declarative-config — the repository that controls every cluster in the fleet. SEAM's OpenBao policy grants `read` on `seam/routes/*` **and nothing else**, and the entire hostile-fragment threat model in Architecture rests on that being literally true: a fragment authored by an agent cannot reach material SEAM's own token has no capability for. Handing a credential-injecting proxy write access to the GitOps repo would make "compromise SEAM" and "commit to every cluster's desired state" the same event, and would do it for a convenience feature. The policy stays closed, with **no named exception**.
  - **State location.** §2 already establishes that the observed max inter-request gap and "quiet since" are read from the durable per-route-version counter in the metrics store, not from gateway memory, precisely because every in-process store here is restart-scoped and would reset the quiet clock on each binary deploy. The evaluator is therefore a *query*, and a query has no reason to live in the request path.

  So retirement evaluation ships as **`seam-retirement-evaluator`, a small scheduled job** (a Deployment with an internal scheduling loop, per this fleet's no-CronJobs rule) that runs **read-only against the metrics store**, computes the window per route-version, and opens the declarative-config PR under **its own narrowly-scoped Forgejo credential** — a token that may open PRs against declarative-config and holds nothing else, stored at its own OpenBao path outside `seam/routes/*` and readable only by that job's identity, never by SEAM. It consumes no gateway API, no gateway memory and no gateway credential; SEAM's only contribution is the exported metric, which it already publishes. This adds **no** request-path state, **no** PVC, and **no** second stateful component to the single-replica gateway. Brownout scheduling (C) stays in the gateway, since serving a `410` is a request-path behavior and needs no credential at all. **(B over A) the window is traffic-adaptive with a calendar floor** — `max(3 × observed max inter-request gap on that route-version, 7 days)` — since a fixed window must exceed the longest inter-call gap of any live caller, a number nobody knows; the floor covers cold route-versions with too little history to self-size. This does not reopen the usage gate: zero observed traffic remains a necessary condition, and the calendar term can only lengthen the wait (see Version Migration Strategy §2). **(C) brownout 410s in the final week** — the deprecated route-version is served as a structured `410 Gone` for short scheduled windows, body naming the replacement version and pointing at `/changes`, converting "we believe nobody calls this" into a tested claim that an LLM caller can actually read and self-migrate from. This is a bounded, scheduled, pre-announced exception to the coexistence guarantee, not a hole in it: it applies only to a route-version already past its quiet threshold and already carrying `Deprecation`/`Sunset`, and a caller that appears during a brownout **cancels the retirement** rather than being sacrificed to it. **(E) post-Phase-7 upgrade to caller-complete drain** — retire once every identity that ever called v-old has either called v-new or left the tailnet, within a bounded lookback; this replaces the window approximation with the thing it approximates and shrinks retirement latency from weeks to days. Depends on 1b (fragment loading + spec-version header); the per-caller breakdown and (E) depend on Phase 7.
- [ ] Phase 9a: `seam lint` — the gateway's merge/validation engine packaged as a CLI and wired as a declarative-config CI gate. Adding a check to a CLI that already exists is genuinely packaging; writing one against a field nobody may yet legally author is not — that is the distinction the original rationale collapsed.

  **The full check list, enumerated** (decided 2026-07-20 — every rule elsewhere in this plan that says "`seam lint` rejects/flags this" is a deliverable of *this* phase, and a rule whose enforcement is asserted in one section and scheduled in none is an unenforced rule):

  - **`x-breaker` conflicts** *(lands with Phase 11, which introduces the field)* — two fragments resolving to the same upstream origin with disagreeing `x-breaker` blocks fail the PR, and every `x-breaker.enabled: false` is flagged for review like the other opt-outs.
  - **`x-cost-per-call` / `x-quota` coherence** *(lands with Phase 13, which introduces both fields; added to this enumeration 2026-07-20)* — three checks, because each catches a defect that reads as a working budget while capping nothing or capping the wrong amount: **`x-quota.unit` must be byte-identical to the same route's `x-cost-per-call.unit`** (a mismatch is a hard error and a merge-time quarantine, since a mixed pair draws a credit balance down in dollars); **`x-quota` without `x-cost-per-call` on the same route is a hard error** (a budget with no per-call draw can never be decremented, so it caps nothing while reading as though it does); and **`window` must match `^[0-9]+(s|m|h|d)$`, with a warning — not an error — on any `window` exceeding `168h`**, since the window is tumbling and restart-scoped and a budget nominally longer than the deploy cadence is a cap that quietly resets (Data Models). The over-`168h` routes are enumerated at `/config/status` as well, so the condition is visible at runtime and not only at the CI gate.

  **The cross-field constraints the schema must encode are themselves now decided**, and are the part most likely to be missed by a schema written field-by-field (recorded 2026-07-20): `x-seam-deprecated` is an **object `{since, sunset?}`** with `since` required — not the boolean it was first specified as, and a bare `true` must fail rather than coerce; `x-api-version` has a **closed grammar** `^v[1-9][0-9]*$` with the literal `_unversioned` rejected as an author-written value; `x-adapter` is **mutually exclusive with `x-upstream` / `x-upstream-map`** and with every other upstream-facing field (`x-vault-path`, `x-inject-as`, `x-upstream-tls`, `x-upstream-plaintext`, `x-credential-probe`, `x-breaker`), which are taken from the target version's fragment; `x-upstream` and `x-upstream-map` are mutually exclusive with each other, and `x-upstream-map` requires an `{instance}` path parameter; `x-upstream-plaintext: acknowledged` is **required** whenever the resolved URL is `http://`; and `insecureSkipVerify`, `x-upstream-plaintext` and `x-unscrubbable` each accept the single value `acknowledged` with **no `false` form**. **Four further cross-field constraints, added 2026-07-20:** `x-quota.unit` must be **byte-identical** to the same route's `x-cost-per-call.unit` (a mixed pair draws a credit balance down in dollars); **`x-quota` requires `x-cost-per-call` on the same route**, since a budget with no per-call draw can never be decremented and so caps nothing while reading as though it does; `x-quota.window` obeys the closed duration grammar **`^[0-9]+(s|m|h|d)$`** with no calendar-aligned form expressible, and a value above `168h` is a **warning, not a schema failure** (the window is tumbling and restart-scoped, so a budget longer than the deploy cadence is a cap that quietly resets — visible rather than forbidden); Each of these is a constraint no per-field schema catches on its own.

  **One constraint the JSON Schema must *assume* rather than encode, and must not be written as though it did not exist** (recorded 2026-07-20): the **upstream-host allowlist is manifest-level, not fragment-level**. The set of permitted `x-upstream` / `x-upstream-map`-entry hosts lives in SEAM's own Deployment manifest and deliberately **never** in a fragment — a fragment-supplied allowlist is self-certifying, since the author naming the malicious host also names the rule permitting it. So the fragment schema can validate `x-upstream` as a well-formed absolute URL and can reject IP literals, but **it cannot decide allowlist membership**, which is checked by `seam lint` against the manifest and re-checked by the gateway at merge time with fragment-wide quarantine. The schema must therefore not be written as if a syntactically valid `url` were an accepted one. Accordingly, add to the constraints-no-per-field-schema-catches list above: **`x-upstream` / `x-upstream-map` entry `url` host ∈ the operator-owned allowlist** — the peer control to the `x-vault-path` prefix, and the one that bounds *where a secret is sent* rather than *which secret is named*. Two forward constraints on that schema: it must leave room for **`x-requires-approval`** (`docs/notes/approval-gated-routes.md`, accepted but unphased — an accepted feature that constrains the schema must not be discovered after the first fragment ships), and `x-seam-schema` remains the escape hatch if evolution is needed anyway.

## ADR-001: 2026-07-20 — Ratify Go as the SEAM Gateway Implementation Language

### Context

Phase 1 (gateway core) has been blocked since the repo was scaffolded on 2026-07-15 — as of this writing (2026-07-20) the repo contains only `docs/`, zero implementation code. `docs/research/language-runtime-choice.md` (2026-07-16) benchmarked Go, Rust, TypeScript/Node, and Python against SEAM's seven hard requirements (3.1 fragment merge, genuine 3.1 request validation with structured field-level errors, reverse-proxy with streaming/WebSocket passthrough, OpenBao client, Tailscale WhoIs/tsnet, file-watch hot reload with atomic swap, small-footprint k8s container), with library-version and 3.1-conformance claims checked against actual registries/repos/changelogs rather than recalled from memory, and the Python findings additionally executed live on this host. The plan's Open Questions section has carried "decision pending review" since that research landed, and every phase from 1 onward depends on it. This ADR closes that gap.

### Decision

Adopt **Go** as the SEAM gateway's implementation language, per the research's primary recommendation.

1. **The validator is the product.** SEAM's differentiating feature is schema-validated, structured, field-level 400 responses. pb33f `libopenapi` + `libopenapi-validator` is the only mature, off-the-shelf OpenAPI 3.1 request validator found across all four languages surveyed, and its error model (JSON pointer + spec line/col + `HowToFix`) maps directly onto SEAM's structured-error feature — it is better than what we would hand-design. Every other language means either building this by hand (Rust) or accepting a wiring/bus-factor risk on SEAM's core value proposition (Python, Node).
2. **Tailscale support is first-party and production-grade in Go alone.** `tsnet` (embedded node) and `client/local` (WhoIs against a sidecar) are maintained by Tailscale itself, directly relevant to the pilot phases and to Phase 7's per-agent Grant-based scoping. The Rust equivalent (`tailscale-rs`) is explicitly experimental with no WhoIs API as of this research.
3. **The proxy layer's security posture comes free.** stdlib `httputil.ReverseProxy`'s `Rewrite` hook strips hop-by-hop and client-spoofable `X-Forwarded-*` headers before user code runs, by design — closing the exact secret/header-injection pitfall flagged in `docs/research/secret-injection-gateways.md`, rather than requiring SEAM to reimplement that hardening itself.
4. **The cost is real but bounded and one-time.** Go is a new toolchain in a shop whose CI is otherwise Rust-centric (`rust-verify` on iad-ci), but that cost is paid once at setup, not per-route the way a hand-rolled validator's maintenance would recur.

### Alternatives Considered

- **Rust** (the shop's primary language; `rust-verify` remote CI already exists) — clears six of seven requirements with best-in-class parts (axum/hyper/tower proxy stack maps cleanly onto validate→strip→inject→forward; `jsonschema` 0.48 is the most actively maintained 2020-12 engine surveyed; best footprint at ~10-25MB). The one gap is the product's core feature: no mature off-the-shelf 3.1 request validator exists (the sole candidate is a 45-star crate with no crates.io release). Building one is a scoped, bounded effort (~1-2k LOC, 1-2 weeks, per the research) — a defensible runner-up if "no new language in this shop" is weighted above "don't own the validator," but it puts SEAM's differentiator on our own maintenance budget instead of an actively-maintained upstream's. Kept as the fallback (see Consequences).
- **Python** — every requirement clears, and uniquely among the four candidates was verified by live execution on this host (openapi-core 0.23.1: 0.59ms/request validation, 16ms full spec rebuild; a real WhoIs call over `tailscaled`'s Unix socket). Ruled out on footprint (170-200MB image vs. Go's 15-30MB) and on openapi-core being a single-maintainer dependency with a documented history of breaking 0.x API churn — a bus-factor risk on the validator, the same category of risk this decision is trying to avoid.
- **TypeScript/Node** — Ajv2020 is the most battle-tested 2020-12 schema engine surveyed, but every library wrapping it for OpenAPI made an overstated 3.1 claim under inspection (openapi-backend silently runs draft-07-mode Ajv despite its README's 3.1 checkbox; express-openapi-validator is Express-bound and drags in `multer`). WebSocket proxy support across the viable options is self-described "partial" (first subprotocol only). No footprint advantage over Go to compensate.

### Consequences

- **The LANGUAGE blocker is closed; Phase 1 is not wholly unblocked** (amended 2026-07-20 — this bullet previously read "Phase 1 (gateway core) is unblocked and can start as a Go scaffold", which overclaimed: this ADR resolved the language question and was then read as having resolved Phase 1's blocker generally). Phase 1's original deliverable included fragment merge, validation, and quarantine — none of which can be built against a fragment schema that does not exist, which is why bead `bf-4ks` ("Scaffold Phase 1 gateway core in Go") remained BLOCKED while this ADR asserted the opposite. Phase 1 is therefore split (see Implementation Phases): **Phase 1a — Go scaffold, HTTP server, `/docs` + `/docs/{route}` + `/openapi.json` over a hand-written whole spec, structured validation errors — is genuinely unblocked by this ADR and can start now.** **Phase 1b — fragment merge, collision detection, quarantine — remains gated on the route-fragment schema (Open Question 1, bead `bf-2wt`), which is now the critical path for the project.** `seam lint`/`seam diff`/`seam import` (Phases 9a/9b, Fragment Toolchain) are Go CLI tools by the same extension, and 9a inherits 1b's gate for the same reason.
- SEAM becomes the second language in this shop's container fleet alongside Rust. The existing `cargo`/`cargo-remote` CI-offload path (`~/.local/bin/cargo`, iad-ci `rust-verify`) does not apply; a Go build path is needed once code exists — tracked as a follow-on bead (see beads filed alongside this ADR), not a blocker to starting Phase 1a, since SEAM ships as a plain container per the Hosting section regardless of build tooling.
- The validator boundary is narrow (parse + merge + validate, per the comparison matrix in `docs/research/language-runtime-choice.md`) — if Go proves to be the wrong call in practice, replacing that one component (or, worst case, the whole gateway in Rust per the runner-up analysis) is a bounded rewrite, not a redesign of SEAM's architecture.
